Go【八】context

并发编程
Go
发布日期:   2022-05-10
文章字数:   1.7k
阅读时长:   7 分
阅读次数:  

前言

context可以在goroutine之间传递上下文信息,这个传递过程以及使用是安全的。从概念上讲有些类似Java中的ThreadLocal,但是在用法上完全不同,context如果想要不断使用那就必须手动传递下去。

作用

  1. context可以存储数据传递到下游服务实现全局数据共享
  2. context可以控制任务的超时、结束流程

1. context的使用

1.1 创建context

context包提供了两种方式创建context：

context.Backgroud()

ctx := context.Background()

创建一个空的上下文,该上下文永远不会取消,也没有任何超时时间,通常会在main函数、初始化、测试使用或者作为请求的顶级上下文。

context.TODO()

ctx := context.TODO()

该方式与context.Background()没有任何区别,一般用于不确定使用哪种上下文时所使用。

在大多数情况下,我们都使用context.Background()作为起始的上下文向下传递。

上面两种方式是创建顶级context,并不具备任何功能,具体的使用是依靠 With系列函数所派生出来的context。

func WithCancel(parent Context) (ctx Context, cancel CancelFunc) 
func WithDeadline(parent Context, d time.Time) (Context, CancelFunc)
func WithDeadline(parent Context, d time.Time) (Context, CancelFunc)
func WithValue(parent Context, key, val any) Context

下面逐一详解这些context

1.2 WithValue 上下文传递数据

通常在Web开发中,我们都希望有一个trace_id能将一个请求产生的所有日志关联起来,这样我们追踪日志会很方便。在Java中我们可以通过ThreadLocal来传递,而在Go中我们就可以使用context来进行数据的传递。通过context.WithValue()来创建一个携带trace_id的context,然后将context不断传递下去。

const TRACE_ID = "trace_id"

func main() {
	//创建一个携带trace_id的上下文
	ctx := context.WithValue(context.Background(), TRACE_ID, uuid.New().String())
	//传递下去
	task1(ctx)
}

func task1(ctx context.Context) {
	//获取ctx中的trace_id数据,然后传递下去
	trace_id := ctx.Value(TRACE_ID)
	fmt.Println("task1 trace_id:", trace_id)
	task2(ctx)
}
func task2(ctx context.Context) {
	//获取ctx中的trace_id数据,并且断言处理
		if v, ok := ctx.Value(TRACE_ID).(string); ok {
		fmt.Println("task2 trace_id:", v)
	}
}

执行结果

task1 trace_id: 04d41402-6973-490c-9c62-f54c91479421
task2 trace_id: 04d41402-6973-490c-9c62-f54c91479421

如一个context携带一个数据,那么从该context衍生出来的所有context都会携带这个数据。

如下改动:

func task1(ctx context.Context) {
	//获取ctx中的trace_id数据,然后传递下去
	trace_id := ctx.Value(TRACE_ID)
	fmt.Println("task1 trace_id:", trace_id)
	//再次衍生context,添加一个新数据,然后传递下去
	ctx = context.WithValue(ctx, "newKey", "newValue")
	task2(ctx)
}
func task2(ctx context.Context) {
	//获取ctx中的trace_id数据
	fmt.Println("task2 trace_id:", ctx.Value(TRACE_ID))
	fmt.Println("task2 newKey:", ctx.Value("newKey"))
}

执行结果

task1 trace_id: bcbf5f80-6ffa-4615-b8ac-cefe142df5ee
task2 trace_id: bcbf5f80-6ffa-4615-b8ac-cefe142df5ee
task2 newKey: newValue

可以看到,在task1函数中再次衍生出来的context传递下游后依然可以获取到顶级context的数据。

1.3 WithDeadline 超时控制

通过context.WithDeadline(ctx,time)和context.WithTimeout(ctx,time)可以获取一个带有超时时间的context,一旦该上下文超时,则会执行该context.Done()方法,该方法会发出一个空信号,所以我们只需要通过select case机制进行监听该context.Done()方法即可,一旦捕获到,则表示已经超时,在该case中处理超时后的操作。

func main() {
	//创建一个带有超时时间的context 为5秒
	ctx, cancel := context.WithDeadline(context.Background(), time.Now().Add(time.Second*5))
	defer cancel()
	//执行任务,传递上下文
	task(ctx)
}
func task(ctx context.Context) {
	taskUseTime := 0
	for {
		time.Sleep(time.Second)
		taskUseTime++
		select {
		//利用select 监听上下文的Done()函数,如果到达超时时间,则会触发该case
		case <-ctx.Done():
			fmt.Println("任务超时.:", ctx.Err())
			return
		//正常任务执行case
		default:
			fmt.Println("任务执行中 time:", taskUseTime)
		}
		if taskUseTime == 10 {
			fmt.Println("任务完成.")
			break
		}
	}
}

执行结果

任务执行中 time: 1
任务执行中 time: 2
任务执行中 time: 3
任务执行中 time: 4
任务超时.: context deadline exceeded

我们把超时时间调大一点,让任务正常结束

ctx, cancel := context.WithDeadline(context.Background(), time.Now().Add(time.Second*15))

执行结果

任务执行中 time: 1
任务执行中 time: 2
任务执行中 time: 3
任务执行中 time: 4
任务执行中 time: 5
任务执行中 time: 6
任务执行中 time: 7
任务执行中 time: 8
任务执行中 time: 9
任务执行中 time: 10
任务完成.

如果我们执行context返回的cancel()方法则表示直接手动发出Done()信号,让该上下文提前超时结束。

//创建一个带有超时时间的context 为5秒
ctx, cancel := context.WithDeadline(context.Background(), time.Now().Add(time.Second*15))
//执行任务,传递上下文
go task(ctx)
time.Sleep(time.Second * 2) //延迟2秒直接强制发出超时信号
cancel()

执行结果

任务执行中 time: 1
任务超时.: context canceled

1.4 WithTimeout 超时控制

context.WithTimeout()与context.WithDeadline()一致,只是入参的时间类型不同,前者底层也是调用的后者。

func WithDeadline(parent Context, d time.Time) (Context, CancelFunc) {
	if parent == nil {
		panic("cannot create context from nil parent")
  }....
  
func WithTimeout(parent Context, timeout time.Duration) (Context, CancelFunc) {
	return WithDeadline(parent, time.Now().Add(timeout))
}

所以使用上基本一致,请直接参考context.WithDeadline()。

1.5 WithCancel 协程控制

其实学了上面三种context后,最后这一种就简单多了,因为context.WithCancel()能做到的context.WithTimtout()也能做到.就是主动调用context返回的cancel()方法,让所有监听该上下文的协程都退出。

func main() {
	//获取一个context,用来控制协程
	ctx, cancel := context.WithCancel(context.Background())
	//开启协程,将上下文传递进去
	go task(ctx)

	//简单睡眠3秒后,发出Done()信号 直接让协程退出
	time.Sleep(time.Second * 3)
	cancel()

	//再延迟一秒,以免main函数结束太快
	time.Sleep(time.Second)
	fmt.Println("main is done")
}
func task(ctx context.Context) {
	i := 0
	//每秒循环一次
	for range time.Tick(time.Second) {
		i++
		select {
		//监听上下文的Done()信号,收到后就结束协程
		case <-ctx.Done():
			fmt.Println("收到ctx Done信号,退出任务.")
			break
		default:
			fmt.Println("协程任务执行中 time:", i)
		}
	}
}

执行结果

协程任务执行中 time: 1
协程任务执行中 time: 2
协程任务执行中 time: 3
收到ctx Done信号,退出任务.
main is done

2. context API

  • Done()

    当上下文工作被取消时,会触发该上下文的Done()方法,返回一个空结构体信号<-chan struct{}。

    如果这个上下文永远不能被取消(如context.Background()),可能会返回一个nil。

  • Deadline()

    deadline, ok := ctx.Deadline() 获取当前上下文工作超时时间。如果该上下文没有超时时间则ok为false。

  • Err()

    获取context的停止原因,是取消或者超时。

  • Value(key)

    根据Key获取context中传递的数据。
