前言

JUC包下提供了很多适合高并发编程的工具集,该工具集基本上都是依赖于AQS来实现,本章就记录一下这些工具集的使用方式及使用场景。

1. CountDownLatch

CountDownLatch是类似于倒计时计数器概念的一个工具。就好似他一个人持有多把锁,只有所有的锁都被打开后,它才会允许你通行,否则将会一直阻塞。

  • 使用场景:

    在主线程中启用十个子线程去完成不同的任务,我们可以通过CountDownLatch来统一等待,只有十个线程都结束后,主线程才能结束。

  • 示例:
public static void main(String[] args) throws InterruptedException {
  //创建时入参为 计数量
  CountDownLatch latch = new CountDownLatch(10);
  IntStream.rangeClosed(0,10).forEach(i->{
    new Thread(()->{
      //TODO 业务操作
      latch.countDown();
    }).start();
  });
  latch.await();
}
  • API
//创建时入参为 计数量
CountDownLatch latch = new CountDownLatch(10);

//计数量 -1
latch.countDown();

//获取当前计数量
latch.getCount();

//阻塞等待,直到计数量为0后才会放行
latch.await();
//带有超时时间的阻塞等待
latch.await(10, TimeUnit.SECONDS);

2. CyclicBarrier

CyclicBarrier字面意思回环栅栏，通过它可以实现让一组线程等待至某个状态之后再全部同时执行。叫做回环是因为当所有等待线程都被释放以后，CyclicBarrier可以被重用。我们暂且把这个状态就叫做barrier，当调用await()方法之后，线程就处于barrier了。

  • 使用场景

假如有这样一个需求,我们将一批数据分为多组任务插入到数据库同一个数据表中,每个任务插入完成后,需要获取全部的数据,然后进行不同的数据分析操作。该需求的重点就是我们如何在任务执行中得知其他数据是否插入完成,这个需求我们就可以使用CyclicBarrier来解决。

  • 示例
public static void main(String[] args) throws InterruptedException {
  //开启2个任务 去插入数据
  //所有任务调用了await()方法后回调函数
  final CyclicBarrier cyclicBarrier = new CyclicBarrier(2,()->{
    System.out.println("所有任务数据插入完成---!");
  });

  //任务一
  new Thread(() -> {
    try {
      //睡眠线程模拟插入数据
      TimeUnit.SECONDS.sleep(5);
      System.out.println("任务一插入数据完成~");
      //阻塞等待其他任务插入数据完成,当两个个任务都插入完成后会放行
      cyclicBarrier.await();

      System.out.println("任务一开始分析数据~");
      TimeUnit.SECONDS.sleep(1);
      System.out.println("任务一数据分析完成,结束任务");
    }catch (Exception e){
      e.printStackTrace();
    }
  }).start();

  //任务二
  new Thread(() -> {
    try {
      //睡眠线程模拟插入数据
      TimeUnit.SECONDS.sleep(10);
      System.out.println("任务二插入数据完成~");
      //阻塞等待其他任务插入数据完成,当两个个任务都插入完成后会放行
      cyclicBarrier.await();

      System.out.println("任务二开始分析数据~");
      TimeUnit.SECONDS.sleep(1);
      System.out.println("任务二数据分析完成,结束任务");
    } catch (Exception e) {
      e.printStackTrace();
    }
  }).start();
}

任务二插入数据完成~
所有任务数据插入完成---!
任务二开始分析数据~
任务一开始分析数据~

任务二数据分析完成,结束任务
  • API
//阻塞等待其他任务插入数据完成,当两个个任务都插入完成后会放行
cyclicBarrier.await();

//获取当前处于 Waiting状态的任务数量 (调用了await方法后处于Waiting状态)
System.out.println(cyclicBarrier.getNumberWaiting());

//获取任务数量
System.out.println(cyclicBarrier.getParties());

//将 实例对象再次初始化为最初始状态.
cyclicBarrier.reset();

3. CountDownLatch 与 CyclicBarrier的区别

从上面两个案例上来看,这两个工具有些类似,那么有哪些区别呢

  1. CountDownLatch 不能reset, 而 CyclicBarrier是可以循环使用的。
  2. CyclicBarrier的思想是让线程之间协同到某一个点,再进行进一步操作这是有一定的关联关系的,而 CountDownLatch则是互不关心的。

4. Exchanger

Exchanger该类主要是用来做数据传递的,通常是两个线程之间的传递。因为它是成对操作的。

  • 使用场景

现在有这样一个需求,启用两个线程分别去获取语文和数学成绩,取到后保留自己获取到的成绩 再进行交换,两个线程都将得到两门成绩。

  • 示例
public static void main(String[] args) {
  //通过Exchanger 来实现线程安全的数据传递
  Exchanger<Integer> score = new Exchanger<>();

  //任务一 获取语文成绩
  new Thread(()->{
    try {
      TimeUnit.SECONDS.sleep(3);
      //随机获取一个成绩,作为语文成绩
      int languageScore = new Random().nextInt(100);

      //将语文成绩传递给另一个任务,并且从另一个任务获取数学成绩
      Integer mathScore = score.exchange(languageScore);

      System.out.println("任务一获取到语文成绩:" + languageScore + ",交换得到数学成绩:" + mathScore + "总分:" + (languageScore + mathScore));
    } catch (InterruptedException e) {
      e.printStackTrace();
    }
  }).start();

  //任务一 获取语数学成绩
  new Thread(()->{
    try {
      TimeUnit.SECONDS.sleep(5);
      //随机获取一个成绩,作为数学成绩
      int mathScore = new Random().nextInt(100);

      //将数学成绩传递给另一个任务,并且从另一个任务获取语文成绩
      Integer languageScore = score.exchange(mathScore);

      System.out.println("任务一获取到数学成绩:" + mathScore + ",交换得到语文成绩:" + languageScore + "总分:" + (languageScore + mathScore));
    } catch (InterruptedException e) {
      e.printStackTrace();
    }
  }).start();
}
  • API
Exchanger<Integer> score = new Exchanger<>();

//交换数据,入参为自己要给另一个线程的数据,返回值为对方要给自己的数据
Integer result = score.exchange(value);
//带有超时时间的交换数据方法
Integer result = score.exchange(value,10,TimeUnit.SECONDS);

注意 :exchange()方法会导致当前线程阻塞,直到另一个线程也执行exchange()方法,进行数据的交换。

5. Semaphore

Semaphore 从名字是理解是类似于信号量的概念,理解起来比较抽象,简单来说就是一个能够存放多个锁或者多个许可证的容器,通过该工具能够有效控制一个或者多个线程并行工作。

  • 通过Semaphore实现同步锁
static class SemaphoreLock{
  //创建一个只有一个许可证信号量 用来表示锁
  //false表示非公平竞争
  private final Semaphore semaphore = new Semaphore(1,false);
  /**
   * 加锁方法
   * @throws InterruptedException
   */
  public void lock() throws InterruptedException {
    //获取许可证,如果获取不到将会一直阻塞
    //因为只有一张许可证,等同于锁,获取到等同于加锁成功
    semaphore.acquire(1);
  }
  /**
   * 释放锁
   */
  public void unlock(){
    //释放一张许可证
    semaphore.release(1);
  }
}
  • API详解
//1.创建 Semaphore
//入参1: 该信号量所拥有的许可证数量
//入参2: 竞争模式为公平|非公平,默认为false - 非公平
Semaphore semaphore = new Semaphore(10, Boolean.FALSE);

//2.获取一个或n个许可证
//如果目前许可证不足,将会一直阻塞,直到有足够的许可证可获取。
semaphore.acquire();
semaphore.acquire(2);

//3.尝试获取一个或n个许可证,该方式不会阻塞,如果许可证不够将会立即终止 返回false
boolean acquire = semaphore.tryAcquire();
boolean acquired2 = semaphore.tryAcquire(2);

//4.尝试获取一个或n个许可证,指定超时时间与时间类型
//如果在指定的时间内还没有获取到许可证,将会终止 返回false
//入参(许可证数量,超时时间,时间单位)
boolean acquire3 = semaphore.tryAcquire(3, TimeUnit.SECONDS);
boolean acquire4 = semaphore.tryAcquire(2,3, TimeUnit.SECONDS);

//5. 释放一个或n个许可证
semaphore.release();
semaphore.release(2);

//6. 该信号量是否是公平模式
semaphore.isFair();

//7. 获取信号量内当前可用的许可证数量
semaphore.availablePermits();

//8. 直接获取所有的许可证,调用该方法后,无法通过其他方法获取许可证
semaphore.drainPermits();

//9. 获取当前阻塞的线程预估数量(该值并不是百分百准确)
semaphore.getQueueLength();

//10. 获取当前是否有线程正在获取许可证(该值并不是百分百准确)
semaphore.hasQueuedThreads();

//11. 获取一个或n个许可证,无视当前线程中断
semaphore.acquireUninterruptibly();
semaphore.acquireUninterruptibly(2);

6. Phaser

Phaser是JDK1.7引入的一个同步工具类,适用一些多阶段的任务处理,与CyclicBarrier和 CountDownLatch类似,但比他们功能更强大。由于这个东西过于复杂,我们先看两个案例,分别模拟CyclicBarrier与 CountDownLatch。

  • 示例一

通过Phaser控制多个线程任务的开始执行时机: 有时候我们需要所有的线程到达指定点后再同时开始运行,该需求可以通过CyclicBarrier来实现,这里使用Phaser方案。

public class PhaserExample {
  public static void main(String[] args) {
    //创建 Phaser
    Phaser phaser = new Phaser();

    //启用五个线程,执行任务
    IntStream.rangeClosed(1,5).forEach(i->{
      //注册各个参与者线程
      phaser.register();
      try {
        TimeUnit.SECONDS.sleep(i);
      } catch (InterruptedException e) {
        e.printStackTrace();
      }
      //开启新任务
      new Thread(new CustomTask(phaser),"Thread-" + i).start();
    });
  }
  static class CustomTask implements Runnable{
    private final Phaser phaser;
    public CustomTask(Phaser phaser) {
      this.phaser = phaser;
    }
    @Override
    public void run() {
      //阻塞等待所有注册的线程到达
      int advance = phaser.arriveAndAwaitAdvance();
      //do start
      System.out.println(Thread.currentThread().getName() + "开始执行任务,currentDate:" +
                         DateTimeFormatter.ofPattern("HH:mm:ss").format(LocalDateTime.now()));
    }
  }
}

以上示例,创建了五个线程,每个线程启动前都通过register()方法注册,参与者数量为5,当其中一个线程调用arriveAndAwaitAdvance()方法后,arrive数量会加1,并且如果arrive未能达到参与者数量(5)的话,会处于等待状态,直到所有线程都调用了arriveAndAwaitAdvance() 方法后,所有等待的线程都会继续执行。

Phaser也可以将执行时机掌握在手中： 如所有子任务到达指定地点后,主动打开开关,让所有子任务开始执行。

public class PhaserExample {
  public static void main(String[] args) throws IOException {
    //创建 Phaser
    Phaser phaser = new Phaser(1);

    IntStream.rangeClosed(1,5).forEach(i->{

    System.out.print("是否打开开关?");
    BufferedReader reader = new BufferedReader(new InputStreamReader(System.in));
    reader.readLine();

    //打开开关
    phaser.arriveAndDeregister();
    System.out.println("main 打开了开关");
  }

  static class CustomTask implements Runnable{
    private final Phaser phaser;
    public CustomTask(Phaser phaser) {
      this.phaser = phaser;
    }
    @Override
    public void run() {
      //阻塞等待所有注册的线程到达
      int advance = phaser.arriveAndAwaitAdvance();
      //do start
      System.out.println(Thread.currentThread().getName() + "开始执行任务,currentDate:" +
                         DateTimeFormatter.ofPattern("HH:mm:ss").format(LocalDateTime.now()));
    }
  }
}

7. Lock

java.util.concurrent.locks包下提供了大量关于锁的实现,如ReentrantLock、ReentrantReadWriteLock等。这些锁并不一定性能上比synchronized锁好,但功能上更加友好以及更完善。

1. ReentrantLock

ReentrantLock是一个可重入的独占锁。

  • 与synchronized的区别
  1. synchronized是关键字是语法层面的,不需要我们考虑异常机制,而ReentrantLock是类层面上的。
  2. 它们都是可重入锁。
  3. ReentrantLock是可中断的锁,有超时时间机制。
  4. ReentrantLock可以是公平的也可以是非公平的。
  • ReentrantLock的使用
//创建一个非公平锁
private static final Lock lock = new ReentrantLock();
public static void main(String[] args) {
  try {
    //加锁
    lock.lock();
    //TODO 业务处理
  } finally {
    //一定要在finally 中释放锁
    lock.unlock();
  }
}

锁的使用非常常见,场景就不模拟了,直接看API

  • API
//创建一个非公平锁
private static final Lock lock = new ReentrantLock();
//创建一个公平锁
private static final Lock lock1 = new ReentrantLock(true);

//查询给定Condition是否有线程正处于等待状态
lock.hasWaiters(condition);

2. ReadWriteLock

ReentrantReadWriteLock 顾名思义,它读写锁的实现,目的就是为了解决某些读取操作过多的情况下,使用独占锁过于浪费。

读写锁的互斥关系

读操作与读操作 不互斥

读操作与写操作 产生互斥

写操作与写操作 产生互斥

怎么使用也很简单,直接看API

3. Condition

Condition是为了结合ReentrantLock 实现 synchronized搭配wait和notify实现线程在条件不满足时阻塞及唤醒的工具,它是基于LockSupport实现的。

  • 实现一个简单的生产者消费者模式
class ProducerConsumerExample {
  //创建锁
  private final static  ReentrantLock LOCK = new ReentrantLock();
  //创建一个Condition条件
  private final static Condition condition = LOCK.newCondition();
  //消费数值
  private static int value = 0;
  //当前是否可消费
  private static volatile boolean consumable;

  /**
   * 生产方法
   */
  public static void producer(){
    try {
      //加锁
      LOCK.lock();
      if (consumable){
        //可消费状态,则进入阻塞,等待消费线程唤醒
        condition.await();
      }
      TimeUnit.SECONDS.sleep(1);
      System.out.println("producer value:" + ++value);
      //设置为可消费状态
      consumable = true;
      condition.signal();//唤醒等待消费的线程
    } catch (InterruptedException e) {
      e.printStackTrace();
    } finally {
      LOCK.unlock();
    }
  }

  /**
   * 消费方法
   */
  public static void consumer(){
    try {
      //加锁
      LOCK.lock();
      if (!consumable){
        //不可消费状态下,则进入阻塞,等待生产线程唤醒
        condition.await();
      }
      //消费数据,修改状态
      System.out.println("consumer value:" + value);
      consumable = false;
      condition.signal();//唤醒等待生产线程
    } catch (InterruptedException e) {
      e.printStackTrace();
    } finally {
      LOCK.unlock();
    }
  }
}
public static void main(String[] args) {
  new Thread(()->{
    while (true){
      ProducerConsumerExample.producer();

  • 实现一个稍微复杂的生产者消费者模式

通过对ConditionAPI的深入学习后,我们使用Condition实现 精准唤醒 的案例,使用两个Condition来控制。

public class DateFactory {
  //重入锁
  private static final ReentrantLock LOCK = new ReentrantLock(Boolean.FALSE);
  //生产者Condition
  private static final Condition PRODUCER_COND = LOCK.newCondition();
  //消费者Condition
  private static final Condition CONSUMER_COND = LOCK.newCondition();
  //存放数据的容器
  private static final LinkedList<Long> TIMESTAMP_POOL = new LinkedList<>();
  //容器最大容量
  private static final int MAX_CAPACITY = 100;

  /**
   * 容器生产方法
   */
  private static void producer(){
    try {
      LOCK.lock();
      //如果容器已经生产满了,该方法应当进入等待状态,等待消费者发出唤醒信号
      while (TIMESTAMP_POOL.size() >= MAX_CAPACITY){
        PRODUCER_COND.await();
      }
      //生产数据: 当前时间戳
      long value = System.currentTimeMillis();
      System.out.println("【producer-"+Thread.currentThread().getName()+"-"+"】value: -- " + value);
      TIMESTAMP_POOL.addLast(value);

      //唤醒消费者线程
      CONSUMER_COND.signalAll();
    } catch (InterruptedException e) {
      e.printStackTrace();
    } finally {
      LOCK.unlock();
    }
  }

  /**
   * 消费方法
   */
  private static void consumer(){
    try {
      LOCK.lock();
      //已经没有可以消费的数据,消费者进入等待状态,等待生产者的唤醒信号
      while (TIMESTAMP_POOL.size() <= 0){
        CONSUMER_COND.await();
      }

      //消费数据
      Long value = TIMESTAMP_POOL.removeFirst();
      System.out.println("【consumer-"+Thread.currentThread().getName()+"-"+"】value: -- " + value);

      //唤醒生产者
      PRODUCER_COND.signalAll();
    } catch (InterruptedException e) {
      e.printStackTrace();
    } finally {
      LOCK.unlock();
    }
  }

  public static void main(String[] args) {
    new Thread(()-> {
      for (;;){
        producer();
        sleep(1);
      }
    },"producer1").start();
    new Thread(()-> {
      for (;;){
        producer();
        sleep(1);
      }
    },"producer2").start();

    new Thread(()-> {
      for (;;){
        consumer();
        sleep(2);
      }
    },"consumer1").start();

    new Thread(()-> {
      for (;;){
        consumer();
        sleep(2);
      }
    },"consumer2").start();
  }

  private static void sleep(int i){
    try {
      TimeUnit.SECONDS.sleep(i);
    } catch (InterruptedException e) {
      e.printStackTrace();
    }
  }
}

4. StampedLock

StampedLock是一个更为强大的锁。ReentrantLock与ReadWriteLock能做到的,它都能做到,甚至能做到更好。

那么StampedLock究竟为什么存在,它解决了哪些痛点？

主要解决了写操作饥饿的情况,如下场景,现在有一共享数据,我们通过ReadWriteLock进行线程安全的保护,现在来了100个线程进行操作,其中99个线程都是read操作,只有一个线程为write操作。因为read与write操作是互斥关系,也就是说 write与read锁的抢占比例为1/99,也就是说最坏的情况下(非公平),write操作可能需要等待所有的read操作执行完毕后才会执行,有可能该write操作比大多数read操作都应当提前执行,但由于写锁饥饿效果,导致write操作延后执行。

StampedLock正是为了解决该场景,它能够通过一个邮戳来进行乐观的read操作。

  • 悲观读写操作

首先我们使用StampedLock实现悲观读写操作

class StampedLockPessimisticExample{
  //创建邮戳锁
  private static final StampedLock LOCK = new StampedLock();
  //读取操作
  private static void read(){
    //记录加读锁时的戳,释放锁时需要用到
    long stamped = -1;
    try {
      //加读锁
      stamped = LOCK.readLock();
      //TODO read business

    }finally {
      //释放读锁
      LOCK.unlockRead(stamped);
    }
  }

  private static void write(){
    //记录加写锁时的戳,释放锁时需要用到
    long stamped = -1;
    try {
      //加写锁
      stamped = LOCK.writeLock();

      //TODO write business
    }finally {
      //释放写锁
      LOCK.unlockWrite(stamped);
    }
  }
}

可以看出,即使是悲观方式,也比ReadWriteLock简洁不少。

  • 乐观的读操作

乐观只是针对于read操作,与write操作无关,因为主要就是解决写饥饿的情况。其原理就是在每次读之前都会校验是否有写操作要执行,如果有将会让出读锁执行权,优先让写操作执行。

class StampedLockOptimismExample {
  //创建邮戳锁
  private static final StampedLock LOCK = new StampedLock();
  /**
   * 使用StampedLock 实现乐观 读操作
   */
  private static void read() {
    //返回一个验证邮戳,如果完全锁定，则为零
    long stamp = LOCK.tryOptimisticRead();
    /*
     * 验证邮戳
     * 如果邮戳从获取后,写锁没有被抢占过,则返回True
     * 写锁如果被抢占过,则返回False
     * 简单来说其流程为: 每次读之前都会校验是否有写操作要执行,如果有将会让出读锁执行权,优先让写操作执行。
     */
    if (LOCK.validate(stamp)){
      try {
        //获取读锁
        stamp = LOCK.readLock();
        //TODO read business
      } finally {
        //释放读锁
        LOCK.unlockRead(stamp);
      }
    }
  }
}

写操作无需修改,正常使用。

StampedLock由于功能更全,性能更好,目前逐渐被广泛使用。本人也强烈推荐。但是需要注意:乐观读取操作需要在特定的场景下才会发挥最大性能,如上面说的的1/99比例的情况下,否则只会适得其反。

  • API
long stamped;
// 加读锁
stamped = LOCK.readLock();
// 加读锁,直到获取到读锁,或者被中断
stamped = LOCK.readLockInterruptibly();
// 释放读锁指定邮戳
LOCK.unlockRead(stamped);
//尝试获取读锁,如果不可用返回0

//如果当前线程在获取锁之前被中断 抛出InterruptedException异常

//释放一个持有的读锁，不需要戳记值
LOCK.tryUnlockRead();

//加写锁
stamped = LOCK.writeLock();
//获取写锁,直到获取到写锁 或者被中断
stamped = LOCK.writeLockInterruptibly();
//释放写锁
LOCK.unlockWrite(stamped);
//尝试获取写锁,如果不可用返回0
stamped = LOCK.tryWriteLock();
//尝试加写锁,超过指定时间内没有获取将返回0
//如果当前线程在获取锁之前被中断 抛出InterruptedException异常
stamped = LOCK.tryWriteLock(10, TimeUnit.SECONDS);
//释放持有的写锁，不需要戳记值
LOCK.tryUnlockWrite();

//返回一个可验证的邮戳,如果当前锁是已锁定状态,直接返回0
stamped = LOCK.tryOptimisticRead();
//验证邮戳
//如果邮戳为零，表示获取邮戳时,锁是已锁定状态,则总是返回false
//如果该邮戳从获取后,写锁没有被抢占过,则返回true
//如果调用这个方法的值不是从tryOptimisticRead获得的，或者该锁的锁方法，则没有定义任何效果或结果。
LOCK.validate(stamped);

//获取该邮戳锁的读锁,但是获取到的读锁不支持 newCondition()方法
Lock readLock = LOCK.asReadLock();
//获取该邮戳的写锁,同样也不支持 newCondition()方法
Lock writeLock = LOCK.asWriteLock();
//获取该邮戳所的读写锁
ReadWriteLock readWriteLock = LOCK.asReadWriteLock();

//获取持有的读锁的数量
int readLockCount = LOCK.getReadLockCount();
//当前锁是否处于 读锁状态
LOCK.isReadLocked();
//当前锁是否处于 写锁状态
LOCK.isWriteLocked();

// 锁模式的转换
//当前邮戳为持有写锁模式，则会释放写锁并获取读锁，并返回读锁邮戳；
//当前邮戳为持有读锁模式，则直接返回当前读锁邮戳；
//当前邮戳持有乐观锁，通过CAS立即获取读锁，则返回读锁邮戳；否则，获取失败返回0；
stamped = LOCK.tryConvertToReadLock(stamped);

//当前邮戳为持有写锁模式，直接返回当前的邮戳；
//当前邮戳为持有读锁模式，则会释放读锁并获取写锁，并返回写锁邮戳；
//当前邮戳持有乐观锁，通过CAS立即获取写锁，成功则返回写锁邮戳；失败则返回0；
stamped = LOCK.tryConvertToWriteLock(stamped);

//当前邮戳为持有读或写锁，则直接释放读写锁，并返回释放后的观察者邮戳值；
//当前邮戳持有乐观锁，若乐观锁邮戳有效，则返回观察者邮戳；
stamped = LOCK.tryConvertToOptimisticRead(stamped);

5. LockSupport

LockSupport是一个工具类，提供了基本的线程阻塞和唤醒功能，它是创建锁和其他同步组件的基础工具，内部是使用sun.misc.Unsafe类实现的。LockSupport与资源关联一个许可证概念,与Semaphor类似,但是不同的是LockSupport不能累计许可证,最多只能一个。线程的挂起与唤醒就是根据这个许可证而控制。

  • API

    **park()**：该方法表示获取许可证,如果没有许可证则阻塞当前线程。直到获取到许可证或线程中断。

    park(Object blocker)： 同park()方法，多了一个阻塞对象blocker参数。

    parkNanos(long nanos)： 同park方法，nanos表示最长阻塞超时时间，超时后自动唤起当前线程。

    parkNanos(Object blocker, long nanos)： 同parkNanos(long nanos)方法，多了一个阻塞对象blocker参数。

    parkUntil(long deadline)： 同park()方法，deadline参数表示最长阻塞到某一个时间点，当到达这个时间点，park方法将自动返回。（该时间为从1970年到现在某一个时间点的毫秒数。

    parkUntil(Object blocker, long deadline)： 同parkUntil(long deadline)方法，多了一个阻塞对象blocker参数。

    unpark(Thread t): 唤醒处于阻塞状态的线程thread。

public static void main(String[] args) {
    Thread lockThread = Thread.currentThread();
    new Thread(()->{
        TimeUnit.SECONDS.sleep(5);
      
        //唤起指定线程
        LockSupport.unpark(lockThread);
    }).start();
    //挂起当前线程
    System.out.println(new Date() + "挂起main线程...");
    LockSupport.park(blocker);
    System.out.println(new Date() + "main线程已恢复...");
}
//挂起线程,直到调用了 unpark方法
LockSupport.park(blocker);
//挂起当前线程三秒
LockSupport.parkNanos(blocker,TimeUnit.SECONDS.toNanos(3));
//挂起当前线程三秒
LockSupport.parkUntil(blocker,System.currentTimeMillis() + 3000);
