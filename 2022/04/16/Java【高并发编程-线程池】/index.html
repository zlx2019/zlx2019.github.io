Java【高并发编程-线程池】

高并发编程
发布日期:   2022-04-16
文章字数:   2.4k
阅读时长:   8 分

前言

线程池（Thread Pool）是一种基于池化思想管理线程的工具，经常出现在多线程服务器中，如MySQL。

线程过多会带来额外的开销，其中包括创建销毁线程的开销、调度线程的开销等等，同时也降低了计算机的整体性能。线程池维护多个线程，等待监督管理者分配可并发执行的任务。这种做法，一方面避免了处理任务时创建销毁线程开销的代价，另一方面避免了线程数量膨胀导致的过分调度问题，保证了对内核的充分利用。

1. 线程池的好处

  • 降低资源消耗：通过池化技术重复利用已创建的线程，降低线程创建和销毁造成的损耗。
  • 提高响应速度：任务到达时，无需等待线程创建即可立即执行。
  • 提高线程的可管理性：线程是稀缺资源，如果无限制创建，不仅会消耗系统资源，还会因为线程的不合理分布导致资源调度失衡，降低系统的稳定性。使用线程池可以进行统一的分配、调优和监控。
  • 提供更多更强大的功能：线程池具备可拓展性，允许开发人员向其中增加更多的功能。比如延时定时线程池ScheduledThreadPoolExecutor，就允许任务延期执行或定期执行。

2. 创建一个线程池

ExecutorService poolService = new ThreadPoolExecutor(
  10,
  15,
  60,
  TimeUnit.SECONDS,
  new LinkedBlockingDeque<>(50),
  Executors.defaultThreadFactory(),
  new ThreadPoolExecutor.AbortPolicy()
);

3. 七大参数

  • corePoolSize: 核心线程数,会一直保留在池中的线程数，即使它们是空闲的。

  • maximumPoolSize: 最大线程数,核心线程数不够用时,可扩容的最大数量。

  • keepAliveTime: 多余线程空闲状态所存活时间 (最大线程-核心线程=多余线程),在指定的时间内,多余线程未接到新任务将会被销毁。

  • unit: 超时时间单位。

  • workQueue: 任务队列,用于保存处于等待线程的任务。

  • threadFactory: 用于生产线程的工厂

  • handler: 拒绝策略

4. 工作流程

接收到一个新任务, 将会调用空闲线程去执行该任务,如果没有空闲线程,则会放入workQueue。直到有了空闲线程后再调用空闲线程去执行。如果任务队列满了,则会进行扩容线程数量,最大会扩容到maximumPoolSize。如果扩容到maximumPoolSize线程后,还是没有空闲线程,则会触发拒绝策略。当所有的任务逐一完成后,多余的空闲线程会在超时时间keepAliveTime到达后进行销毁。

5. 优雅关闭

通常我们在使用线程池时,如果不手动关闭,那么该主线程将不会停止,因为线程池内存在活跃的线程池,我们需要释放掉池内所有的线程。

  • 优雅关闭
//异步关闭线程池
//该方法流程
//1. 处于running状态的线程照常执行
//2. 中断掉所有空闲状态的线程
// 该方式较为优雅,会等待所有的任务结束后才会关闭线程池。
pool.shutdown();
  • 暴力关闭

该方式不推荐使用,较为危险。

// 强制关闭线程池
// 1.直接中断所有的线程
// 2.取出任务队列中还未执行的任务
List<Runnable> tasks = pool.shutdownNow();
  • 定时强制关闭线程

假如我们有这样的场景,我们需要在指定的超时时间内,如果任务还没有结束,就强制结束线程池,

  1. 首先我们需要将生产的线程改为守护线程
private static ThreadPoolExecutor createThreadPool(){
  //创建线程池
  return new ThreadPoolExecutor(
    1,
    1,
    60,
    TimeUnit.SECONDS,
    new LinkedBlockingDeque<>(1),
    r -> {
      Thread t = new Thread(r);
      t.setDaemon(true);//守护线程
      return t;
    },
    new ThreadPoolExecutor.AbortPolicy()
  );
}
  1. 然后只需要在主线程内 阻塞到指定的时间即可
public static void main(String[] args) throws InterruptedException {
  ThreadPoolExecutor pool = createThreadPool();
  pool.submit(()->{
    while (true){}
  });
  //阻塞
  pool.awaitTermination(10,TimeUnit.SECONDS);
}

6. Executors

Executors是官方原生提供的专门针对于线程池的工具类,里面提供了大量工厂方法,以及一些可以直接使用的线程池。

接下来我们来看一下,该工具类提供了哪些线程池,有哪些特点又有哪些缺点。

1. newCachedThreadPool

ExecutorService executorService = Executors.newCachedThreadPool();

直接看一下它的方法实现

public static ExecutorService newCachedThreadPool() {
  return new ThreadPoolExecutor(0, Integer.MAX_VALUE,
                                60L, TimeUnit.SECONDS,
                                new SynchronousQueue<Runnable>());
}

很明显,这个线程池其实就是通过ThreadPoolExecutor类构造出来的,与我们自定义的线程池无非就是参数不同。

corePoolSize为0,也就意味着该线程池,创建后是没有初始化线程的,只有当接到任务后才会进行扩展线程数量。

当任务执行完后,60秒后会销毁所有的线程。**注意: 该线程数的最大线程数为 **Integer.MAX_VALUE(2147483647),这意味着该线程池如果不断接到任务,那将会不停的创建线程,直到内存溢出。这是件很恐怖的事情,也就是说该线程池的拒绝策略将永远不会触发。该线程池选用SynchronousQueue作为任务队列,这是一个同步的阻塞队列,最大只能包含一个元素。

总结:

​ 该线程池是一个自动无限扩容并且频繁创建以及频繁销毁线程的线程池。

2. newFixedThreadPool

ExecutorService executorService = Executors.newFixedThreadPool(10);

该方法需要一个int类型参数,下面我们来看一下方法实现

public static ExecutorService newFixedThreadPool(int nThreads) {
  return new ThreadPoolExecutor(nThreads, nThreads,
                                0L, TimeUnit.MILLISECONDS,
                                new LinkedBlockingQueue<Runnable>());
}

corePoolSize与maximumPoolSize均为我们指定的方法参数,很显然,该线程池的线程数量是由我们固定的,并且不会产生线程的销毁,除非我们手动关闭线程池。重点是该线程池的任务队列选用的是LinkedBlockingQueue,这是一个基于链表实现的阻塞队列,它的默认容量为Integer.MAX_VALUE。

/**
 * Creates a {@code LinkedBlockingQueue} with a capacity of
 * {@link Integer#MAX_VALUE}.
 */
public LinkedBlockingQueue() {
  this(Integer.MAX_VALUE);
}

也就是说该线程池会不断的接受新的任务,放入到任务队列,等待线程的调度,直到内存溢出。

总结:

​ 该线程池是一个固定线程数的线程池,并不会扩容线程数,但是它会不断的接收新的任务,可能存在内存溢出的情况。

3. newSingleThreadExecutor

ExecutorService executorService = Executors.newSingleThreadExecutor();
public static ExecutorService newSingleThreadExecutor() {
  return new FinalizableDelegatedExecutorService
    (new ThreadPoolExecutor(1, 1,
                            0L, TimeUnit.MILLISECONDS,
                            new LinkedBlockingQueue<Runnable>()));
}

该线程池由始至终只有一个线程执行任务。

总结:

​ 该线程池能够实现一些任务的串行化。

4. newWorkStealingPool

ExecutorService executorService1 = Executors.newWorkStealingPool();
ExecutorService executorService2 = Executors.newWorkStealingPool(4);
ExecutorService executorService4 = Executors.newWorkStealingPool(8);
public static ExecutorService newWorkStealingPool() {
  return new ForkJoinPool
    (Runtime.getRuntime().availableProcessors(),
     ForkJoinPool.defaultForkJoinWorkerThreadFactory,
     null, true);
}

该方法是JDK1.8提供的,其实本身就是一个ForkJoinPool

7. ThreadPoolExecutor API

ThreadPoolExecutorAPI 详解。

  1. shutdown()

    异步关闭线程池。

  2. isShutdown()

    该线程池是否已经关闭,返回为true不一定是已经关闭,可能还存在正在执行的线程。

  3. isTerminated()

    该线程池是否已经真正的结束,表示是否所有的任务及线程都被释放。

    调用了shutdown()方法后,isShutdown()必定为true,但是 isTerminated()不一定会true,只有所有的线程释放掉,并且任务队列里的任务全都执行完成后,isTerminated()才为true。

  4. isTerminating()

    该方法是ThreadPoolExecutor的方法,表示返回该线程池是否处于结束中: 调用了shutdown()后,还未真正结束的状态。

  5. allowCoreThreadTimeOut(boolean b)

    默认为false,如果为true,表示核心线程也会随着超时时间而销毁。

  6. Remove()

    将任务从队列中移除。

  7. prestartCoreThread()

    启动一个核心线程。如果已到达最大线程数,则返回false。

  8. prestartAllCoreThreads()

    启动所有核心线程。

8. 拒绝策略

线程池提供了四种拒绝策略

  • DiscardPolicy

    new ThreadPoolExecutor.DiscardPolicy();

    该策略是直接抛出RejectedExecutionException异常。

  • DiscardPolicy

    new ThreadPoolExecutor.DiscardPolicy();

    该策略表示直接抛弃到该任务。

  • DiscardOldestPolicy

    new ThreadPoolExecutor.DiscardOldestPolicy();

    该策略会抛弃掉当前任务队列中较为老的任务。

  • CallerRunsPolicy

    new ThreadPoolExecutor.CallerRunsPolicy();

    该策略表示 将任务抛给任务提交者来执行,如果当前线程池已关闭,则直接抛弃该任务。

源码解读

1.执行流程

public void execute(Runnable command) {
        if (command == null)
            throw new NullPointerException();
        int c = ctl.get();
        // workerCountOf(c): 获取正在运行的线程数量
        // 1.如果当前当前正在运行的线程数 < corePoolSize
        if (workerCountOf(c) < corePoolSize) {
            // 启动一个新的线程来运行该任务,true表示不用扩充线程
            if (addWorker(command, true))
                return;
            c = ctl.get();
        }
        //// isRunning(c): 判断当前线程池是否处于正在运行
        // 2. 如果已经正在运行的线程数量已经大于了CorePoolSize
        // 校验线程池状态 && 把任务提交到任务队列
        if (isRunning(c) && workQueue.offer(command)) {
            // 双重检查,如果线程池已经关闭,那么直接走拒绝策略
            // 否则启动新的线程
            int recheck = ctl.get();
            if (! isRunning(recheck) && remove(command))
                reject(command);
            else if (workerCountOf(recheck) == 0)
                addWorker(null, false);
        }
        // 3. 任务提交到任务队列失败,表示队列也满了
        //    然后尝试去扩充到最大线程数,false表示需要扩充线程数量.
        else if (!addWorker(command, false))
            // 如果线程数量已经到最大可用线程数,则执行拒绝策略。
            reject(command);
    }
