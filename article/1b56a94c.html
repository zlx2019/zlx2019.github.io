Rust【七】错误处理

Rust 错误处理

Rust 对可靠性的执着也延伸到了错误处理。错误对于软件来说是不可避免的，所以 Rust 有很多特性来处理出现错误的情况。在很多情况下，Rust 要求你承认出错的可能性，并在编译代码之前就采取行动。这些要求使得程序更为健壮，它们确保了你会在将代码部署到生产环境之前就发现错误并正确地处理它们！

Rust 将错误组合成两个主要类别：可恢复错误（recoverable）和 不可恢复错误（unrecoverable）。可恢复错误通常代表向用户报告错误和重试操作是合理的情况，比如未找到文件。不可恢复错误通常是 bug 的同义词，比如尝试访问超过数组结尾的位置。

大部分语言并不区分这两类错误，并采用类似异常这样方式统一处理他们。Rust 并没有异常，但是，有可恢复错误 Result<T, E> ，和不可恢复(遇到错误时停止程序执行)错误 panic!。这一章会首先介绍 panic! 调用，接着会讲到如何返回 Result<T, E>。此外，我们将探讨决定是尝试从错误中恢复还是停止执行时的注意事项。

panic

不可恢复的错误,这种错误与Go中的错误类似,一旦发生panic表示程序受到严重错误,会直接崩溃掉。

Rust中使用panic!宏可以主动抛出一个不可恢复的错误,致使程序停止并且会打印出一个错误信息，展开并清理栈数据，然后接着退出。

主动panic

fn main(){
    panic!("模拟一个不可恢复的错误.")
}

然后通过cargo运行结果:

thread 'main' panicked at '模拟一个不可恢复的错误.', src/main.rs:2:5
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

这两行包含 panic! 调用造成的错误信息。

第一行显示了panic提供的信息并指明了源码中panic出现的位置src/main.rs:2:5 表明这是 src/main.rs 文件的第二行第五个字符。

RUST_BACKTRACE=1表示可以打印出错误栈信息.

被动panic

通过一些执行一些不规范的代码,也会产生panic。

如下: 通过一个不存在的索引,访问vector中的元素:

fn main(){
    let v = vec![1,2,3];
    v[100];
}

然后执行:

thread 'main' panicked at 'index out of bounds: the len is 3 but the index is 100', src/main.rs:3:5
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

一样会产生不可恢复的panic错误。

Result

在大多数情况下,一些错误并没有严重到需要让程序停止,这个时候如果使用panic就显得有点小题大做了。Rust对此提供了一种可恢复错误的枚举Result。通过该类型来做一些错误恢复的处理,如果实在处理不了,再考虑panic。

#![allow(unused)]
fn main() {
enum Result<T, E> {
    Ok(T),
    Err(E),
}

一些核心库都会使用该类型来表示操作结果分别表示函数的成功返回值泛型和错误消息。而Ok和Err分别代表了该操作是否成功。

通过Result能够更友好的处理一些操作,如打开一个文件流:

通过标准库中的File库的open函数打开一个文件流

use std::fs::File;
use std::io::Error;

fn main(){
    // 打开文件,获取一个Result类型
    let f: Result<File,Error> = File::open("hello.txt");
}

该函数返回值就是一个Result类型,我们可以通过match模式来处理该返回值:

fn main(){
    // 打开文件
    let f: Result<File,Error> = File::open("hello.txt");
    // 通过match判断该操作是否成功
    // 如果成功就将文件的句柄传给new_f
    let new_f = match f {
        // 如果打开文件成功了,则会执行Ok分支,并且参数为打开的文件句柄
        Ok(file)=> {
            println!("{}","成功打开了文件.");
            file
        },
        // 打开错误走Err分支
        Err(error)=> {
         panic!("打开文件错误,err: {:?}",error)
        }
    };
}

目前我在根目录是没有hello.txt文件的.接下来我们执行程序看看结果:

thread 'main' panicked at '打开文件错误,err: Os { code: 2, kind: NotFound, message: "No such file or directory" }', src/main.rs:17:10
note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

可以看到,程序直接panic了,并且错误信息也很明确 code: 2, kind: NotFound, message: "No such file or directory"找不到这个文件。

这个时候就有人想问了,这不是和直接panic没什么区别吗,反而更复杂了？是的目前是这样的,但是我们可以继续改进,如下:

我们来改动一下Err()分支代码:

fn main(){
    // 打开文件
    let f: Result<File,Error> = File::open("hello.txt");
    // 通过match判断该操作是否成功
    // 如果成功就将文件的句柄传给new_f
    let new_f = match f {
        // 如果打开文件成功了,则会执行Ok分支,并且参数为打开的文件句柄
        Ok(file)=> {
            println!("{}","成功打开了文件.");
            file
        },
        // 打开错误走Err分支
        Err(error)=> {
            // 我们在错误分支,对错误类型进行match,如果是文件不存在,那我们就创建该文件
            match error.kind() {
                // 创建该文件
                ErrorKind::NotFound => match File::create("hello.txt") {
                    // 将创建好的文件返回回去
                    Ok(fc)=> fc,
                    // 创建文件时错误,panic
                    Err(e)=> panic!("创建文件错误,err: {:?}",e)
                } ,
                // 其他错误继续panic
                other_error=> panic!("打开文件错误,err: {:?}",other_error)
            }
        }
    };
}

上面代码改动目的: 如果第一次打开文件错误了,我们对错误再次进行match,如果时文件不存在的错误类型,那我们就创建该文件。否则就panic。也就是说专门对NotFound错误做了一次兜底的处理。

接下来我们再次执行以下代码:

Finished dev [unoptimized + debuginfo] target(s) in 0.00s
Running `target/debug/Coder`

可以看到,代码成功执行了,并且在项目根目录下多了一个hello.txt文件。

不过似乎这种方式代码及其庞大,可读性并不好,使用match来处理Result有些过于麻烦了。

其实Result提供了许多方法来简化这些操作,下面我们来一一介绍.

unwrap()

unwrap()函数用来直接获取Result中的成功后的结果值。该方法简单粗暴,但是如果Result是失败结果,则会直接panic。所以使用的时候要慎重。

fn main() {
    // 打开文件,直接获取文件句柄
    let f: File = File::open("hello.txt").unwrap();
}

unwrap(default T)

为Result设置失败后的默认值。该函数失败后不会panic,会使用设置的默认值。

let f: File = File::open("hello.txt").unwrap_or(File::create("default.txt").unwrap());

unwrap_or_else()

如果Result包含Ok值,则返回该值。否则,应用函数f并返回其返回值。

unwrap_err()

如果 Result 包含 Err 值，则返回该值；否则，panic。

expect(&str)

expect()函数与unwrap()函数很类似,区别就是它可以传入一个字面量作为panic时的错误信息。

let f: File = File::open("hello.txt").expect("打开文件错误...");
thread 'main' panicked at '打开文件错误...: Os { code: 2, kind: NotFound, message: "No such file or directory" }', src/main.rs:6:44

map()

map(f: FnOnce(T) -> U) -> Result<U, E>：如果 Result 包含 Ok 值，则应用函数 f，并将结果包装在 Ok 中返回；否则，返回原始 Err 值。

map_err()

map_err(f: FnOnce(E) -> F) -> Result<T, F>：如果 Result 包含 Err 值，则应用函数 f，并将结果包装在 Err 中返回；否则，返回原始 Ok 值。

?操作符的使用

Result 值之后的 ? 被定义为与示例 9-6 中定义的处理 Result 值的 match 表达式有着完全相同的工作方式。如果 Result 的值是 Ok，这个表达式将会返回 Ok 中的值而程序将继续执行。如果值是 Err，Err 将作为整个函数的返回值，就好像使用了 return 关键字一样，这样错误值就被传播给了调用者。

如下所示:

// 打开一个文件
fn open_file(file_name: &str)-> Result<File,Error>{
    // ? 表示如果这里Err了,会直接将Err返回,不会继续执行下去了
    let f = File::open(file_name)?;
    Ok(f)
}
fn main() {
    let f = open_file("hellos.txt").expect("打开文件错误...");
}
thread 'main' panicked at '打开文件错误...: Os { code: 2, kind: NotFound, message: "No such file or directory" }', src/main.rs:6:37
stack backtrace:
fn main() {
    let f = open_file("hello.txt").expect("打开文件错误...");
} //执行成功...
