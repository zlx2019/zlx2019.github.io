什么是进程?

进程(Process)是对一个程序运行时的封装,也就是当一个程序开始运行后,就会生成一个进程。

什么意思呢？比如说我们打开了一款游戏或者任何一种工具软件,也可以是一个可执行的.exe二进制文件,这些都统称为程序,这些程序在运行时都会产生对应的进程。如下图所示:

左侧红框中的都是当前系统正在运行的程序，

简而言之: 当一个程序处于运行时就可以认为是进程,并且每个进程都是相互独立的,各自拥有自己的资源,互不干扰。

什么是线程?

线程(Thread)是进程的执行单元,也可以理解为执行器。我们在进程中的任何操作,都需要依赖于线程来完成。一个进程内可以存在多个线程,并且至少会有一个主线程来负责进程的运行。

举个栗子:

当我们启动了一款聊天程序例如QQ,那么这个时候就相当于启动了一个名为QQ的进程,并且这个进程至少会有一个主线程来保证当前进程不会自动停止,当我们想给某个好友或者某个群聊发送消息的时候就需要在进程内衍生出一些子线程(任务线程)来执行这些操作。

进程与线程

  • 线程是进程的执行单元(子级),线程需要依赖于进程而存在,一旦进程结束,那么进程内的所有线程也会随之消亡。同时进程的存活也至少需要一个线程的运行而存在,一旦所有的线程都执行结束,那么进程就会停止,它们两者之间有着相互依赖而生存的契约。
  • 进程与进程之间是相互隔离的,它们都是一个独立的执行环境,具有自己的系统资源,一个进程结束不会影响其他进程。
  • 一个线程只能属于一个进程,并且线程之间可以共享同一进程内的系统资源。

并发和并行

经常有人将并发(Concurrent)和并行(Parallel)混淆,但本质上这两者并非同一种含义。

  • 并发是指在一定的时间内应对多件事的处理能力。
  • 并行是指在同一时间能够同时处理多件事。

上图很直观的体现了两者的区别:

  • 并发: 是多个队列使用同一个咖啡机，然后两个队列轮换着使用（未必是 1:1 轮换，也可能是其它轮换规则），最终每个人都能接到咖啡。
  • 并行: 是每个队列都拥有一个咖啡机，最终也是每个人都能接到咖啡，但是效率更高，因为同时可以有两个人在接咖啡。

Rust线程

放在十年前，多线程编程可能还是一个少数人才掌握的核心概念，但是在今天，随着编程语言的不断发展，多线程、多协程、Actor 等并发编程方式已经深入人心，同时多线程编程的门槛也在不断降低，本章节我们来看看在 Rust 中该如何使用多线程。

多线程风险

由于多线程之间都是独立同时运行的,因此无法保证它们之间的执行顺序,这会导致一些问题:

  • 竞态条件

    多个线程以非一致性的顺序同时访问数据资源。

  • 死锁

    两个线程都想使用某个资源，但是又都在等待对方释放资源后才能使用，结果最终都无法继续执行。

  • 还有一些因为多线程导致的很隐晦的BUG,难以控制…

下面正式开始学习Rust中的线程~

创建线程

Rust通过thread::spawn()函数来创建一个线程,参数为一个闭包函数,也就是该线程的执行逻辑:

use std::thread;
fn main(){
    thread::spawn(||{
        println!("Hello Thread")
    });
}

以上代码就创建了一个线程,并且输出一句Hello Thread,但是执行后不会得到任何结果。这是因为main(主线程)太早过于结束,导致所产生的的子线程还没创建完成就死亡.需要手动控制一下主线程的停止。

use std::thread;
use std::time::Duration;

fn main(){
    thread::spawn(||{
        println!("Hello Thread")
    });
    thread::sleep(Duration::from_secs(1));
}

这里我们通过thread::sleep()让主线程睡眠1秒再结束,这样就可以等待子线程结束。

通过上面我们得知如下几点:

  • 线程内部代码使用闭包来执行
  • main线程一旦结束,整个程序就立刻结束,因此需要手动保持它的存活直到其他子线程完成自己的任务。
  • thread::sleep 会让当前线程休眠指定的时间，随后其它线程会被调度运行（上一节并发与并行中有简单介绍过），因此就算你的电脑只有一个 CPU 核心，该程序也会表现的如同多 CPU 核心一般，这就是并发！

JoinHandle

上面我们通过thread::sleep()来控制主线程的结束,但是这样并不推荐,因为你无法知道线程执行所需要的时间,通常我们更推荐使用JoinHandle来等待一个线程的结束。

JoinHandle就是通过thread::spawn创建一个线程的返回值

let handler:JoinHandle<()> =thread::spawn(||{
  println!("Hello Thread")
});

我们可以通过JoinHandle的join()方法来阻塞当前调用线程,直到JoinHandle对应的线程结束后才会释放:

fn main(){
    let handler:JoinHandle<()> =thread::spawn(||{
        println!("Hello Thread")
    });
    handler.join();
}

线程move闭包

在闭包中有讲过 move 关键字在闭包中的使用可以让该闭包拿走环境中某个值的所有权，同样地，你可以使用 move 来将所有权从一个线程转移到另外一个线程。

举例: 在一个子线程中使用父线程的值

首先，来看看在一个线程中直接使用另一个线程中的数据会如何：

use std::thread;
use std::thread::JoinHandle;

fn main() {
    let numbers:Vec<usize> = vec![1,2,3];
    let handle:JoinHandle<()> = thread::spawn(||{
        println!("numbers is: {:?}",numbers)
    });
    handle.join().unwrap();
}

以上代码在子线程的闭包中捕获了main线程的numbers变量,来看看结果:

其实代码本身并没有什么问题，问题在于Rust无法确定新的线程会活多久(多个线程的结束顺序并不是固定的),所以也无法确定新线程所引用的 v 是否在使用过程中一直合法,比如如下这种情况:

use std::thread;
use std::thread::JoinHandle;

fn main() {
  let numbers:Vec<usize> = vec![1,2,3];
  let handle:JoinHandle<()> = thread::spawn(||{
    println!("numbers is: {:?}",numbers)
  });
  drop(numbers);// 丢弃
  
  handle.join().unwrap();
}

由于线程的启动时间和结束时间是不确定的,因此存在一种可能,当主线程执行完，numbers被释放掉时，新的线程很可能还没有结束甚至还没有被创建成功，此时新线程对numbers的引用立刻就不再合法！

好在报错里进行了提示：to force the closure to take ownership of v (and any other referenced variables), use themovekeyword，让我们使用 move 关键字拿走numbers的所有权即可:

use std::thread;
use std::thread::JoinHandle;

fn main() {
  // 父线程创建值
  let numbers:Vec<usize> = vec![1,2,3];

  // 创建子线程,并且使用move 将引用的值的所有权转移给子线程
  let handle:JoinHandle<()> = thread::spawn(move ||{
    //使用父线程的值
    println!("numbers is: {:?}",numbers)
  });
  // 等待子线程结束
  handle.join().unwrap();
  
  // main线程无法继续使用
  println!("{:?}",numbers)

}
numbers is: [1, 2, 3]

线程的性能

据不精确估算，创建一个线程大概需要 0.24 毫秒，随着线程的变多，这个值会变得更大，因此线程的创建耗时并不是不可忽略的，只有当真的需要处理一个值得用线程去处理的任务时，才使用线程，一些鸡毛蒜皮的任务，就无需创建线程了。

那么创建多少线程合适呢?

因为 CPU 的核心数限制，当任务是 CPU 密集型时，就算线程数超过了 CPU 核心数，也并不能帮你获得更好的性能，因为每个线程的任务都可以轻松让 CPU 的某个核心跑满，既然如此，让线程数等于 CPU 核心数是最好的。

但是当你的任务大部分时间都处于阻塞状态时，就可以考虑增多线程数量，这样当某个线程处于阻塞状态时，会被切走，进而运行其它的线程，典型就是网络 IO 操作，我们可以为每一个进来的用户连接创建一个线程去处理，该连接绝大部分时间都是处于 IO 读取阻塞状态，因此有限的 CPU 核心完全可以处理成百上千的用户连接线程，但是事实上，对于这种网络 IO 情况，一般都不再使用多线程的方式了，毕竟操作系统的线程数是有限的，意味着并发数也很容易达到上限，而且过多的线程也会导致线程上下文切换的代价过大，使用 async/await 的 M:N 并发模型，就没有这个烦恼。

Barrier

Barrier被翻译为屏障,在Rust中可以使用std::sync::Barrier来让多个线程执行到某个点后进行阻塞,直到所有的线程都到达再释放所有线程。类似于Java中的CyclicBarrier。

示例:

use std::sync::{Arc, Barrier};
use std::thread;
use std::thread::JoinHandle;

fn main(){
    //任务状态容器,共6ge任务
    let mut task_handler:Vec<JoinHandle<()>> = Vec::with_capacity(6);
    // 创建支持多线程环境的屏障器,共6个
    let barrier: Arc<Barrier> = Arc::new(Barrier::new(6));

    for num in 0..6{
        // 克隆一份屏障器的所有权
        let temp_barrier = barrier.clone();
        // 创建任务
        task_handler.push(thread::spawn(move||{
            println!("{}号任务Start...",num);
            // 阻塞
            temp_barrier.wait();
            println!("{}号任务End...",num);
        }));
    }
    // 等待所有线程执行完成
    for task in task_handler {
        task.join().unwrap();
    }
}

输出结果如下:

0号任务Start...
3号任务Start...
2号任务Start...
5号任务Start...
1号任务Start...
4号任务Start...
3号任务End...
4号任务End...
1号任务End...
0号任务End...
2号任务End...
5号任务End...

当任何一个线程调用了Barrier.wait()方法时都会陷入一个阻塞状态,直到6个线程都调用了wait()方法后阻塞的线程全部都会唤醒。

Once

有时，我们会需要某个函数在多线程环境下只被调用一次，例如初始化全局变量，无论是哪个线程先调用函数来初始化，都会保证全局变量只会被初始化一次，随后的其它线程调用就会忽略该函数。

Rust中也提供了类似于Go中的Once.Do()同步原语,用于确保在多线程环境下只执行一次特定的操作: std::sync::Once

use std::sync::Once;
use std::thread;

//用于保证多线程环境只执行一次闭包
static Run: Once = Once::new();

fn main(){
    // 启动两个线程分别执行Run闭包
    let task1 = thread::spawn(move ||{
        Run.call_once(||{
            println!("Run task1");
        })
    });
    let task2 = thread::spawn(move ||{
        Run.call_once(||{
            println!("Run task2");
        })
    });
    task1.join().unwrap();
    task2.join().unwrap();
}

无论执行多少次,结果只有一条,可能是Run task1,也有可能是Run task2.

总结

Rust 的线程模型是 1:1 模型，因为 Rust 要保持尽量小的运行时。

我们可以使用 thread::spawn 来创建线程，创建出的多个线程之间并不存在执行顺序关系，因此代码逻辑千万不要依赖于线程间的执行顺序。

main 线程若是结束，则所有子线程都将被终止，如果希望等待子线程结束后，再结束 main 线程，你需要使用创建线程时返回的句柄的 join 方法。

在线程中无法直接借用外部环境中的变量值，因为新线程的启动时间点和结束时间点是不确定的，所以 Rust 无法保证该线程中借用的变量在使用过程中依然是合法的。你可以使用 move 关键字将变量的所有权转移给新的线程，来解决此问题。

父线程结束后，子线程仍在持续运行，直到子线程的代码运行完成或者 main 线程的结束。

线程通信-Channel

在Go语言中有这样一句名言:

不要通过共享内存来进行通信,而是通过通信来共享内存

这是一种目前越来越流行的思想和实现方案,它能在保证安全并发的情况下更简单优雅的实现线程之间的通信。

那么这句话到底是什么意思呢? 简单来说就是:

不要在多个线程内直接对一个数据进行操作,而是抽象出一个容器,数据放在这个安全的容器内,然后由这个容器来保证数据的安全性,并且对外暴露操作功能。

在Go中,由于这则思想从而衍生出了chan类型。也就是Channel(通道)。这样就不需要操作者通过主动加锁等复杂化机制来保证数据的安全性。

在Rust中也提供了类似的方案: Channel(标准库提供)

一个Channel包含两个端: 发送端(Sender)、接收端(Receiver)。

  • 发送端: 负责将数据发送到Channel。
  • 接收端: 负责读取Channel内的数据。

如果发送端、接收端任何一个被销毁了,那么Channel就被关闭了。

use std::sync::mpsc::*;

fn main() {
    // 创建一个Channel,获取Channel的发送端和接收端
    let (send, recv) = channel();
}

单发送者-单接收者

通过std::sync::mpsc::channel()来创建一个Channel(多个生产者,一个消费者),该函数返回一个tuple(元组),包含两个元素分别是发送端和接收端。当然，支持多个发送者也意味着支持单个发送者。

先来看看如何实现一个发送者和一个接收者示例:

use std::sync::mpsc::*;

fn main() {
    // 创建一个Channel,获取Channel的发送端和接收端
    let (send, recv) = channel();
    // 开启一个子线程,向通道发送一条数据
    std::thread::spawn(move || {
        // 发送消息
        send.send(String::from("Hello")).unwrap();
    });
    // 主线程接收消息
    // 会一直阻塞,直到读取到消息
    let msg = recv.recv().unwrap();
    println!("channel msg: {}",msg) // channel msg: Hello
}

上面代码并不复杂,是一个1:1模型的生产消费者模型,子线程负责生产,主线程负责接收。但有如下几点需要注意:

  • send,recv分别对应发送者和接收者,它们的类型由编译器自动推导,send.send(String::from("Hello"))发送了String,因此它们的类型分别是mpsc::Sender<String>和mpsc::Receiver<String>。

  • recv.recv()会阻塞当前线程,直到读取到值或者通道被关闭。

  • send.send()方法放回一个Result<T,E>表示可能会发送消息错误,比如接收者被drop,导致了发送的所有数据都不会被接收,所以继续发送是毫无意义的,所以返回一个错误。同样的，对于recv方法来说，当发送者关闭时，它也会接收到一个错误，用于说明不会再有任何值被发送过来。

多发送者-单接收者

实现N:1模型也很简单,我们只需要克隆出多份发送者,然后让每个线程拿走它的一份拷贝即可:

use std::sync::mpsc;
use std::thread;

/// 使用channel 进行线程通信 N:1模型
fn main(){
    // 创建通道
    let (tx,rx) = mpsc::channel();
    // 克隆一个发送端
    let tx1 = tx.clone();
    
    // 第一个发送者
    thread::spawn(move||{
       tx.send(1).unwrap();
    });
    // 第二个发送者
    thread::spawn(move|| {
       tx1.send(2).unwrap();
    });
    //循环接收消息
    for msg in rx {
        println!("message:{}",msg)
    }
}

执行结果:

message:1
message:2

非阻塞接收消息

除了上述的recv()方法,还可以使用try_recv()来尝试接收一次消息,该方法不会阻塞线程,当通道中没有消息时,它会立刻返回一个错误。

use std::sync::mpsc::{Receiver, Sender};

fn main(){
    let (tx,rx):(Sender<usize>,Receiver<usize>) = std::sync::mpsc::channel();

    // 尝试接收一个消息
    let msg = rx.try_recv();
    println!("{:?}",msg); //Err(Empty)
}

如上通过try_recv()方法尝试接收一个消息,返回内容为Err(Empty),Empty则表示通道并没有消息。

如果返回一个Err(Disconnected)则表示发送者已经drop。

发送所有权数据

使用通道来传输数据，一样要遵循 Rust 的所有权规则：

  • 若值的类型实现了Copy特征，则直接复制一份该值，然后传输过去，例如之前的i32类型
  • 若值没有实现Copy，则它的所有权会被转移给接收端，在发送端继续使用该值将报错

一起来看看第二种情况:

use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        let s = String::from("我，飞走咯!");
        tx.send(s).unwrap();
        println!("val is {}", s); // value borrowed here after move
    });

    let received = rx.recv().unwrap();
    println!("Got: {}", received);
}

以上代码中，String底层的字符串是存储在堆上，并没有实现Copy特征，当它被发送后，会将所有权从发送端的s转移给接收端的received，之后s将无法被使用。

for循环接收消息

下面来看看如何使用for来连续接收通道中的消息:

use std::sync::mpsc;
use std::thread;
use std::time::Duration;

fn main() {
    let (tx, rx) = mpsc::channel();
    thread::spawn(move || {
        let vals = vec![
            String::from("hi"),
            String::from("from"),
            String::from("the"),
            String::from("thread"),
        ];
        // 循环发送多条消息
        for val in vals {
            tx.send(val).unwrap();
          	// 发送一条睡眠1秒
            thread::sleep(Duration::from_secs(1));
        }
    });
		// 使用通道接收者,循环阻塞式读取通道内的消息
    for received in rx {
        println!("Got: {}", received);
    }
}

在上面代码中，主线程和子线程是并发运行的，子线程在不停的发送消息 -> 休眠 1 秒，与此同时，主线程使用for循环阻塞的从rx迭代器中接收消息，当子线程运行完成时，发送者tx会随之被drop，此时for循环将被终止，最终main线程成功结束。

同步和异步通道

Rust标准库的mpsc通道分为两种类型: 同步和异步。

  • 异步通道

    发送端将消息发送到通道内,然后立刻返回,不关心消息有没有被接收端接收。该通道消息容纳上限取决于内存大小,直到撑爆…请谨慎使用。

  • 同步通道

    发送端发送消息时会阻塞住,直到消息被接收后才会解除发送端的阻塞。同步通道可以指定消息缓冲区大小,通过这个参数可以实现Go中的两种特点的通道:

    • 假设设置为0,表示没有缓冲区。等同于Go中的无缓冲通道。
    • 假设设置为n,表示同步通道,最多可以接纳n条消息,如果已经满了,发送端再发送消息时则会被阻塞住,直到有消息被接收。等同于Go中的缓冲区通道。

异步通道

之前我们使用的都是异步通道：无论接收者是否正在接收消息，消息发送者在发送消息时都不会阻塞:

use std::sync::mpsc;
use std::thread;
use std::time::Duration;
fn main() {
    let (tx, rx)= mpsc::channel();

    let handle = thread::spawn(move || {
        println!("发送之前");
        tx.send(1).unwrap();
        println!("发送之后");
    });

    println!("睡眠之前");
    thread::sleep(Duration::from_secs(3));
    println!("睡眠之后");

    println!("receive {}", rx.recv().unwrap());
    handle.join().unwrap();
}

运行输出如下:

睡眠之前
发送之前
发送之后
// 睡眠3秒...
睡眠之后
receive 1

主线程因为睡眠阻塞了 3 秒，因此并没有进行消息接收，而子线程却在此期间轻松完成了消息的发送。等主线程睡眠结束后，才姗姗来迟的从通道中接收了子线程老早之前发送的消息。

同步通道

与异步通道相反，同步通道发送消息是阻塞的，只有在消息被接收后才解除阻塞，例如：

use std::sync::mpsc;
use std::thread;
use std::time::Duration;
fn main() {
    let (tx, rx)= mpsc::sync_channel(0);

    let handle = thread::spawn(move || {
        println!("发送之前");
        tx.send(1).unwrap();
        println!("发送之后");
    });

    println!("睡眠之前");
    thread::sleep(Duration::from_secs(3));
    println!("睡眠之后");

    println!("receive {}", rx.recv().unwrap());
    handle.join().unwrap();
}

运行输出结果:

睡眠之前
发送之前
// 睡眠3秒...
睡眠之后
receive 1
发送之后

可以看出，主线程由于睡眠被阻塞导致无法接收消息，因此子线程的发送也一直被阻塞，直到主线程结束睡眠并成功接收消息后，发送才成功：发送之后的输出是在receive 1之后，说明只有接收消息彻底成功后，发送消息才算完成。

注意: 这里使用mpsc::sync_channel(0);创建了一个通道,这个0表示的就是通道的缓冲区大小,0则表示为没有缓冲区,所以会一直阻塞住发送端,直到接收端也上线。

缓冲通道

创建同步通道时,可以指定通道的缓冲区大小,当你设定为N时，发送者就可以无阻塞的往通道中发送N条消息，当消息缓冲队列满了后，新的消息发送将被阻塞(如果没有接收者消费缓冲队列中的消息，那么第N+1条消息就将触发发送阻塞)。

问题又来了，异步通道创建时完全没有这个缓冲值参数mpsc::channel()，它的缓冲值怎么设置呢？ 额。。。都异步了，都可以无限发送了，都有摩托车了，还要自行车做啥子哦？事实上异步通道的缓冲上限取决于你的内存大小，不要撑爆就行。

因此，使用异步消息虽然能非常高效且不会造成发送线程的阻塞，但是存在消息未及时消费，最终内存过大的问题。在实际项目中，可以考虑使用一个带缓冲值的同步通道来避免这种风险。

关闭通道

之前我们数次提到了通道关闭，并且提到了当通道关闭后，发送消息或接收消息将会报错。那么如何关闭通道呢？ 很简单：所有发送者被drop或者所有接收者被drop后，通道会自动关闭。

神奇的是，这件事是在编译期实现的，完全没有运行期性能损耗！只能说 Rust 的Drop特征 YYDS!

线程同步-Mutex

除了消息传递这种方式,Rust同样是支持使用共享内存来实现线程同步,例如通过锁和原子操作等并发原语来实现多个线程同时且安全访问一个资源。

互斥锁Mutex

既然是共享内存，那并发原语自然是重中之重，先来一起看看皇冠上的明珠: 互斥锁std::sync::Mutex(mutual exclusion 的缩写)。

Mutex让多个线程并发的访问同一个值变成了排队访问：同一时间，只允许一个线程A访问该值，其它线程需要等待A访问完成后才能继续。

单线程使用Mutex

先来看看在单线程环境中,Mutex如何使用:

use std::sync::Mutex;

/// 单线程中使用 Mutex
fn main(){
    // 创建互斥锁,其值为usize=10
    let counter:Mutex<usize> = Mutex::new(10);
    {
        // 获取锁,然后deref为`counter`的引用
        let mut value = counter.lock().unwrap();
        //修改值
        *value = 100;
    } //作用域结束后,默认解锁

    println!("counter = {:?}",*counter.lock().unwrap()); // 100
}

和Box类似，数据被Mutex所拥有，要访问内部的数据，需要使用方法m.lock()向m申请一个锁, 该方法会阻塞当前线程，直到获取到锁，因此当多个线程同时访问该数据时，只有一个线程能获取到锁，其它线程只能阻塞着等待，这样就保证了数据能被安全的修改。

m.lock()方法也有可能报错，例如当前正在持有锁的线程panic了。在这种情况下，其它线程不可能再获得锁，因此lock方法会返回一个错误。

这里你可能奇怪，m.lock明明返回一个锁，怎么就变成我们的num数值了？聪明的读者可能会想到智能指针，没错，因为Mutex<T>是一个智能指针，准确的说是m.lock()返回一个智能指针MutexGuard<T>:

  • 它实现了Deref特征，会被自动解引用后获得一个引用类型，该引用指向Mutex内部的数据。
  • 它还实现了Drop特征，在超出作用域后，自动释放锁，以便其它线程能继续获取锁。

正是因为智能指针的使用,使得我们使用Mutex无需任何操作就能直接其中的数据,如果释放锁你需要做的仅仅是做好锁的作用域管理即可。

如果不使用作用域来管理锁的使用,很可能会进入一种死锁状态,如下:

fn main(){
  let counter:Mutex<usize> = Mutex::new(10);
  // 获取锁
  let mut value = counter.lock().unwrap();
  *value = 100;

  //再次获取锁,死锁
  let mut value2 = counter.lock().unwrap();
  *value2 = 200;
  ....
}

运行后你会发现,程序会直接卡死住,因为我们在同一个作用域中重复获取了两次同一把锁,由于上一把锁还未释放,再次尝试获取就陷入死锁状态。由此也可以得出: Mutex是不可重入的。

多线程使用Mutex

单线程中使用锁，说实话纯粹是为了演示功能，毕竟多线程才是锁的舞台。 现在，我们再来看看，如何在多线程下使用Mutex来访问同一个资源.

示例: 使用多线程对一个数字进行递增,递增到10.

既然是多线程环境,那么就需要考虑值的多所有权,Rc智能指针无法在多线程环境下使用,需要使用Arc智能指针。代码如下:

use std::sync::{Arc, Mutex};
use std::thread;
use std::thread::JoinHandle;

fn main(){
    // 计数器
    // 使用Mutex保证数据一致性
    // 使用Arc实现多所有权
    let counter:Arc<Mutex<usize>> = Arc::new(Mutex::new(0));
    // 任务集合
    let mut tasks = vec![];

    // 循环创建任务
    for _ in 0..10{
        // 克隆一份counter的所有权
        let counter = Arc::clone(&counter);
        // 创建任务
        let task: JoinHandle<()> = thread::spawn(move||{
            // 获取锁
            let mut count = counter.lock().unwrap();
            // 计数递增
            *count += 1;
        });
        tasks.push(task);
    }

    // 等待所有的任务结束
    for task in tasks{
        task.join().unwrap();
    }
    
    //输出结果值
    println!("最终的结果值: {}",*counter.lock().unwrap())
}

不出意外的话,输出结果如下:

最终的结果值: 10
