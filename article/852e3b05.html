ElasticSearch【三】高级特性

发表于2022-06-26|更新于2023-03-22|搜索引擎
|字数总计:4.5k|阅读时长:21分钟|阅读量:

前言

ElasticSearch作为目前最流行的开源搜索引擎,工作中难免会经常用到,大多数情况下都是一些基础使用,本章主要介绍一些关于它的高级特性。

1. Index Template

ElasticSearch提供了索引模板特性,简单来说就是在创建新的索引库时,可以自动应用模板中的settings、mappings等配置。

举个栗子: 我们的日志平台希望每天生成的日志索引库都具有相同的设置和字段,这种情况下就能很好用利用该特性。

1. 创建及使用模板

模板语法

//创建模板
PUT /_template/<模板名称>

//查询模板
GET /_template/<模板名称>

// 删除模板
DELETE /_template/<模板名称>

下面我们创建一个索引模板

PUT _template/log_template
{
  "index_patterns": ["test_log_*"], 
  "order": 1,
  "settings": {
        "number_of_shards": 4,
        "number_of_replicas": 1
  },
  "mappings": {
    "properties": {
        "message":{
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_max_word",
        "norms": false
      },
      "@timestamp":{
        "type": "date"
      }
    }
  }
}

属性解释:

index_patterns: 索引匹配规则,表示创建的索引库名称凡是以test_log-开始的都会应用该模板。

order：优先级,如果有多个模板并且匹配规则冲突,将会以此值优先选择,值越小,优先级越高。

settings与mappings就不用多说了,是该模板的配置与映射,创建的索引库都会使用该配置。

创建一个新的索引库

该索引库名称显然是满足 test_log_*匹配规则。

2. 设置别名

创建索引模板时,可以指定索引库别名,如下:

PUT _template/log_template
{
  "index_patterns": ["test_log_*"], 
  "order": 1,
  "settings": {
        "number_of_shards": 4,
        "number_of_replicas": 1
  },
  "mappings": {
    "properties": {
        "message":{
        "type": "text",
        "analyzer": "ik_max_word",
        "search_analyzer": "ik_max_word",
        "norms": false
      },
      "@timestamp":{
        "type": "date"
      }
    }
  },
  "aliases": {
    "{index}-alias" : {}
  }
}

{index}最终会替换成创建的索引库名称。

2. Pipeline

Pipeline是 ElasticSearch5提供的预处理特性。能够对要插入索引库的文档数据进行多段处理操作。

Pipeline的组成:

  • description: 描述该pipeline功能
  • processors: 一系列的处理链

1. 调试Pipeline

Es提供了_simulate来调试Pipeline。

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "description": "pipeline功能描述",
    // 处理链
    "processors": [
      {}
    ]
  },
  //文档列表
  "docs": [
    {
        "_source":{}
    }
  ]
}

如下案例: 对tags字符串根据,进行分割。

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "description": "将文档的tags属性根据 , 分割",
    "processors": [
      {
        "split": {
          "field": "tags",
          "separator": ","
        }
      }
    ]
  },
  "docs": [
    {
    "_source":{
      "name": "张三",
      "tags": "乒乓球,篮球,足球"
      }
    }
  ]
}

处理结果:

{
  "docs" : [
    {
      "doc" : {
        "_index" : "_index",
        "_type" : "_doc",
        "_id" : "_id",
        "_source" : {
          "name" : "张三",
          "tags" : [
            "乒乓球",
            "篮球",
            "足球"
          ]
        },
        "_ingest" : {
          "timestamp" : "2022-06-26T01:32:00.018Z"
        }
      }
    }
  ]
}

2. 定义Pipeline

将预处理定义成固定的Pipeline

//创建语法
PUT _ingest/pipeline/<Pipeline名称>
{
  "description": "功能描述",
  "processors": [
    {
      //处理链
    }
  ]
}

//查看语法
GET _ingest/pipeline/<Pipeline名称>

//删除语法
DELETE _ingest/pipeline/<Pipeline名称>

PUT _ingest/pipeline/tagsSplit
{

    {
      "split": {
        "field": "tags",
        "separator": ",",
        "ignore_failure": true // 这里指定发生错误时忽略

    }
  ]
}

3. 定义Processors错误处理

一般有两种方式:

  • 忽略: 配置ignore_failure=true。
  • 错误处理: 配置 on_failure,这里可以定义多个错误处理。

如下:

PUT _ingest/pipeline/tagsSplit
{
  "description": "对tags属性根据,进行分割",
  "processors": [
    {
      "split": {
        "field": "tags",
        "separator": ",",
        // 错误处理链
        "on_failure": [
          {
            //set 表示添加一个新的字段处理器
            // 当发生错误后,添加一个`error`属性,其值为发生的异常错误消息
            "set":{
              "field": "error",
              "value": "{{ctx.error}};{{_ingest.on_failure_message}}"
            }
          }
        ]
      }
    }
  ]
}

接下来我们来测试一下异常情况

PUT /user/_doc/2?pipeline=tagsSplit
{
  "name": "张三"
}

GET /user/_doc/2

结果:

{
  "_index" : "user",
  "_type" : "_doc",
  "_id" : "2",
  "_version" : 1,
  "_seq_no" : 1,
  "_primary_term" : 1,
  "found" : true,
  "_source" : {
    "name" : "张三",
    "error" : ";field [tags] not present as part of path [tags]"
  }
}

显然,这条数据因为没有tags属性而出错了.

看到了，在on_failure中，我们能通过_ingest能获取并仅能获取到三个属性：

  • on_failure_message(错误信息)
  • on_failure_processor_type(处理器类型)
  • on_failure_processor_tag(处理器的tag)

3. 内置Processors

这里介绍一些常用的处理器,需要查看全部请参考官网 地址

每个内置处理器都包含一个if属性,只有该属性为真的时候才会执行该处理器。

如下:

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "processors": [
      {
        "set": {
          "if": "false", 
          "field": "newField",
          "value": "111"
        }
      }
    ]
  },
  "docs": [
    {
    "_source":{
      
      }
    }
  ]
}

结果:

{
  "docs" : [
    {
      "doc" : {
        "_index" : "_index",
        "_type" : "_doc",
        "_id" : "_id",
        "_source" : { },
        "_ingest" : {
          "timestamp" : "2022-06-26T09:13:08.243Z"
        }
      }
    }
  ]
}

if为假,该处理器并未执行,没有新增newField字段。

POST _ingest/pipeline/_simulate

  "docs": [
    {
    "_source":{
      
      }
    }
  ]
}
{

      "doc" : {
        "_index" : "_index",
        "_type" : "_doc",
        "_id" : "_id",

if为真才执行。

1. append

向指定的属性追加元素。如果属性不存在,则生成属性并且添加元素。如果属性不是数组,则将属性转为数组类型。

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "processors": [
      {
        "append": {
          "field": "products",
          "value": ["排球","羽毛球"]
        }
      }
    ]
  },
  "docs": [
    {
    "_source":{
      "products": ["乒乓球","篮球","足球"]
      }
    }
  ]
}
{
  "docs" : [
    {
      "doc" : {
        "_index" : "_index",
        "_type" : "_doc",
        "_id" : "_id",
        "_source" : {
          "products" : [
            "乒乓球",
            "篮球",
            "足球",
            "排球",
            "羽毛球"
          ]
        },
        "_ingest" : {
          "timestamp" : "2022-06-26T02:04:22.351Z"
        }
      }
    }
  ]
}

2. convert

类型转换处理器

// 将number 转换为 string类型,并且赋值到textNumber属性中.
// 如果不指定target_field 则会原地改变。
POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "processors": [
      {
        "convert": {
          "field": "number",
          "type": "string",
          "target_field": "textNumber"
        }
      }
    ]
  },
  "docs": [
    {
    "_source":{
      "number": 10086
      }
    }
  ]
}

转换结果:

{
  "docs" : [
    {
      "doc" : {
        "_index" : "_index",
        "_type" : "_doc",
        "_id" : "_id",
        "_source" : {
          "number" : 10086,
          "textNumber" : "10086"
        },
        "_ingest" : {
          "timestamp" : "2022-06-26T02:12:15.996Z"
        }
      }
    }
  ]
}

3. date

日期时间格式化处理器

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "processors": [
      {
        // 格式化timestamp属性,然后赋值到@timestamp属性中,如果@timestamp不存在则生成。
        "date": {
          "field": "timestamp",
          "formats": ["yyyy-MM-dd HH:mm:ss"], //格式
          "timezone": "Asia/Shanghai",				//时区
          "target_field": "@timestamp"				//要赋值的属性名
        }
      }
    ]
  },
  "docs": [
    {
    "_source":{
      "timestamp": "2022-06-25 18:39:29"
      }
    }
  ]
}

格式化结果:

{
  "docs" : [
    {
      "doc" : {
        "_index" : "_index",
        "_type" : "_doc",
        "_id" : "_id",
        "_source" : {
          "@timestamp" : "2022-06-25T18:39:29.000+08:00",
          "timestamp" : "2022-06-25 18:39:29"
        },
        "_ingest" : {
          "timestamp" : "2022-06-26T02:23:48.700Z"
        }
      }
    }
  ]
}

4. drop

文档删除处理器,该处理器会直接删除当前这条文档。

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "processors": [
      {
        "drop": {
          "if" : "true"
        }
      }
    ]
  },
  "docs": [
    {
    "_source":{
      "message": "hello"
      }
    }
  ]
}
{
  "docs" : [

  ]
}

5. fail

管道执行失败,传递消息处理器。比如说,某个属性不存在,我们希望将该错误消息传递给请求者。

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "processors": [
      {
        "fail": {
          "if": "ctx.tags.contains('苹果') != true", 
          "message": "tags属性中不包含苹果!"
        }
      }
    ]
  },
  "docs": [
    {
    "_source":{
      "tags": ["香蕉","橘子"]
      }
    }
  ]
}
{
  "docs" : [
    {
      "error" : {
        "root_cause" : [
          {
            "type" : "fail_processor_exception",
            "reason" : "tags属性中不包含苹果!"
          }
        ],
        "type" : "fail_processor_exception",
        "reason" : "tags属性中不包含苹果!"
      }
    }
  ]
}

7. foreach

循环遍历处理器。该处理器需要指定一个数组和一个处理器,其数组中的每个元素都会被内部处理器所执行。

将数组中的每一个元素转换为大写。

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "processors": [
      {
        "foreach": {
          // 要遍历的数组字段
          "field": "values",
          // 执行遍历的处理器
          "processor": {
            // 将每个元素都转换为大写
            "uppercase": {
              // _ingest._value 表示数组的元素
              "field": "_ingest._value"
            }
          }
        }
      }
    ]
  },
  "docs": [
    {
    "_source":{
        "values": ["username","password","address"]
      }
    }
  ]
}

结果:

{
  "docs" : [
    {
      "doc" : {
        "_index" : "_index",
        "_type" : "_doc",
        "_id" : "_id",
        "_source" : {
          "values" : [
            "USERNAME",
            "PASSWORD",
            "ADDRESS"
          ]
        },
        "_ingest" : {
          "_value" : null,
          "timestamp" : "2022-06-26T09:34:55.548Z"
        }

21. Dissect(重要)

文本解析提取处理器,与Grok处理器类似。然而与Grok处理器不同的是,Dissect不使用正则表达式,也没有任何语法,只是使用占位符 %{paramName}。

这使得剖析的语法变得简单，并且在某些情况下比Grok Processor更快。

提取到的所有数据都会称为文本类型。

如下案例:

解析[zeus-music] 2022-06-25 18:39:29 INFO 4703 [default] [main] com.zero.music.ZeusMusicApplication 这是一条日志]日志,映射成一个json文档。

该日志所对应的语法结构:

[%{serviceName}] %{date} %{time} %{logLevel} %{pid} [%{traceId}] [%{threadName}] %{classPath} %{logContent}

处理器使用

POST _ingest/pipeline/_simulate
{
  "pipeline": {
    "processors": [
      {
        "dissect": {
          "field": "message",
          "pattern": "[%{serviceName}] %{date} %{time} %{logLevel} %{pid} [%{traceId}] [%{threadName}] %{classPath} %{logContent}"
        }
      }
    ]
  },
  "docs": [
    {
    "_source":{
      "message": "[zeus-music] 2022-06-25 18:39:29 INFO 4703 [default] [main] com.zero.music.ZeusMusicApplication 这是一条日志"
      }
    }
  ]
}

处理结果:

{
  "docs" : [
    {
      "doc" : {
        "_index" : "_index",
        "_type" : "_doc",
        "_id" : "_id",
        "_source" : {
          "date" : "2022-06-25",
          "traceId" : "default",
          "logContent" : "这是一条日志",
          "pid" : "4703",
          "message" : "[zeus-music] 2022-06-25 18:39:29 INFO 4703 [default] [main] com.zero.music.ZeusMusicApplication 这是一条日志",
          "serviceName" : "zeus-music",
          "threadName" : "main",
          "logLevel" : "INFO",
          "classPath" : "com.zero.music.ZeusMusicApplication",
          "time" : "18:39:29"
        },
        "_ingest" : {
          "timestamp" : "2022-06-26T08:47:09.314Z"
        }
      }
    }
  ]
}

这种方式效率不错,但只适合抽取文本类型数据,如这里的date和time就没有办法抽取到一块,除非使用额外的处理器。
