Gin-源码及原理剖析

发表于2023-08-17|更新于2023-08-18|Go
|字数总计:4.6k|阅读时长:17分钟|阅读量:

1.前言

Gin 是 Golang 语言中最流行的 Web 框架，它基于原生的 net/http 库，并且其性能超越近40倍，在 Github 上有极高的Star;

Gin 的特点:

  • 高性能;
  • 扩展性强，支持Handlers处理器责任链;
  • 全局上下文的传递;
  • 更高效的路由解析机制，基于 redix-tree 前缀树结构实现的路由树;
  • 服务端渲染
  • …

注: 本章阅读的 Gin 版本为 v1.9.0

net/http示例

Go 原生的net/http 库就已经很强大，可以非常简单的编写一个Web服务，只需要寥寥几行代码即可实现，示例如下:

import (
	"net/http"
)
func main() {
	// 注册一个请求Path和对应的处理函数
	http.HandleFunc("/ping", func(w http.ResponseWriter, r *http.Request) {
		w.Write([]byte("ping"))
	})
	// 阻塞启动一个HTTP服务，使用默认的http.DefaultServeMux作为服务的实现
	if err := http.ListenAndServe(":8989", nil); err != nil {
		panic(err)
	}
}

在上述案例中，完成了两件事:

  1. 调用http.HandleFunc函数，注册了对应请求路径为/ping的处理函数;
  2. 调用http.ListenAndServe，启动了一个端口为8989的http服务;

启动之后，我们便可测试该服务:

$ curl localhost:8989/ping
ping

以上就是通过net/http 库实现一个Web服务的实现。

Gin示例

Gin 本身也是基于原生的 net/http 基础之上再度进行的封装，下面我们同样也使用 Gin 来实现一个Web服务，示例如下:

import (
	"fmt"
	"github.com/gin-gonic/gin"
)

func main() {
	// 创建一个Gin Engine，本质上是一个Handler的实现
	app := gin.Default()

	// 注册中间件
	app.Use(func(ctx *gin.Context) {
		fmt.Println("one")
	})
	// 多个中间件会形成一条处理链，按顺序执行
	// 在处理中，决定继续还是终止
	app.Use(func(ctx *gin.Context) {
		fmt.Println("two")
	})

	// 注册一个请求Path和对应的处理函数
	app.POST("/ping", func(ctx *gin.Context) {
		ctx.Writer.Write([]byte("ping"))
	})

	// 启动服务
	if err := app.Run(":8989"); err != nil {
		panic(err)
	}
}

上述示例分别做了以下行为:

  1. 创建一个 gin.Engine，这个结构体实现了net/http 库中的Handler接口;
  2. 注册多个 gin.HandlerFunc函数到 HandlersChain作为中间件处理链，接到请求后会依次执行处理链;
  3. 注册一个POST请求以及对应的处理函数;
  4. 启动服务;

测试Gin服务:

$ curl -X POST localhost:8989/ping
ping

服务日志:

one
two
[GIN] 2023/08/17 - 20:25:57 | 200 |      104.25µs |       127.0.0.1 | POST     "/ping"

两者的关系

Gin 是在 Golang HTTP 标准库 net/http 基础之上的再封装，两者的交互边界如下图:

从上图可以看到 Gin 与 net/http的 职责分工:

  1. net/http 负责服务底层的运行以及请求连接的处理;
  2. net/http中的Handler就是Go中HTTP请求处理的抽象接口，而它的ServeHTTP()抽象方法表示了一次HTTP请求的处理;
  3. net/http 接到一个HTTP请求后，就会将请求通过Handler.ServeHTTP()转交给 Gin.Engine 来完成后续的操作;
  4. 由于Gin.Engine实现了ServeHTTP()方法，所以HTTP请求的处理、响应后续流程都由Gin来实现;

2.Gin核心组件

Engine

Engine 是 Gin 的核心组件，它实现了http.Handler接口的ServeHTTP()方法，因此可以注入到http.Server中;

// Handler 负责HTTP请求的处理实现
type Handler interface {
  // HTTP请求的执行函数
	ServeHTTP(ResponseWriter, *Request)
}
// Gin 处理HTTP请求的总入口
func (engine *Engine) ServeHTTP(w http.ResponseWriter, req *http.Request) {
  //...
}

结构如下:

type Engine struct {
  // 服务的路由组
  RouterGroup

  // ...

  // 不同情况下的处理链
  allNoRoute       HandlersChain
  allNoMethod      HandlersChain
  noRoute          HandlersChain
  noMethod         HandlersChain

  // 上下文gin.context 的复用对象池
  pool             sync.Pool

  // 用于管理不同HTTP方法(method)的路由树
  trees            methodTrees

RouterGroup

RouterGroup 路由组，Gin 支持通过相同的Path前缀，将多个Path组合到一个组中，其中一个组中的配置由该组的所有路由共享;

结构如下:

// 路由组
type RouterGroup struct {
  // 中间件处理链，只有本组的路由才会执行该处理链
  Handlers HandlersChain
  // 路由组的前缀path
  basePath string
  // 指向所关联的Gin Engine的指针
  engine   *Engine
  // 标识当前组是否为默认的根组，也就是直接通过Engine添加的路由
  root     bool
}

methodTrees

路由树，所有的 请求Path 以及对应的 HandlerFunc 都会保存到路由树中，根据不同的 Method 组成一前缀树;

// 路由树列表
type methodTrees []methodTree

// 路由树结构
type methodTree struct {
  // HTTP Method类型
  method string
  // 根节点
  root   *node
}

// 路由树节点结构
type node struct {
  // 当前节点相对路径
  path      string
  indices   string
  // 表示当前节点是否为通配符节点。通配符节点用于匹配路径中的通配符部分，如参数或通配符片段。
  wildChild bool
  // 节点类型
  nType     nodeType
  // 表示节点的优先级。优先级用于节点排序，以便在查找路由时快速找到匹配的节点。
  priority  uint32
  // 当前节点的子节点列表
  children  []*node // child nodes, at most 1 :param style node at the end of the array
  // 当前节点对应的处理函数链
  handlers  HandlersChain
  // 当前节点绝对路径，表示从树的根节点到当前节点的完整路径
  fullPath  string
}

路由树数组 trees：共有 9 棵路由树，对应于 9 种 http 方法. 路由树基于压缩前缀树实现;

Context

gin.Context 是一个请求的全局上下文对象，它允许我们在多个处理函数(中间件)中传递数据、管理执行流程、验证请求等操作，并且内置了HTTP请求的大量信息;

源码结构如下:

// Gin 用于处理每个HTTP请求的的全局上下文对象
type Context struct {
  writermem responseWriter

  // HTTP Request 指向原生的http.Request指针
  Request   *http.Request
  // 响应器，通过这个接口，可以将响应数据发送回客户端，里面内置了原生的http.ResponseWriter
  Writer    ResponseWriter

  // 请求参数，用于存储路由参数，这些参数从请求路径中提取而来。
  Params   Params
  // 中间件处理链（HandlersChain），表示与当前上下文关联的中间件列表。这些中间件将按顺序执行，处理请求和响应。
  handlers HandlersChain
  // 表示处理链中下一个要执行的处理函数的索引;
  index    int8
  // 完整请求路径
  fullPath string
  // 指向 Engine 结构体的指针，表示整个 Gin 框架的核心。
  engine       *Engine

  // 用于保证Keys数据安全的读写锁
  mu sync.RWMutex
  // 该请求上下文中传递的所有K-V数据
  Keys map[string]any

  // 错误列表，上下文的所有处理程序和中间件中的错误列表。
  Errors errorMsgs

  // 请求 Query参数
  queryCache url.Values

  // 请求表单参数
  formCache url.Values

  //...
}

HandlerFunc

HandlerFunc是 Gin 中的核心处理函数，请求的处理流程由N多个该函数进行链式处理，形成一条处理链，并且该函数持有请求的全局上下文;

type HandlerFunc func(*Context)

HandlersChain

HandlersChain是HandlerFunc的一组集合，也就是一条处理链;

type HandlersChain []HandlerFunc

1.创建Engine流程

2.注册middleware流程

无论是通过Engine还是RouterGroup注册 中间件，最终都是追加到RouterGroup的Handlers处理链中:

// 将中间件注册到根路由组
func (engine *Engine) Use(middleware ...HandlerFunc) IRoutes {
	engine.RouterGroup.Use(middleware...)
	return engine
}

// 将中间件注册到路由组的执行链中
func (group *RouterGroup) Use(middleware ...HandlerFunc) IRoutes {
	group.Handlers = append(group.Handlers, middleware...)
	return group.returnObj()
}

3.注册HandlerFunc

以注册一个Post请求为例，大致分为三大步骤:

  1. 根据路由组基础路径计算得出请求的完整路径，后续在路由树中会根据完整路径挂载;
  2. 根据路由组中公共的中间件链，拼接上请求的处理函数，得出该请求的完整处理函数链;
  3. 根据请求的Method，以及请求的完整路径，挂载到路由树中;

源码流程如下:

// 为当前路由组，注册一个Post请求
// relativePath: 请求路径
// handlers: 要注册的多个处理函数
func (group *RouterGroup) POST(relativePath string, handlers ...HandlerFunc) IRoutes {
  // 步入注册方法
  return group.handle(http.MethodPost, relativePath, handlers)
}


// 注册一个请求到当前路由组上
// httpMethod: HTTP请求类型
// relativePath: 请求路径
// handlers: 对应的处理函数列表
func (group *RouterGroup) handle(httpMethod, relativePath string, handlers HandlersChain) IRoutes {
  // 1. 根据路由组基础Path + 请求Path 得出请求的完整Path;
  absolutePath := group.calculateAbsolutePath(relativePath)
  // 2. 由路由组的公共处理链(中间件) + 请求处理函数列表，为当前请求组成一组完整的执行函数链;
  handlers = group.combineHandlers(handlers)
  // 3. 将请求挂载到路由树中
  group.engine.addRoute(httpMethod, absolutePath, handlers)
  return group.returnObj()
}

(1) 拼接绝对Path

// 计算relativePath的绝对路径
func (group *RouterGroup) calculateAbsolutePath(relativePath string) string {
  return joinPaths(group.basePath, relativePath)
}


// 将两个Path拼接到一起
// absolutePath: 路由组的基础Path
// relativePath: 请求Path
func joinPaths(absolutePath, relativePath string) string {
  // 请求Path为空，直接使用基础路径
  if relativePath == "" {
    return absolutePath
  }
  // 拼接
  finalPath := path.Join(absolutePath, relativePath)
  if lastChar(relativePath) == '/' && lastChar(finalPath) != '/' {
    // 填补请求路径末尾的'/'
    return finalPath + "/"
  }
  return finalPath
}

(2) 拼接处理函数

所谓的中间件(middleware)本质上也是一个HandlerFunc执行函数，所以我们需要将路由组中的所有中间件，拼接上注册的请求处理函数，组成一条完整的执行链;

// 将一个HandlersChain与当前路由组的Handlers处理链拼接到一起，返回一条新的处理链
// handlers 要拼接的处理函数
func (group *RouterGroup) combineHandlers(handlers HandlersChain) HandlersChain {
  // 计算处理链的总长度
  finalSize := len(group.Handlers) + len(handlers)
  // 判断处理函数是否过多
  assert1(finalSize < int(abortIndex), "too many handlers")
  // 分配一条新的处理链
  mergedHandlers := make(HandlersChain, finalSize)
  // 拷贝路由组中的处理链
  copy(mergedHandlers, group.Handlers)
  // 拷贝请求处理函数到处理链尾部
  copy(mergedHandlers[len(group.Handlers):], handlers)
  return mergedHandlers
}

(3) 将请求挂载到路由树

// 将请求添加到路由树中
// method: 请求类型
// path: 请求完整路径
// handlers: 请求对应的处理函数链
func (engine *Engine) addRoute(method, path string, handlers HandlersChain) {
	// 校验参数的合法性
	assert1(path[0] == '/', "path must begin with '/'")
	assert1(method != "", "HTTP method can not be empty")
	assert1(len(handlers) > 0, "there must be at least one handler")
	// debug模式下的日志输出
	debugPrintRoute(method, path, handlers)

	// 根据method 获取对应的路由树根节点，每种method都有一颗路由树
	root := engine.trees.get(method)
	if root == nil {
		// 初始化根节点
		root = new(node)
		root.fullPath = "/"
		engine.trees = append(engine.trees, methodTree{method: method, root: root})
	}
	// 从根节点检索路由树，将path和对应的处理函数挂载到合适的节点位置
	root.addRoute(path, handlers)
  //...
}

4.Engine启动流程

// 运行Gin 服务
// 将gin.Engine 作为http.Handler的实现，注册到http.Server中，实现HTTP的请求路由以及处理响应
func (engine *Engine) Run(addr ...string) (err error) {
  //...
  address := resolveAddress(addr)

  // 通过net/http 启动一个HTTP服务，而HTTP服务的实现由Engine实现
  err = http.ListenAndServe(address, engine.Handler())
  return

接下来我们步入到net/http的详细流程:

func ListenAndServe(addr string, handler Handler) error {
  // 实例化一个Server,表示HTTP服务，并且将Gin.Engine作为Handler的实现注入Server中;
  server := &Server{Addr: addr, Handler: handler}
  return server.ListenAndServe()
}
// 启动一个TCP服务,然后调用 Serve 来处理传入连接上的请求
func (srv *Server) ListenAndServe() error {
  if srv.shuttingDown() {
    return ErrServerClosed
  }
  // HTTP服务端口
  addr := srv.Addr
  if addr == "" {
    addr = ":http"
  }
  // 建立一个TCP服务侦听器
  ln, err := net.Listen("tcp", addr)
  if err != nil {
    return err
  }
  return srv.Serve(ln)
}
// 接收TCP侦听器传来的每一条连接，为每个连接创建一个新的服务 goroutine来处理连接的 解析与响应;
func (srv *Server) Serve(l net.Listener) error {
  // ...
  var tempDelay time.Duration // how long to sleep on accept failure
  // 创建一个新的上下文 ctx，并将其作为服务器上下文传递给后续连接的处理。
  ctx := context.WithValue(baseCtx, ServerContextKey, srv)
  // 循环处理
  for {
    // 基于IO多路复用Epoll，阻塞等待TCP连接
    rw, err := l.Accept()
    // 将上下文传递给连接
    connCtx := ctx
    //...
    // 基于接收到的TCP连接，创建一个新的连接
    c := srv.newConn(rw)
    c.setState(c.rwc, StateNew, runHooks)
    // 为每个连接创建一个协程进行处理
    go c.serve(connCtx)
  }
}

c.serve(connCtx)方法，最终会调用Handler.ServeHTTP()方法，将请求传递给gin.Engine.ServeHTTP()方法来处理请求;

5.Engine请求处理流程

  1. 从上一步启动流程得出，最终http.Server会针对于连接的TCP连接，进行解析得出http.ResponseWriter和http.Request两个属性，作为原生的HTTP请求与响应器;
  2. 然后传递给gin.Engine的ServeHTTP(w http.ResponseWriter, req *http.Request)方法;

接下来我们就继续步入后续的执行流程:

  1. 从对象池中获取一个context对象，并且清除原来的上下文信息;
  2. 调用handleHTTPRequest()方法进行请求的处理:
    1. 根据Method、Path从路由树中找到对应的路由节点;
    2. 将路由节点中的信息如路由参数、路由处理链等挂载到context中;
    3. 由context按照顺序执行请求的处理函数链;
    4. 如果没有找到请求路由节点，则返回404;
  3. 将使用完的context放回对象池;
// Gin 对于一个HTTP请求的处理的入口
func (engine *Engine) ServeHTTP(w http.ResponseWriter, req *http.Request) {
	// 从对象池中获取一个context进行复用
	c := engine.pool.Get().(*Context)
	// 将响应器设置到上下文
	c.writermem.reset(w)
	// 将请求信息赋值给上下文中的Request,方便使用
	c.Request = req
	// 重置context信息
	c.reset()

	// HTTP请求的处理
	engine.handleHTTPRequest(c)

	// 将context上下文 放回对象池中
	engine.pool.Put(c)
}
// HTTP请求的处理流程
func (engine *Engine) handleHTTPRequest(c *Context) {
  // 获取请求的method类型
  httpMethod := c.Request.Method
  // 获取请求完整Path
  rPath := c.Request.URL.Path

  //...

  // 获取路由树，并且遍历
  t := engine.trees
  for i, tl := 0, len(t); i < tl; i++ {
    if t[i].method != httpMethod {
      continue
    }
    // 根据method获取对应的路由树
    root := t[i].root
    // 根据请求path、请求参数在路由树中寻找路径节点
    value := root.getValue(rPath, c.params, c.skippedNodes, unescape)
    // 将路由节点的参数挂载到context.Params中
    if value.params != nil {
      c.Params = *value.params
    }
    // 执行请求对应的处理函数
    if value.handlers != nil {
      // 将处理函数链挂载到上下文中，由上下文开始执行处理
      c.handlers = value.handlers
      // 挂载请求完整路径
      c.fullPath = value.fullPath
      // 按照顺序来执行请求的处理函数链，直到全部执行完成，或者中途调用c.Abort()方法导致中断;
      c.Next()
      c.writermem.WriteHeaderNow()
      // 请求执行结束
      return
    }

    //...

    break
  }
  // 没有在路由树中找到对应的请求节点
  // 判断是否允许从其他method路由树中寻找匹配的请求节点
  if engine.HandleMethodNotAllowed {
    for _, tree := range engine.trees {
      if tree.method == httpMethod {
        continue
      }
      if value := tree.root.getValue(rPath, nil, c.skippedNodes, unescape); value.handlers != nil {
        c.handlers = engine.allNoMethod
        // 响应405 405 method not allowed
        serveError(c, http.StatusMethodNotAllowed, default405Body)
        return
      }
    }
  }
  c.handlers = engine.allNoRoute
  // 响应404 404 page not found
  serveError(c, http.StatusNotFound, default404Body)
}

4.Context

gin.Context 是 Gin 中非常核心的一个组件，该组件贯穿了一个HTTP请求，从开始到结束的完整流程，所以我们本小节针对该组件进行进一步剖析;

结构图所示:

gin.Context 的生命周期为一次完整的HTTP请求处理开始，直至HTTP请求结束，并且中途会在整个请求处理链HandlersChain中传递;核心字段如下:

  • Request: 本次HTTP请求的数据读取入口;
  • Writer: 本次HTTP请求的响应入口;
  • Params: 请求参数列表;
  • handlers: 本次请求要执行的处理函数链;
  • index: 表示执行的进度，即当前所执行的处理函数索引位置;
  • fullPath: 请求的完整路径;
  • engine: 指向Engine的指针;
  • Keys: 上下文中所保存的自定义数据;

源码结果如下:

// Gin 用于处理每个HTTP请求的的全局上下文对象
type Context struct {
  writermem responseWriter

  // HTTP Request 指向原生的http.Request指针
  Request   *http.Request
  // 响应器，通过这个接口，可以将响应数据发送回客户端，里面内置了原生的http.ResponseWriter
  Writer    ResponseWriter

  // 请求参数，用于存储路由参数，这些参数从请求路径中提取而来。
  Params   Params
  // 中间件处理链（HandlersChain），表示与当前上下文关联的中间件列表。这些中间件将按顺序执行，处理请求和响应。
  handlers HandlersChain
  // 即当前所执行的处理函数索引位置;
  index    int8
  // 完整请求路径
  fullPath string
  // 指向 Engine 结构体的指针，表示整个 Gin 框架的核心。
  engine       *Engine

  // 用于保证Keys数据安全的读写锁
  mu sync.RWMutex
  // 该请求上下文中传递的所有K-V数据
  Keys map[string]any

  // 错误列表，上下文的所有处理程序和中间件中的错误列表。
  Errors errorMsgs

  // 请求 Query参数
  queryCache url.Values

  // 请求表单参数
  formCache url.Values

  //...
}

1.复用策略

gin.Context 作为处理 http 请求的通用数据结构，不可避免地会被频繁创建和销毁. 为了缓解 GC 压力，gin 中采用对象池 sync.Pool 进行 gin.Context 的缓存复用，处理流程如下:

  1. 当HTTP请求到达时，从对象池中获取一个 Context,倘若池子已经没有可用的 Context,则通过pool.New方法构建出新的 Context对象;
  2. 当HTTP请求处理完后，将Context放入对象池，方便下次复用;

sync.Pool 并不是真正意义上的缓存，将其称为回收站或许更加合适，放入其中的数据在逻辑意义上都是已经被删除的，但在物理意义上数据是仍然存在的，这些数据可以存活两轮 GC 的时间，在此期间倘若有被获取的需求，则可以被重新复用.

源码实现如下:

type Engine struct {
  // context 对象池
  pool             sync.Pool
}

func New() *Engine {
  // ...
  // 注册Context构建函数
  engine.pool.New = func() any {
    return engine.allocateContext(engine.maxParams)
  }
  return engine
}
