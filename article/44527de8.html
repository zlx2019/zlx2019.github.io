Mysql学习之旅【五】锁

发表于2021-05-08|更新于2023-07-23|Mysql系列面试系列
|字数总计:6.1k|阅读时长:24分钟|阅读量:

1. 锁分类

按照粒度来分:

  1. 表级锁

    每次操作锁住整张表。锁定粒度大,发生锁冲突的概率最高，并发度最低。应用在MyISAM、 InnoDB、 BDB等存储引擎

  2. 行级锁

    每次操作锁住一行数据,粒度最小,并发度最高,应用在InnoDB引擎中

  3. 页级锁

    每次锁定相邻的一组记录,锁定粒度界于表锁和行锁之间，开销和加锁时间界于表锁和行锁之间，并发度-般。应用在BDB存储引擎中

行锁 表锁 页锁
MyISAM 支持
BDB 支持 支持
InnoDB 支持 支持

按照类型来分:

  1. 读锁(共享锁)

    针对同一份数据，多个读操作可以同时进行而不会互相影响。

  2. 写锁(独占锁)

    当前写操作没有完成前，它会阻断其他写锁和读锁。

按照性能来分:

  1. 乐观锁

    通过版本号控制,在数据更新同时比对版本号是否一致,如果不一致则意味着发生了锁冲突,提交失败

  2. 悲观锁

    在对一条数据修改的时候,为了避免同时被其他人修改,在修改数据之前先锁定，再修改的控制方式。共享锁和排他锁是悲观锁的不同实现，但都属于悲观锁范畴

2. 表级锁

#给表加上表锁 可选 读锁 or 写锁
lock table `tableName` READ|WRITE
#释放锁
UNLOCK TABLES

A会话 对该表设置 表级READ锁后,A会话和其他会话都可以进行读操作,但A会话不允许进行增删改操作,会报错,其他会话进行增删改查将会进入阻塞状态。

A会话对该表设置 表级WRITE锁后,A会话可以对其做增删改查操作,其他会话对表的所有操作都将进入阻塞状态。

总结: 表级读锁,会阻塞写操作,但不会阻塞读操作,而写锁会阻塞其他连接会话的所有操作

3. 行级锁

行锁是通过给索引上的索引项加锁来实现的。只有通过索引条件检索数据，InnoDB才使用行级锁,否则，InnoDB将使用表锁.

读锁

加锁语法: select … lock in share mode

如:

SELECT * from account WHERE `name` = 'A' lock in SHARE MODE

Example:

  1. 给一行数据加上行锁,再起一个会话去修改

    #开启一个事务
mysql> start TRANSACTION;
Query OK, 0 rows affected (0.00 sec)

#查询 并且加上行锁
mysql> SELECT * from account WHERE `name` = 'A' lock in SHARE MODE
    -> ;
+------+--------+
| name | money  |
+------+--------+
| A    | 100.00 |
+------+--------+
1 row in set (0.00 sec)

  2. 另起一个会话去修改name为A的数据

    mysql> update account set money = 90 where name = 'A'; #这里将会一直阻塞,直至超时,抛出下面的异常消息
1205 - Lock wait timeout exceeded; try restarting transaction
mysql>

  3. 我们试着用查询语句去查这条数据

    mysql> mysql> select * from account;
+------+--------+
| name | money  |
+------+--------+
| A    | 100.00 |
| B    | 100.00 |
+------+--------+
2 rows in set (0.00 sec)

  4. 可以看到查询语句是没有问题的!

  5. 那么接下来我们将事务提交。

    mysql> commit;
Query OK, 0 rows affected (0.00 sec)

  6. 再次使用其他会话去修改。

    mysql> update account set money = 90 where name = 'A';
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0
 
mysql> select * from account;
+------+--------+
| name | money  |
+------+--------+
| A    | 90.00  |
| B    | 100.00 |
+------+--------+
2 rows in set (0.00 sec)

  7. 修改完成!

总结: 行级的读锁,同样不会阻塞其他会话的读操作,但是会阻塞修改操作

写锁

加锁语法: 在SQL末尾加上 for update

InnoDB 引擎默认会在update、delete语句加上for update。

Example:

  1. 开启一个事务,修改某一条数据,默认会加上行级写锁

    mysql> start transaction;
Query OK, 0 rows affected (0.00 sec)
 
mysql> update account set money=200 where name = 'A';
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0

  2. 开启新的会话去读取这条数据

    mysql> select * from account where name = 'A';
+------+-------+
| name | money |
+------+-------+
| A    | 90.00 |
+------+-------+
1 row in set (0.00 sec)

    可以看到,读取是没有问题的!

  3. 接下来试试修改

    mysql> update account set money = 300 where name = 'A';
...
1205 - Lock wait timeout exceeded; try restarting transaction

    修改则会阻塞在这里…,直到释放锁,或者超时

  4. 把事务提交

    mysql> commit;
Query OK, 0 rows affected (0.00 sec)

  5. 再次修改

    mysql> update account set money = 300 where name = 'A';
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0

    修改完成!

    总结:行级的写锁,本身是会阻塞其他会话的任何操作,但是上面测试却发现并不会阻塞其他会话的读操作,这是因为Mysql使用了MVCC机制来提高了读写的并发能力,所以Mysql的写锁只会阻塞除了当前会话以外的写操作

4. 行锁模式

假设现在有如下数据表:

mysql> select * from users;
+----+--------+
| id | name   |
+----+--------+
|  1 | 小张 |
|  5 | 小兰 |
| 10 | 小绿 |
| 15 | 小王 |
| 20 | 小曾 |
+----+--------+
5 rows in set (0.00 sec)

id为主键索引。

recordlock

记录锁顾名思义就是对记录进行加锁,也即是我们俗称的行锁。事实上它锁定的是该行的索引记录,而不是锁的真正的数据。

  1. 如果条件中的字段是非主键索引,会先在自己的索引上加锁,再去主键索引上加锁。

  2. 如果表上没有索引(包括没有主键)，则会使用隐藏的主键索引进行加锁。

  3. 如果Sql语句导致索引失效,或者说条件不涉及到索引,行锁会直接退化为表锁。

简单来说就是根据索引查询一条已经存在的记录时,会使用该模式,如select * from users where id = 5 for update。下面看案例

  • 案例1:

    使用行锁查询,根据id主键查询一条已经存在的记录.

#开启事物
mysql> begin; 
Query OK, 0 rows affected (0.00 sec)
 
# 查询并且添加行锁
mysql> select * from users where id = 5 for update;
+----+--------+
| id | name   |
+----+--------+
|  5 | 小兰 |
+----+--------+
1 row in set (0.00 sec)

    然后可以通过 data_locks 查看锁信息，SQL 如下：

    结果主要包含引擎、库、表等信息，咱们需要重点关注以下几个字段:

    • INDEX_NAME：锁定索引的名称 PRIMARY 表示使用的是主键索引。
    • LOCK_TYPE: 锁的类型。对于 InnoDB,允许的值为 RECORD(行锁)和TABLE(表锁)。
    • LOCK_MODE： 锁的模式。
      • S: 共享锁，允许多个用户读取同一行记录，但不允许其他用户写入该行记录。
      • X: 排他锁，允许一个用户对行记录加写锁，其他用户无法写入或读取该行记录。
      • IS: 意向共享锁，允许多个用户读取同一行记录，但不允许其他用户加写锁。
      • IX: 意向排他锁，允许一个用户加写锁，其他用户既不能写入也不能读取。
      • REC_NOT_GAP: 单行锁定,不存在间隙锁。因此，它不会阻止列中的其他行被更新，只会阻止表中锁定的单行被更新。
      • Gap: 间隙锁，用于锁定非索引记录的范围，以防止并发事务在记录之间插入新记录。
    • LOCK_DATA: 锁关联的数据，对于 InnoDB,当LOCK_TYPE是RECORD(行锁),则显示实际的值。当锁在主键索引上时，则值是锁定记录的主键值。当锁是在辅助索引上时，则显示辅助索引的值，并附加上主键值。

gaplock

间隙锁,顾名思义锁间隙，不锁记录。锁间隙的意思就是锁定某一个范围，间隙锁又叫 gap 锁，其不会阻塞其他的 gap 锁，但是会阻塞插入间隙锁，这也是用来防止幻读的关键。

    mysql> select * from users where id < 20 and id > 5 for update;

| id | name   |
+----+--------+
|  6 | 小赵   |
| 10 | 小绿   |
| 15 | 小王   |
+----+--------+
3 rows in set (0.00 sec)

    然后查看锁信息:

    *************************** 5. row ***************************
               ENGINE: INNODB
       ENGINE_LOCK_ID: 5258646984:83:4:7:5670882160
ENGINE_TRANSACTION_ID: 56354

             EVENT_ID: 73

    如下所示: 会一直阻塞着,除非锁释放后,或者超时。

    mysql> insert into users(id,name)values(11,"小刘");
1205 - Lock wait timeout exceeded; try restarting transaction

next-keylock

Next-KeyLock是结合了GapLock和RecordLock 的一种锁定算法，其主要目的是为了解决幻读问题。

在 InnoDB 默认的隔离级别 REPEATABLE-READ 下，行锁默认使用的算法就是 Next-KeyLock。但是，如果操作的索引是唯一索引或主键，InnoDB 会对 Next-Key Lock 进行优化，将其降级为 Record Lock，即仅锁住索引本身，而不是范围。

由于主键也是一种唯一索引，所以我们可以这么说：Record Lock 是基于唯一索引的，而 Next-Key Lock 是基于非唯一索引的。

现在有如下数据:

对age字段添加非唯一索引.

mysql> select * from users;
+----+--------+------+
| id | name   | age  |
+----+--------+------+
|  1 | 小张   |   15 |
|  5 | 小兰   |   16 |
|  6 | 小赵   |   20 |
| 10 | 小绿   |   27 |
| 15 | 小王   |   29 |
| 20 | 小曾   |   35 |
+----+--------+------+
6 rows in set (0.00 sec)

然后对age为20的记录进行加锁:

mysql> begin;
Query OK, 0 rows affected (0.00 sec)

mysql> select * from users where age = 20 for update;
+----+--------+------+
| id | name   | age  |
+----+--------+------+
|  6 | 小赵   |   20 |
+----+--------+------+
1 row in set (0.00 sec)

这个时候,mysql会产生next-key lock的加锁模式,会对age=20记录的前后间隙分别加锁.

age为(16 - 20)和(20 - 27)的间隙全部被加锁。

如下: 添加age为18、25全部被阻塞到超时.

mysql> INSERT into users(id,`name`,age)values(14,"小小",18);
1205 - Lock wait timeout exceeded; try restarting transaction

mysql> INSERT into users(id,`name`,age)values(31,"小小",25);
1205 - Lock wait timeout exceeded; try restarting transaction

而添加age为 13、30的数据全部成功.

mysql> INSERT into users(id,`name`,age)values(14,"小小",13);
Query OK, 1 row affected (0.00 sec)
mysql> INSERT into users(id,`name`,age)values(30,"小小",30);
Query OK, 1 row affected (0.00 sec)

总结:

Recordlock 是针对唯一索引的加锁方式。而gaplock和next-keylock是针对非唯一索引加锁方式。如果不指定索引,则会直接退化到表锁,会全部阻塞住。

对于唯一索引来说,gap-lock(间隙锁)上下边界的记录都不会锁住,因为是唯一索引,是不需要加锁的。

案例

Innodb默认使用的写锁为next-key-lock,但是它会随着索引而优化。

常见的写锁有record-lock、gap-lock、next-key-lock。

定义数据表和数据

mysql> select * from t;
+----+------+------+
| a  | b    | c    |
+----+------+------+
|  1 |    1 |    1 |
|  5 |    5 |    5 |
| 10 |   10 |   10 |
| 14 |   14 |   14 |
| 20 |   20 |   20 |
| 23 |   23 |   23 |
+----+------+------+
6 rows in set (0.00 sec)

其中a是主键索引、b是非唯一索引、c没有索引。

1. 唯一索引精确查询

现在我们通过主键索引,进行一次等值查询:

# 关闭自动提交
mysql> set autocommit = 0;
Query OK, 0 rows affected (0.00 sec)
# 开启事物
begin;

mysql> select * from t where a = 5 for update;
+---+------+------+
| a | b    | c    |
+---+------+------+
| 5 |    5 |    5 |
+---+------+------+
1 row in set (0.00 sec)

注意,先不要提交事物哦.我们来分析一下加锁情况:

  1. for update会进行加X锁,也就是写锁.
  2. 然后默认会加next-key-lock,加锁范围是索引[1~5]。也就是目标索引至前一个索引之间的范围。
  3. 但是由于条件a字段是唯一索引(主键索引),所以锁会优化为record-lock。加锁范围会优化成[5]。

其实很简单,我们可以想一下,既然条件字段已经是唯一的了,那就不可能存在重复的记录,所以只需要防止它被修改即可.不存在幻读的情况。

然后我们来看一下锁的信息:

通过select * from performance_schema.data_locks\G;语句查看锁信息:

           INDEX_NAME: PRIMARY
OBJECT_INSTANCE_BEGIN: 5670877208
            LOCK_TYPE: RECORD
            LOCK_MODE: X,REC_NOT_GAP
          LOCK_STATUS: GRANTED
            LOCK_DATA: 5
2 rows in set (0.00 sec)

显然可以看到和我们推断的一模一样.

INDEX_NAME: PRIMARY: 表示锁定的索引。

LOCK_TYPE: RECORD:表示为行级锁。

LOCK_MODE: X,REC_NOT_GAP:表示为写锁 + 记录锁。

LOCK_DATA: 5: 表示加锁的范围,当锁在主键索引上时，则值是锁定记录的主键值。

这种情况下,不会阻塞任何新增数据.只会阻塞对a=5这条记录的修改操作。

2. 普通索引精确查询

通过普通索引b,进行精确查询。

mysql> begin;
Query OK, 0 rows affected (0.00 sec)

mysql> select * from t where b = 5 for update;
+---+------+------+
| a | b    | c    |
+---+------+------+
| 5 |    5 |    5 |
+---+------+------+
1 row in set (0.00 sec)

分析加锁情况:

  1. for update 会加X锁(写锁)。
  2. 然后加next-key-lock锁,加锁范围为[1~5].
  3. 由于b字段不是唯一索引,所以不会进行锁的优化,加锁范围不变,还是[1~5].
  4. 由于b字段不是唯一索引,还会添加一把gap-lock锁,锁的范围与netx-key-lock相反,是目标索引与下一行记录索引之间的间隙,范围为[5~10].

查看锁的信息:

           INDEX_NAME: b_indx
OBJECT_INSTANCE_BEGIN: 5670877896
            LOCK_TYPE: RECORD
            LOCK_MODE: X,GAP
          LOCK_STATUS: GRANTED
            LOCK_DATA: 10, 10
4 rows in set (0.00 sec)

LOCK_MODE: X,GAP: 写锁+GAP(间隙锁).

LOCK_DATA: 10, 10: 加锁范围.

尝试插入新数据:

mysql> insert into t(a,b,c)values(2,2,2);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(2,1,2);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(2,5,2);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(2,9,2);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(2,10,2);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

分别尝试在b字段插入2、1、5、9、10数值,都会陷入阻塞,因为它们都处于nextlock[1~5]和gaplock[5~10]的加锁范围内.

并且修改也会导致阻塞。

只有插入的值,不存在锁的范围内才可以插入:

mysql> insert into t(a,b,c)values(11,11,11);
Query OK, 1 row affected (0.00 sec)

注意: 这个时候被锁住的索引的上边界与下边界是可以进行修改的,gap锁主要是防止幻读。

如下所示: 分别对索引值为1和10的记录进行修改

mysql> update t set c = 11 where b = 1;
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0

mysql> update t set c = 100 where b = 10;
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0

mysql> select * from t;
+----+----+-----+
| a  | b  | c   |
+----+----+-----+
|  1 |  1 |  11 |
|  5 |  5 |   5 |
| 10 | 10 | 100 |
| 11 | 11 |  11 |
| 14 | 14 |  14 |
| 20 | 20 |  20 |
| 23 | 23 |  23 |
+----+----+-----+
7 rows in set (0.00 sec)

但是索引值为5的记录是不能被修改的.

mysql> update t set c = 55 where b = 5;
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

3. 唯一索引精确查询不存在的记录1

通过唯一索引查询一条不存在的记录。

条件:

  1. 查找的主键值 > 当前已有的最大的主键值
  2. 记录且不存在.
mysql> begin;
Query OK, 0 rows affected (0.00 sec)

mysql> select * from t where a = 25 for update;
Empty set (0.01 sec)

可以看到,并没有查到任何记录.然后我们来看一下锁的信息:

          INDEX_NAME: NULL
OBJECT_INSTANCE_BEGIN: 5668775032
            LOCK_TYPE: TABLE
            LOCK_MODE: IX
          LOCK_STATUS: GRANTED
            LOCK_DATA: NULL
3 rows in set (0.00 sec)

可以看到,这种情况直接升级为表锁了LOCK_TYPE: TABLE.并且显示没有锁到任何数据.

表面上好像没有加锁,实际已经加了一把巨大的锁!!!

这种情况下,会默认形成一把next-key-lock锁,并且锁的范围是[当前最大的a的值 ~ 无上限]

简单来说,目前a字段最大的值是23,如果此时插入任何一条a>23的记录,全部都会阻塞住.如下所示:

mysql> insert into t(a,b,c)values(50,11,11);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(24,11,11);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(25,11,11);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(99999,11,11);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

这种情况只能插入a小于当前最大记录a的值才允许插入:

mysql> insert into t(a,b,c)values(18,11,11);
Query OK, 1 row affected (0.00 sec)

4. 唯一索引精确查询不存在的记录2

条件:

  1. 记录不存在
  2. 唯一索引的值,介于已有索引最大值与最小值之间

先看一下目前的数据:

+----+------+------+
| a  | b    | c    |
+----+------+------+
|  1 |    1 |    1 |
|  5 |    5 |    5 |
| 10 |   10 |   10 |
| 11 |   11 |   11 |
| 14 |   14 |   14 |
| 20 |   20 |   20 |
| 23 |   23 |   23 |
+----+------+------+

我们查询主键a为15的记录.

mysql> begin;
Query OK, 0 rows affected (0.00 sec)

mysql> select * from t where a = 15 for update;
Empty set (0.00 sec)

显然没有查到,然后看一下锁信息:

           INDEX_NAME: PRIMARY
OBJECT_INSTANCE_BEGIN: 5670877208
            LOCK_TYPE: RECORD
            LOCK_MODE: X,GAP
          LOCK_STATUS: GRANTED
            LOCK_DATA: 20
2 rows in set (0.00 sec)

  1. 默认加的next-key-lock锁,范围为[14~15]

  2. gap-lock锁,范围为[15~20].

  3. 所以加锁范围故为: [14~20].

可以看到,尽管使用的是唯一索引,但是由于记录不存在,导致锁退化为了gap-lock.成为了间隙锁.

mysql> insert into t(a,b,c)values(18,11,11);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(15,11,11);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction


#OK
mysql> insert into t(a,b,c)values(13,11,11);
Query OK, 1 row affected (0.01 sec)

注意哦： 这种情况下,是可以对间隙两侧的记录进行修改的:

mysql> update t set c = 200 where a=20;
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0

mysql> update t set c = 140 where a=14;
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0

哈哈哈,是不是很头大,明明退化为gap-lock了,但是居然和之前根据普通索引精确查询引起的gap-lock时的结果不同。

仔细想想也很简单,虽然退化为了gap-lock锁,但是本质上我们的查询条件还是唯一索引,也就是说间隙两侧的记录肯定是存在的,既然已经存在并且还是唯一索引的,也就意味着不会出现重复的记录,所以即便是可以修改也是没问题的.之前根据普通索引查询引起的gap-lock是因为是非唯一索引,不能保证会不会有相同索引的记录随时插入进来,所以要进行加锁.

5. 普通索引精确查询不存在的记录

查看目前数据:

+----+------+------+
| a  | b    | c    |
+----+------+------+
|  1 |    1 |    1 |
|  5 |    5 |    5 |
| 10 |   10 |   10 |
| 11 |   11 |   11 |
| 14 |   14 |   14 |
| 20 |   20 |   20 |
| 23 |   23 |   23 |
+----+------+------+

精确查询b=为13的记录.

mysql> begin;
Query OK, 0 rows affected (0.00 sec)

mysql> select * from t where b = 13 for update;
Empty set (0.00 sec)

查看锁信息:

           INDEX_NAME: b_indx
OBJECT_INSTANCE_BEGIN: 5670877208
            LOCK_TYPE: RECORD
            LOCK_MODE: X,GAP
          LOCK_STATUS: GRANTED
            LOCK_DATA: 14, 14
2 rows in set (0.00 sec)
  1. next-key-lock锁范围为[11~13]
  2. gap-lock锁范围为[13~14]
mysql> insert into t(a,b,c)values(13,13,13);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(13,11,13);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(13,14,13);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

然后尝试对被锁住的索引边界进行修改:

mysql> update t set c = 110 where b=11;
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0

mysql> update t set c = 140 where b=14;
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0

同样也是可以修改的~

6. 唯一索引范围查询

+----+------+------+
| a  | b    | c    |
+----+------+------+
|  1 |    1 |    1 |
|  5 |    5 |    5 |
| 10 |   10 |   10 |
| 11 |   11 |   11 |
| 14 |   14 |   14 |
| 20 |   20 |   20 |
| 23 |   23 |   23 |
+----+------+------+
mysql> begin;
Query OK, 0 rows affected (0.00 sec)

mysql> select * from t where a between 10 and 22 for update;
+----+------+------+
| a  | b    | c    |
+----+------+------+
| 10 |   10 |   10 |
| 11 |   11 |   11 |
| 14 |   14 |   14 |
| 20 |   20 |   20 |
+----+------+------+
4 rows in set (0.00 sec)
           INDEX_NAME: PRIMARY
OBJECT_INSTANCE_BEGIN: 5670877896
            LOCK_TYPE: RECORD
            LOCK_MODE: X,GAP
          LOCK_STATUS: GRANTED
            LOCK_DATA: 23

其实就如同,唯一索引查询不存在的记录一般,会退化为gap-lock。

  1. next-key-lock锁范围为: [10~22]
  2. gap-lock锁范围为:[22~23]
  3. 所以会为[10~23]区间进行间隙加锁.
mysql> insert into t(a,b,c)values(11,14,13);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

mysql> insert into t(a,b,c)values(22,14,13);
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction
mysql> insert into t(a,b,c)values(9,14,13);
Query OK, 1 row affected (0.00 sec)

mysql> insert into t(a,b,c)values(24,14,13);
Query OK, 1 row affected (0.00 sec)

唯一索引区间范围查询导致的gap-lock这种情况,只能对上间隙边界的记录进行修改操作:

比如对索引为23这条记录进行修改:

mysql> update t set c = 12312 where a = 23;
Query OK, 1 row affected (0.00 sec)
Rows matched: 1  Changed: 1  Warnings: 0

修改成功,但是对下间隙边界10就无法修改了:

mysql> update t set c = 12312 where a = 10;
ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction

7. 总结:

  • Innodb默认写锁使用的是next-key-lock.它默认会对目标索引值至比他小的上一条索引值之间的间隙进行加锁。

    例如有索引: 1,4,8,12,34。如果锁定的索引值为8,那么next-key-lock锁定的范围就是4-8。

  • 对唯一索引进行精确查询时,next-key-locK会优化为record-lock,会将锁的范围缩小到某行记录。

    如锁定的索引值为8,那么锁定的范围就是8。

  • 对普通索引进行精确查询时,使用的是gap-lock(间隙锁),会对目标索引的上间隙和下间隙全部锁定。如索引包含3、7、9,对7进行精确匹配,那么会锁定[3~7~9]之间的所有间隙(包括3、7、9这三行记录).

  • 对唯一索引进行精确查询时,如果查询的记录不存在,锁则会退化为gap-lock(间隙锁).查找的记录索引值的上间隙与下间隙加锁。如索引包含4、8、13,查询条件为11,则会对[8~13]间隙进行加锁。注意: 如果查询的记录索引值已经超过了目前最大的索引值,则会对当前最大的索引值后面的所有间隙进行加锁.

  • 唯一索引退化为gap-lock后是不会锁定上下边界,可以修改上下边界记录。以案例4为证。

  • 非唯一索引退化为gap-lock后会锁定上下边界,因为不能保证唯一性,所以不能插入新的和边界索引值一样的记录,但是同样是可以修改。以案例2和5为证。
  • 唯一索引如果因为范围查询而退化为gap-lock,那么会锁住下边界(小)而不会锁住上边界(大)。以案例6为证.
