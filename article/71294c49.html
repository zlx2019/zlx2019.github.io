GRPC-Go语言篇章

1. gRPC是什么

gRPC全称(Google Remote Procedure Call).是Google公司推出的一款开源的RPC框架,依赖于HTTP2.0、协议缓冲区(Protobuf)等技术.具有高性能、跨语言通信等特点.

2. gRPC的优势

  • 高性能

    使用Protobuf 和HTTP/2，gRPC服务提供了高达10倍于REST+JSON交互的高性能和API保护。通过使用服务器推送、多路复用和报头压缩，HTTP/2为gRPC服务提供了高性能排名。虽然多路复用消除了线头延迟，但服务器推送使HTTP/2有可能在需要之前从服务器推送材料到客户端。使用HTTP/2，消息被更有效地压缩，从而使gRPC服务的加载速度更快。

  • 流媒体

    流媒体gRPC服务的服务描述已经包括客户端或服务器端的流媒体原则。因此，构建gRPC客户端和流式服务变得非常容易。

  • 代码生成

    gRPC客户端和gRPC服务器程序的代码生成是gRPC网络方法的关键组成部分。为了从.proto 文件中生成代码，gRPC模块采用了.protoc 编译器。Protobuf 格式是通过gRPC中的代码生成来控制的，它被用来定义数据格式和应用端点。它可以创建客户端的网络存根和服务器端的骨架，这减少了在gRPC服务中设计各种服务的程序所需的时间。

  • 跨语言

    众多的系统和编程语言，如Java, Ruby, Go、C#等，都受到gRPC资源和库的支持。通过这些编程语言，开发人员可以创建高性能的应用程序，同时利用gRPC的完全跨平台兼容性。这要归功于Protobuf 二进制接线形式和几乎所有系统的有效代码生成。

  • 安全性

    gRPC使用HTTP/2通过TLS端到端加密会话来确保API的安全性。gRPC提倡采用SSL/TLS在服务器和gRPC客户端之间进行数据加密和认证。

3. 安装所需工具

  1. 如果你使用的是JetBrains公司的Goland请先确保已经安装了如下插件:

    • gRPC
    • Protocol Buffers

    高版本的Goland一般会自带这两个插件.

  2. 安装protoc

    定义好的proto数据格式是无法直接使用的,需要经过protoc编译器,来转换成各种语言的数据结构才可以实现通用.

    安装包下载地址

    这里推荐使用v3版本的 如 v3.20.0

    然后将解压后的bin目录添加到环境变量中即可.

    当然,如果你是Mac用户,可以直接使用brew进行下载安装

    brew install protobuf

    测试是否安装完成

    % protoc --version
libprotoc 3.20.0

  3. 安装Go集成protobuf与grpc插件

    $ go install google.golang.org/protobuf/cmd/protoc-gen-go@v1.28
$ go install google.golang.org/grpc/cmd/protoc-gen-go-grpc@v1.2

    如果已经安装过,则不再需要安装.

4. 编写一个简单的案例

首先建立Go项目

go mod init

然后引入grpc的依赖

go get google.golang.org/grpc

简述一下目录结构

因为只是一个简单的案例,所以只需要创建如下4个文件夹即可:

% mkdir server
% mkdir client
% mkdir proto
% mkdir pb

然后结构如下:

├── client
├── go.mod
├── pb
└── proto
└── server

client与server分别用来编写Go相关代码,这无需多说.

proto用来存放我们定义好的xxx.proto文件.

pb主要用来存放 proto文件编译后的Go语言结构体.

定义proto

// 定义proto的版本
syntax = "proto3";

// proto文件的 package,其他proto文件引用时使用
package hello;

// 指定生成go语言后的 package名,package为pb
option go_package = "./;pb";

// 定义实体,用于作为接口请求参数
message Request{
  string req_message = 1;
}

// 定义实体,用于作为接口返回值
message Response{
  string resp_message = 1;
}

// 定义服务,服务内可以定义多个接口
service HelloService{

  // 定义接口
  // 接口格式为: rpc 接口名(请求参数)returns(返回参数)
  rpc SendMessage(Request)returns(Response);
}
  • syntax = "proto3"表示正在使用proto3的语法,如果不指定则默认为proto2.
  • 通过message关键字定义实体.如Go中的struct,或者Java中的class.
  • 通过service定义服务,而服务内可以定义多个rpc接口.
  • 每个message中的参数,必须指定从1起始的序号,主要为了序列化一致性。
  • 如果你要编译成Go语言,go_package是必不可少的,它指定了Go文件的目录生成,最后的.hello就是Go文件的package。

编译proto文件

定义好Hello.proto文件后,就可以通过protc编译为对应的go数据结构了.

通过如下命令进行编译：

protoc --go_out=./pb  --go-grpc_out=./pb  ./proto/*.proto

这是最简单的一条编译命令,解释如下:

--go_out=./pb: 生成的Go文件放在当前目录的pb文件下.

--go-grpc_out=./pb:生成的GoGrpc文件放在当前目录的pb文件下.

./proto/*.proto: 要编译的proto文件.

注意: 这里的输出指令是会和proto文件中的go_package搭配使用的,如这里是指定了生成在pb文件下,结合文件中的option go_package = "com.zero.hello" 最后会生成在./pb/com/zero/hello文件中.

命令执行后的目录结构为:

├── go.mod
├── go.sum
├── pb
│     ├── Hello.pb.go
│     └── Hello_grpc.pb.go
├── proto
│   └── Hello.proto

这样一个简单的proto文件近编译完成了.

由于命令过于复杂,这里建议写到一个可执行文件内方便使用,如Makefile或shell文件.

build:
	 protoc --go_out=./pb  --go-grpc_out=./pb  ./proto/*.proto

服务端

Hello.proto文件中,定义了一个服务(service)叫HelloService,被编译之后,会在Hello_grpc.pb.go文件中有一个名为UnimplementedHelloServiceServer的结构体,结构体的名字是Proto中的service加上一些规范而来.

所以服务端第一步就是要定义一个结构体,然后重写这个UnimplementedHelloServiceServer的SendMessage()方法.

  1. 定义一个结构体,继承proto生成的UnimplementedHelloServiceServer结构体.
// HelloServer GO服务端需要定义一个结构体与 Proto中的Service一一对应
// 然后实现proto编译后生成的一个结构体,然后重写它的方法
type HelloServer struct {
	pb.UnimplementedHelloServiceServer
}
  1. 然后重写Hello.proto中定义的SendMessage()接口.

这里暂且就先简单处理一下即可.

// SendMessage 重写proto中定义的rpc SendMessage方法
func (server *HelloServer) SendMessage(ctx context.Context, req *pb.Request) (*pb.Response, error) {
	// 获取到请求参数中的Message
	reqMsg := req.ReqMessage
	fmt.Printf("收到gRPC Client 请求, message: %s \n", reqMsg)
	// 然后构建一个响应参数
	respMsg := &pb.Response{RespMessage: "gRpc Server的返回参数"}
	// 然后将参数返回,并且没有err,所以直接返回nil
	return respMsg, nil
}
  1. 开启服务
func main() {
	// 1.创建一个tcp服务
	listen, err := net.Listen("tcp", ":7777")
	if err != nil {
		fmt.Println(err)
		return
	}
	// 2.创建一个grpc服务
	grpcServer := grpc.NewServer()
	// 3.创建实现完成的接口服务
	helloService := &HelloServer{}
	// 4.将接口服务注册到grpc服务中
	pb.RegisterHelloServiceServer(grpcServer, helloService)
	fmt.Printf("gRPC Server Start listen: %s \n", listen.Addr())
	// 5.借助tpc服务 开启grpc服务器
	err = grpcServer.Serve(listen)
	if err != nil {
		fmt.Printf("GRPC服务启动失败 原因: %s \n", err.Error())
	}
}

然后直接运行即可.

gRPC Server Start listen: [::]:7777

客户端

func main() {
	//1. 创建一个GRPC客户端,连接到服务端,关闭安全认证,否则无法连接
	conn, err := grpc.Dial("localhost:7777", grpc.WithTransportCredentials(insecure.NewCredentials()))
	if err != nil {
		fmt.Printf("gRPC Client Conn Server err: %s \n", err.Error())
		return
	}
	// 2. 创建HelloService的客户端
	client := pb.NewHelloServiceClient(conn)

	// 3. 准备需要的参数
	request := &pb.Request{ReqMessage: "i is Client"}
	// 3. 调用接口
	response, err := client.SendMessage(context.Background(), request)
	if err != nil {
		fmt.Printf("gRPC 接口调用失败 err: %s \n", err.Error())
		return
	}
	fmt.Printf("gRPC 接口调用成功 resp: %s", response.RespMessage)
}

5. 流式请求

向上边的案例,实现的是一种最基本的请求,即一个请求对应一个响应.

当然gRPC也是支持流式数据传输.

你可以入参为一个数据流,也可以出参为一个数据流,也可以入参和出参都是一个数据流。下面会注意讲解.

1. 请求参数为流

执行流程:

  1. 客户端向服务端发起请求,然后不断发送流数据
  2. 服务端接收请求,不断接收客户端的流数据
  3. 当服务端接收完所有的流数据或者未接收数据但是发生错误,向客户端响应数据,表示已完成或者已结束
  4. 客户端收到响应,请求结束.

定义一个新的produce.proto文件

syntax = "proto3";
package payload;
option go_package = "./;payload";

// 单个请求参数流消息
message ProduceRequest{
  // 产品id
  sint32 produce_id = 1;
}

// 响应参数
message ProduceReply{
  // 是否已读取完请求流的参数
  bool completed = 1;
  // 响应消息
  bool msg = 2;
}

service ProduceService{
  // 定义一个请求参数为流、响应参数为消息的接口
  rpc SendStream(stream ProduceRequest)returns(ProduceReply);
}

然后进行编译.

protoc --go_out=./pb --go-grpc_out=./pb  ./proto/*.proto

我们来看一下,编译后的接口有何不同:

先看一下服务端的接口变化

// 定义一个请求参数为流、响应参数为消息的接口
	SendStream(ProduceService_SendStreamServer) error

Proto中定义的SendStream(stream ProduceRequest)参数已经被编译成了ProduceService_SendStreamServer接口.

服务端就是通过这个ProduceService_SendStreamServer接口来持续读取来自于客户端的数据流,并且响应客户端.下面来看一下这个接口的结构.

type ProduceService_SendStreamServer interface {
	SendAndClose(*ProduceReply) error// 响应客户端信号,表示数据已经读取完毕,然后关闭连接。
	Recv() (*ProduceRequest, error) //读取客户端发送的的数据
	grpc.ServerStream
}

再来看一下客户端接口的变化

type ProduceServiceClient interface {
	// 定义一个请求参数为流、响应参数为消息的接口
	SendStream(ctx context.Context, opts ...grpc.CallOption) (ProduceService_SendStreamClient, error)
}

客户端的发送接口,显然已经不需要入参,而是先和客户端建立连接,然后拿到返回值中的ProduceService_SendStreamClient对象,然后通过该对象向服务端发送数据流。结构如下:

type ProduceService_SendStreamClient interface {
	Send(*ProduceRequest) error					//向客户端发送数据
	CloseAndRecv() (*ProduceReply, error)//向客户端信号,表示数据已经发送完成,然后等待服务端的响应。
	grpc.ClientStream
}

接下来开始实现服务端与客户端

服务端实现

type ProduceServer struct {
	produce.UnimplementedProduceServiceServer
}

// SendStream 重写SendStream方法,服务提供者
func (*ProduceServer) SendStream(stream produce.ProduceService_SendStreamServer) error {
	// 循环读取客户端的数据流
	for {
		// 阻塞式等待客户端数据流
		request, err := stream.Recv()
		if err != nil && err == io.EOF {
			// 接收到 err = EOF 表示数据发送完成。响应客户端,跳出循环直接结束.并且关闭连接
			stream.SendAndClose(&produce.ProduceReply{Completed: true, Msg: "消费完毕"})
			break
		}
		// 读取到数据流
fmt.Printf("Server接收到数据,消费第 %d 条数据, 消费时间: %v\n", request.ProduceId, time.Now().Format("15:04:05"))
	}
	return nil
}

开启服务端

func main() {
	// 1.创建一个tcp服务
	listen, err := net.Listen("tcp", ":7777")
	if err != nil {
		fmt.Println(err)
		return
	}
	grpcServer := grpc.NewServer()
	produceServer := &ProduceServer{}
	produce.RegisterProduceServiceServer(grpcServer, produceServer)
	fmt.Printf("gRPC Server Start listen: %s \n", listen.Addr())

		fmt.Printf("GRPC服务启动失败 原因: %s \n", err.Error())

客户端调用

func main() {
	conn, err := grpc.Dial("localhost:7777", grpc.WithTransportCredentials(insecure.NewCredentials()))
	if err != nil {
		fmt.Printf("gRPC Client Conn Server err: %s \n", err.Error())
		return
	}
	defer conn.Close()
	// 建立 grpcClient
	grpcClient := produce.NewProduceServiceClient(conn)
	// 发起请求,获取数据流客户端
	streamClient, err := grpcClient.SendStream(context.Background())
	// 流式发送十条消息到服务端
	i := 0
	for {
		// 1秒发送1条
		time.Sleep(time.Second)
		// 发送数据
		err = streamClient.Send(&produce.ProduceRequest{ProduceId: int32(i)})
		if err != nil {
			fmt.Println("发送数据失败...")
			continue
		}
		i++

		fmt.Printf("已发送完第 %d 条数据,发送时间: %v \n", int32(i), time.Now().Format("15:04:05"))
		if i >= 10 {
			// 当发送完数据后,响应给服务端 数据已经发送完毕,然后等服务端的结束响应
			fmt.Println("数据发送完毕,等待客户端响应...")
			reply, _ := streamClient.CloseAndRecv()
			fmt.Printf("客户端响应: 【ok: %v,msg: %s】 连接结束!", reply.GetCompleted(), reply.GetMsg())
			break
		}
	}
}

2. 返回参数为流

其实通过上面一个案例,就已经差不多能够熟知全部的知识了,所以就不写这么麻烦了,就简单从服务端读取一下就好.

import "google/protobuf/empty.proto";


message BodyReply{
  string body = 1;
}

service ProduceService{
	// 定义一个返回值为数据流的接口
  rpc GetStream(google.protobuf.Empty)returns(stream BodyReply);
}

编译.

服务端实现


// GetStream 重写GetStream方法,提供服务
func (*ProduceServer) GetStream(e *emptypb.Empty, stream produce.ProduceService_GetStreamServer) error {
	i := 0
	for {
		i++
		body := fmt.Sprintf("这是来自于服务端的第 %d 条消息 产生时间: %s", i, time.Now().Format("15:04:05"))
		stream.Send(&produce.BodyReply{Body: body})
		time.Sleep(time.Second)
		if i >= 10 {
			// 发送数据完毕,注意：这里不会需要通知,直接返回nil即可
			break
		}
	}
	return nil
}

然后main函数不用改,直接启动即可.

客户端调用

func main() {
	conn, err := grpc.Dial("localhost:7777", grpc.WithTransportCredentials(insecure.NewCredentials()))
	if err != nil {
		fmt.Printf("gRPC Client Conn Server err: %s \n", err.Error())
		return
	}
	defer conn.Close()
	// 建立 grpcClient
	grpcClient := produce.NewProduceServiceClient(conn)
	streamClient, err := grpcClient.GetStream(context.Background(), &emptypb.Empty{})
	if err != nil {
		panic(err)
	}
	for {
		reply, err := streamClient.Recv()
		if err != nil && err == io.EOF {
			// 服务的数据发送完毕,已关闭连接,直接退出即可
			break
		}
		fmt.Printf("接收到Server的数据, 内容: %s \n", reply.GetBody())
	}
}
