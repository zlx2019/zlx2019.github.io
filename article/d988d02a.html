前言

Rust错误处理基础篇章

错误处理是程序语言必不可少的特性,因为程序的运行总是无法完全保证的。因此一门优秀的编程语言必须有其完整的错误处理哲学。
在一些语言中也被称之为异常(Exception)或者错误(Error)。在很多情况下，Rust 需要你承认自己的代码可能会出错，并提前采取行动，来处理这些错误。

Rust中的错误主要分为两种:

  • 可恢复错误

    该错误通常指一些允许接收的错误,例如用户访问、权限、操作等错误,这些错误只会影响到某个用户自身操作,而不会影响到全局程序的运行。

  • 不可恢复错误

    该错误通常指一些系统级别的致命错误,会直接导致程序崩溃,从而产生Panic。

Rust通常用Rust<T,E>来处理可恢复错误。而panic!用于不可恢复错误。

Golang错误处理

Go语言在错误处理机制上和Rust非常类似,所以我们先简单了解一下Go的错误处理,看看它有哪些特点？又存在哪些不足?

Go错误处理特点:

  • 默认所有的错误都是不可恢复错误，一旦产生Panic,程序会直接终止。

  • 不支持try catch来捕获异常,使用defer recover来捕获错误。

  • 错误处理粒度非常精细,每个可能存在错误的函数都需要返回一个error然后对此进行处理,使得代码非常健壮，但是不够美观。

Go与Rust错误处理相同点:

它们都使用主动返回错误的方式,来处理错误,都是使用选择来处理错误。简单来说就是手动返回一个Error,再手动接收它并且处理。不同的是Go通常是直接返回一个error,而Rust是返回一个Result枚举。

Error

Go中的Error是一个接口:

type error interface {
	Error() string
}

通常我们想返回一个错误时,都是使用这个error接口作为返回参数:

func DoRun() (int,error){
  return nil
}

只要一个类型,实现了Error这个方法,就表示实现了error这个接口,就可以作为具体的返回值。

自定义Error

在Go中自定义一个Error非常简单,只需要重写Error()string这个接口方法即可:

// 自定义业务Error
type BusinessError struct {
	Message string
}

// 实现error接口的Error方法,返回错误信息
func (be BusinessError) Error() string {
	return be.Message
}

然后我们就可以使用这个BusinessError错误作为返回值:

func DoRun() (int,error) {
	return 1,BusinessError{Message: "自定义错误类型: 模拟异常"}
}

如果是程序正常运行结束的话,只需要返回一个nil表示没有错误即可。

func DoRun() (int,error) {
	return 1001,nil
}

然后我们就可以通过该逻辑来处理这个函数的返回值:

code,err := DoRun()
if err != nil {
  // 发生了错误
}else{
  // 正常执行
}

以上就是Go的最基本的错误处理。下面我们来总结一下:

优点:

  • 简单粗暴,可读性非常高

  • 健壮性非常好

  • 错误处理的粒度非常细

不足之处:

代码量非常多、不够优雅、太过冗余。

存在一些多余的操作,如下:

return 1001,nil

这里本质上是返回一个成功的返回值即可,却还要指定一个nil来表示没有error,明明是两件毫不相关的事情,简单来说这两个值应该是完全对立的,但是还必须都要返回。

return 0,BusinessError{Message: "自定义错误类型: 模拟异常"}

这里也是一样,明明只需要返回一个error即可,却还要一个类型的零值作为占位符返回,其实完全是没有必要的。

Rust错误处理

上面我们简单了解了一下Go的错误处理,并且知道了它于传统语言的不同之处以及它的优点和不足之处,下面我们开始学习Rust的错误处理。

Rust和Go一样,也是采用了返回错误的方式来处理错误,但是与Go不同的是Rust使用枚举来处理错误。
Rust基本上拥有Go错误处理的所有特点,并且在它的基础之上更加优雅简洁。

Result

Rust使用Result<T,E>枚举来处理错误:

pub enum Result<T, E> {
    /// Contains the success value
    #[lang = "Ok"]
    #[stable(feature = "rust1", since = "1.0.0")]
    Ok(#[stable(feature = "rust1", since = "1.0.0")] T),

    /// Contains the error value
    #[lang = "Err"]
    #[stable(feature = "rust1", since = "1.0.0")]
    Err(#[stable(feature = "rust1", since = "1.0.0")] E),
}

以上就是Result枚举的源码:

  • T表示没有错误的情况下,正常返回的参数类型。通过Ok(T)函数来返回。
  • E表示发送错误的情况下,返回的错误类型。通过Err(E)函数来返回。

如下定义:

fn do_run() -> Result<String,std::io::Error>{
    todo!()
}

该方法返回一个Result<String,std::io:Error>。

表示如果没有错误情况下,需要返回一个值类型为String的Result枚举，如下:

fn do_run()-> Result<String,std::io::Error>{
    Ok(String::from("Hello"))
}

如果发生错误的情况下,需要返回一个错误类型为std::io::Error的Result枚举,如下:

fn do_run()-> Result<String,std::io::Error>{
    Err(std::io::Error::new(std::io::ErrorKind::Other,"其他错误~"))
}

Rust枚举的特性

Rust为了安全,使用match匹配枚举时,是默认强制处理枚举的所有可能项。

pub enum Result<T, E> {
    Ok(T),
    Err(E),
}

fn main() {
    match do_run() {
	      // 处理Ok成功结果
        Ok(val)=> println!("操作成功,Result结果值为:{}",val),
	      // 处理Err错误结果
        Err(error) => println!("操作错误: {:?}",error)
    }
}

Rust很好的利用了这一项特性,所以当一个方法的返回值为Result时,只能返回Ok(T)和Err(E)其中的一个。因为Result只有Ok和Err两个可能项。本质上只是返回了一个枚举项,要么是Ok(T)要么是Err(E),所以只返回了一个值,这是它与Go不同之处,也是上面我们说到的Go错误处理的不足之处之一。

Go与Rust返回错误示例

  1. 无返回值示例

    • Go

    func doRun(value string)error

    • Rust

    fn do_run(value: String) -> Result<(),Box<dyn std::error:Error>>

  2. 有返回值示例

    • Go

    func doRun(value string) (int,error)

    • Rust

    fn do_run(value: String) -> Result<usize,Box<dyn std::error:Error>>

  3. 多返回值示例

    • Go

    func doRun(value string) int,string,error

    • Rust

    fn do_run(value: String) -> Result<(usize,String),Box<dyn std::error:Error>>

解释:

Box<dyn std::error:Error> 是一种通用的方式来表示任何类型的错误。可接受所有类型的错误

Result<(),xxx>表示一个空的元组类型,也就是没有返回值Result。

Result<(i8,String),xxx>表示一个元组类型的Result,该元组有两个参数分别为i8和String类型。

Result相关方法

Result<T,E>枚举是Rust中最最最常用的枚举之一,想要基础扎实,熟练使用它的方法是必不可少的。下面我们一一来介绍Result的方法。

unwrap()

fn unwrap(self) -> T

该方法用于获取Result<T,E>中T参数的具体值。

let result: Result<usize,&str> = Ok(1001);
assert_eq!(result.unwrap(),1001)

如果Result是Err的话,则会产生panic。通常并不推荐使用该方法。

let result: Result<usize,&str> = Err("这是错误的");
assert_eq!(result.unwrap(),1001)// panic

输出结果:

//thread 'main' panicked at 'called `Result::unwrap()` on an `Err` value: "这是错误的"', src/main.rs:36:23

unwrap_or()

fn unwrap_or(self, default: T) -> T

该方法相当于执行unwrap()方法的同时,为T指定一个默认值,一旦Result是Err,则返回默认值。

相当于做了一个兜底的操作,防止直接发生panic!。

let result: Result<usize,&str> = Err("这是错误的");
assert_eq!(result.unwrap_or(1001),1001);

unwrap_or_else()

fn unwrap_or_else T>(self, op: F) -> T {

该方法与unwrap_or()方法类似,同样是为T设置一个默认值,不同的是这里并不是直接设置具体的值,而是传入一个闭包,具体返回的默认值是什么由闭包的返回值而决定。

同时产生的Error也是闭包的参数,可以通过闭包来访问错误信息。

let result: Result<usize,&str> = Err("这是错误的");

// 传入闭包,设置默认值
assert_eq!(result.unwrap_or_else(|error|{
  println!("输出错误信息: {}",error);
  // 返回一个默认值作为兜底值
  // 返回错误消息的字节长度,作为默认值
  error.len()
}),15)

unwrap_unchecked()

fn unwrap_unchecked(self) -> T

该方法用于获取Result<T,E>中的T的值,如果是Err的话则不做任何处理。

如果使用不当，可能会导致程序的不可预测行为，因此必须非常小心地使用它。

let id = unsafe { Ok::<usize,&str>(1).unwrap_unchecked() };
println!("{}",id); // 1

let a = unsafe { Err::<usize,&str>("错误的").unwrap_unchecked() };
println!("{}",id); //没有任何处理

expect()

fn expect(self, msg: &str) -> T

该方法也是用于提取T的值,如果是Err则会直接panic,并且输出错误消息或者传递的错误消息。

let result: Result<usize,&str> = Err("这是错误的");
result.expect("错误消息"); // thread 'main' panicked at '错误消息: "这是错误的"', src/main.rs:36:12

expect_err()

fn expect_err(self, msg: &str) -> E

该方法用于提取Result中E的错误值,如果是Ok则会panic并且输出错误信息.

let result: Result<usize,&str> = Ok(1001);
result.expect_err("获取Err失败,Result为Ok,具体的值为:"); //thread 'main' panicked at '获取Err失败,Result为Ok,具体的值为:: 1001',

如果是Err则会直接获取Error。

let result: Result<usize,&str> = Err("这是一个模拟错误的消息");
// 获取Error
let err_msg = result.expect_err("获取Err失败:");
assert_eq!(err_msg,"这是一个模拟错误的消息");

is_ok()

fn is_ok(&self) -> bool

该方法用于判断Result是否为Ok,如果是则返回true,反之如果是Err则返回false。

let result: Result<usize,&str> = Ok(1);
assert_eq!(result.is_ok(),true);

let result2: Result<(), &str>  = Err("错误信息");
assert_eq!(result2.is_ok(),false);

is_err()

fn is_err(&self) -> bool

该方法与is_ok()完全相反。is_err() -> bool方法用于判断Result是否为Err,如果是则返回true,如果是Ok则返回false。

let result:Result<(), &str> = Err("这是一个错误响应");
assert_eq!(result.is_err(),true);

let result1:Result<usize,&str> = Ok(1);
assert_eq!(result1.is_err(),false);

ok()

fn ok(self) -> Option< T>

该方法用于将Result转换为Option。

如果Result是Ok(T)则转换为Some(T),如果是Err(E),则转换为None。

// Result-Ok To Opotion-Some
let result: Result<usize,&str> = Ok(1001);
assert_eq!(result.ok(),Some(1001));

// Result-Err To Option-None
let result1: Result<usize,&str> = Err("这是错误的");
assert_eq!(result1.ok(),None);

err()

fn err(self) -> Option< E>

该方法用于获取Result转换为Option。

如果Result是Ok(T)则转换为Option中的None。如果是Err(E)则转换为Some(E)。

// Result-Ok To Option-None
let result: Result<usize,&str> = Ok(1001);
assert_eq!(result.err(),None);

// Result-Err To Option-Some
let result1: Result<usize,&str> = Err("这是错误的");
assert_eq!(result1.err(),Some("这是错误的"));

as_ref()

fn as_ref(&self) -> Result<&T, &E>

该方法用于获取Result的引用。

如果是Ok<T>则获取T的引用:Ok<&T>。

如果是Err<E>则获取E的引用: Err<&E>。

  • 获取Ok(T)的引用:
let result: Result<usize,&str> = Ok(1001);
assert_eq!(result.as_ref(),Ok(&1001));
  • 获取Err(E)的引用:
let result1: Result<usize,&str> = Err("这是错误的");
assert_eq!(result1.as_ref(),Err(&"这是错误的"));

as_mut()

fn as_mut(&mut self) -> Result<&mut T, &mut E>

该方法用于获取一个可变的Result的可变引用。

如果是Ok<T>则获取T的可变引用:Ok<&mut T>。

如果是Err<E>则获取E的可变引用: Err<&mut E>。

let mut result: Result<usize,&str> = Ok(1001);
assert_eq!(result.as_mut(),Ok(&mut 1001));

let mut result1: Result<usize,&str> = Err("这是错误的");
assert_eq!(result1.as_mut(),Err(&mut "这是错误的"));

错误处理案例

案例需求: 打开一个文件,读取文件内容。

传统编码方式

use std::{fs::File, io::Read};

fn main() -> Result<(),Box<dyn std::error::Error>>{
    // 打开文件
    let file_result: Result<File,std::io::Error> = File::open("Cargo.toml");
    if file_result.is_ok() {
        println!("打开文件成功");
        // 读取文件内容
        let mut file:File = file_result.unwrap();
        // 创建一个字符串作为缓冲区
        let mut buf:String = String::new();
        // 读取文件内容
        let read_result:Result<usize, std::io::Error> = file.read_to_string(&mut buf);

        if  read_result.is_ok(){
            println!("读取成功,文件内容: \n{}",buf);
        }else {
            println!("读取文件失败:{}",read_result.unwrap_err());
        }
    }else {
        println!("打开文件失败:{}",file_result.unwrap_err())
    }
    Ok(())
}

输出结果:

打开文件成功
读取成功,文件内容: 
[package]
name = "err_process"
version = "0.1.0"
edition = "2021"

这种方式和传统编码风格完全一致,主要通过if...else+Result.is_ok()来处理分支结构。

这样编写的好处:

  • 简单粗暴
  • 结构清晰明了
  • 可读性高

不过通常我们在Rust中不会这样编写,下面我们使用match来改善一下这个案例。

使用Match简化

通常对于Result或者Option等常用枚举,使用match表达式来处理也是个不错的手段.

use std::{fs::File, io::Read};

fn main() -> Result<(),Box<dyn std::error::Error>>{
    // 打开文件
    let file_result: Result<File,std::io::Error> = File::open("Cargo.toml");
    // 使用match 来处理Result
    match file_result {
        // Ok处理,这里使用mut file 是因为file.read_to_string()方法需要一个可变的file
        Ok(mut file ) => {
            let mut buf:String = String::new();
            //使用match 再次处理读取文件的Result
            match file.read_to_string(&mut buf) {
              	//_ 表示省略掉这个T
                Ok(_)=> println!("读取文件成功,文件内容: \n{}",buf),
                Err(error) => println!("读取文件内容失败:{}",error)
            }
        }
        //Err处理
        Err(error) => println!("打开文件失败:{}",error)
    }
    Ok(())
}

Rust中最常用的就是使用match来处理Result和Option枚举。

从上面的案例上就可以看出来,似乎这种形式更加难以阅读了,还不如传统式编码,所以Rust又提供了一种简化操作符？来简化match表达式。

下面我们就使用?来再次简化我们这个案例。

使用?简化Match

一个Result值后面追加一个?表示使用match来处理这个Result

如果为Ok(T)则直接获取T值。反之如果是Err(E)直接将E作为方法返回值。

如下:

let mut file:File = File::open("Cargo.toml")?;

等同于如下代码:

// 打开文件,获取一个Result
let file_result: Result<File, std::io::Error> = File::open("Cargo.toml");
// 使用match处理Result
let mut file:File = match file_result {
  // 如果是Ok，直接返回T,也就是File
  Ok(f) => f,
  // 如果是Err,直接结束方法,将error作为Err返回
  Err(error) => return  Err(error)
};

接下来我们开始简化这个案例~

简化一

使用?来处理Result结果值

fn main() -> Result<(),Box<dyn std::error::Error>>{
    let buf:String = file_read()?;
    println!("{}",buf);
    Ok(())
}

/// 读取一个文件的内容
fn file_read() -> Result<String,std::io::Error>{
    // 读取文件 ? 表示如果Result为Err 则直接返回Err
    let mut file:File = File::open("Cargo.toml")?;
    // 创建缓冲区
    let mut buf:String = String::new();
    // 读取文件内容
    _ = file.read_to_string(&mut buf)?;
    Ok(buf)
}

简化二

使用?并且使用流式编码来简化案例

fn file_read() -> Result<String,std::io::Error>{
    // 创建缓冲区
    let mut buf:String = String::new();
    // 打开文件并且读取文件
    File::open("Cargo.toml")?.read_to_string(&mut buf)?;
    // 返回
    Ok(buf)
}

fn main() -> Result<(),Box<dyn std::error::Error>>{
    let buf:String = file_read()?;
    println!("{}",buf);
    Ok(())
}

简化到这里基本上就是我们常用的Rust编码风格了~

自定义Error

Rust中自定义一个Error就没有Go中这么方便了,相对来说还是较为复杂的。在Rust中,通常都会选用枚举(enum)来自定义Error。

定义错误枚举

/// 自定义业务Error
#[derive(Debug)]
enum BussinessError{
    // 基础错误
    BasicError(String),
    // 状态错误
    StatueError(usize),
    // IO错误
    IoError,
    // 嵌套错误,网络相关的所有错误
    Network(NetworkError),
}

这里简单定义了一个BussinessError枚举,内部持有三个Error和一个嵌套错误Network。下面我们定义NetworkError:

/// 网络相关错误
#[derive(Debug)]
enum NetworkError {
  // 超时错误
  TimeOutError,
  // 未找到资源错误
  NotFoundError,
}

impl Display

接下来为自定义的BussinessError和NetworkError实现fmt::Display这个特征,该特征负责其类型作为{}占位符时,映射出的字符串值。

实现了这个特征后,它允许你将本类型格式化输出为字符串。具体来说，实现了 fmt::Display 特征的类型可以使用 {} 占位符将其输出为字符串，或者在使用 println! 和 format! 等宏时作为参数传递。

/// 为BussinessError实现Display trait
/// 实现该特征后,该类型就可以使用`{}`占位符输出成字符串
impl fmt::Display for BussinessError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        // 处理Bussioness的所有可能项,分别在`{}`时 映射返回的字符串内容
        match self {
            BussinessError::BasicError(msg) => write!(f,"BussinessError Basic Error: {}",msg),
            BussinessError::StatueError(code) => write!(f,"BussinessError Status Error Code: {}",code),
            BussinessError::IoError => write!(f,"BussinessError IO Error"),
            // 由于NetworkError也实现了Display特征,所以可以直接使用`{}`占位符输出
            BussinessError::Network(network_error) => write!(f,"{}",network_error),
        }
    }
}

/// 为NetworkError实现Display trait
impl fmt::Display for NetworkError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            NetworkError::TimeOutError => write!(f,"BussinessError_NetworkError TimeOutError"),
            NetworkError::NotFoundError => write!(f,"NotFound_NetworkError NotFoundError"),
        }
    }
}

impl Error

一个类型只有实现了std::error:Errortrait才能表示为一个真正的错误类型。

/// 为BussinessError实现Error trait
/// 只有实现了Error trait的类型才算是真正的错误类型
/// 由于BussinessError 下存在Network子Error,所以需要重写source方法
impl Error for BussinessError  {
    fn source(&self) -> Option<&(dyn Error + 'static)> {
        match self {
            BussinessError::BasicError(_) => None,
            BussinessError::StatueError(_) => None,
            BussinessError::IoError => None,
            // 只有存在子错误的类型才需要这样返回
            BussinessError::Network(e) => Some(e),
        }
    }
}
/// 为NetworkError实现Error trait
/// 只有实现了Error trait的类型才算是真正的错误类型
/// 由于NetworkError 下不存在任何子Error,所以不需要重写source方法
impl Error for NetworkError {}

测试

到这里错误就已经定义完成了,接下来我们来简单使用并且测试一下.


// 测试返回一个BussinessError::BasicError
fn test_baisc_error()-> Result<(),BussinessError>{
    Err(BussinessError::BasicError(String::from("这是自定义的基本错误类型")))
}

// 测试返回一个BussinessError::StatueError
fn test_status_error() -> Result<(),BussinessError>{
    Err(BussinessError::StatueError(1001))
}

// 测试返回一个BussinessError::IOError
fn test_io_error()-> Result<(),BussinessError>{
    Err(BussinessError::IoError)
}

// 测试返回一个BussinessError::Network(NetworkError::TimeOutError)
fn test_timeout_error() -> Result<(),BussinessError>{
    Err(BussinessError::Network(NetworkError::TimeOutError))
}
