1. NSQ是什么?

NSQ是一个由Go语言编写的消息中间件。它是一个实时分布式消息传递平台，旨在大规模运行，每天处理数十亿条消息。它促进无单点故障的分布式和去中心化拓扑，实现容错和高可用性以及可靠的消息传递保证。
NSQ易于配置和部署（所有参数都在命令行上指定，编译后的二进制文件没有运行时依赖性。为了获得最大的灵活性，它与数据格式无关（消息可以是 JSON、MsgPack、Protocol Buffers 或任何其他格式）

官方文档: nsq.io

2. 特性

  • 分布式: 去中心化,无单点故障。
  • 可扩展性: 支持水平扩展。集群中的节点的加入和退出都可以动态的处理。
  • 高性能: 基于低延迟推送的消息传递。在流式（高吞吐量）和面向工作（低吞吐量）的工作负载方面表现出色。
  • 简易性: 部署与配置很简单,内置UI控制界面。
  • 通用性: 与主流编程语言都有相应的集成。

3. 架构模型

NSQ是支持单机与集群环境的,它的架构模型含有以下几种组件组成。

nsqd

nsqd就是一个节点实例,负责消息的发布与订阅处理,它可以独立运行。

如果想部署单机环境,那么只需要启动一个nsqd即可。

如果是部署集群环境,那么需要注册到nsqlookupd中。

一个nsqd会占用三个端口:

  • 4150(TCP)
  • 4151(HTTP)
  • 4152(HTTPS,可选)

默认提供了通过HTTP和TCP协议进行通信,使开发者更加灵活方便的使用。如果配置了TSL功能,那么将会使用4152端口提供服务。

nsqlookupd

该组件主要用于集群环境,用于管理nsqd的所有集群节点,提供服务发现和协调功能。功能如下:

  • 注册nsqd

    当一个 nsqd 节点启动时，它会向 nsqlookupd 注册自己的信息，包括自己的地址、监听的 Topic 和 Channel 等信息。这样其他的 nsqd 和 nsqadmin 等组件就可以通过 nsqlookupd 找到它们需要的信息，从而进行消息的发布和订阅等操作。

  • 发现nsqd

    当一个客户端需要连接到一个 Topic 的某个 Channel 时，它可以向 nsqlookupd 发送一个查询请求，nsqlookupd 将会返回该 Topic 所有可用的 nsqd 节点地址，客户端可以选择其中一个节点建立连接，并向该节点发送消息。

  • 主题管理

    nsqlookupd 还提供了一些管理 Topic 和 Channel 的 API，例如创建、删除、查询等。通过这些 API，管理员可以方便地对整个 nsq 集群进行管理。

该组件也支持HTTP和TCP两种协议通信,默认端口分别为: 4160(TCP)、4161(HTTP)。

总之，nsqlookupd 是 nsq 集群中非常重要的一个组件，它通过提供服务发现和协调功能，实现了高可用性、可扩展性和灵活性等优良特性。

nsqadmin

内置提供的一个可视化管理平台。

utils

nsq同时也提供了一些已经写好的工具,如下:

  • nsq_stat

    拉取一个topic/channel的所有消费者消费信息并显示汇总统计信息。

  • nsq_tail

    消费一个topic/channel的消息,并且将消息内容输出到控制台。

  • nsq_to_file

    消费一个topic/channel的消息,并且将消息内容写入到文件中。

  • nsq_to_http

    消费一个topic/channel的消息,将消息发送到指定的HTTP端点。

  • nsq_to_nsq

    消费一个topic/channel的消息,将消息通过TCP转发到另一个nsqd中。

  • to_nsq

    通过标准输入流,将数据发送到指定的nsqd中。

4. 核心概念

Topic

topic消息主题,一个nsqd服务实例可以有多个topic,用于将不同的类型的消息进行分组。

与RocketMQ中的topic有着相似度含义。

Channel

channel是topic主题中的一个订阅频道。每个主题都可以有多个频道(channel)，这些频道表示不同的消息订阅者。

当一个客户端(消费者)与topic建立连接后,就会产生一个新的频道。而topic就是通过该频道将消息传递给消费者。

单机部署

下载地址

根据自己不同的操作系统下载不同的版本即可。

下载完后直接解压即可:

tar -zxvf nsq-1.2.1.darwin-arm64.go1.16.6

解压后进入文件内,发现只有一个bin文件夹,里面是一些已经编译好的二进制可执行文件

6. 消费模式

nsq的消息消费模式完全取决于消费者如何消费。

广播模式

多个消费者同时消费同一个topic但不同channel时,那么将会形成一种广播模式。也就是该消息会被所有的消费者所消费一次。

需求: 启动多个消费者,分别监听users主题下的v1、v2的channel:

./nsq_tail --nsqd-tcp-address 127.0.0.1:4150 --topic=users --channel=v1
2023/03/26 15:44:19 Adding consumer for topic: users
2023/03/26 15:44:19 INF    1 [users/v1] (127.0.0.1:4150) connecting to nsqd
./nsq_tail --nsqd-tcp-address 127.0.0.1:4150 --topic=users --channel=v2
2023/03/26 15:44:59 Adding consumer for topic: users
2023/03/26 15:44:59 INF    1 [users/v2] (127.0.0.1:4150) connecting to nsqd

然后启动生产者向users主题中发布消息:

curl -d "这是一条users主题的消息 001" 'http://127.0.0.1:4151/pub?topic=users'
OK                                                                                                                                           curl -d "这是一条users主题的消息 002" 'http://127.0.0.1:4151/pub?topic=users'
OK

然后我们观察消费者控制台:

2023/03/26 15:44:19 INF    1 [users/v1] (127.0.0.1:4150) connecting to nsqd
这是一条users主题的消息 001
这是一条users主题的消息 002
2023/03/26 15:44:59 INF    1 [users/v2] (127.0.0.1:4150) connecting to nsqd
这是一条users主题的消息 001
这是一条users主题的消息 002

显然,两个消费者都消费到了这条消息,这就是nsq的消息广播模式。

轮询模式

刚刚我们介绍了消息的广播模式,但是在一些常见的场景下我们并不希望一条消息被重复消费,如有一个user服务,这个服务负责处理一些user相关的消息,但是这个服务有好多个节点所部署,这种时候我们只希望一条消息只被一个服务节点消费即可。nsq也是支持此模式的。

多个消费者同时消费同一个topic并且同一个channel时,nsqd会将消息以负载均衡的方式分发给消费者组的其中一个实例。保证一条消息只会被消费一次。

需求: 启动两个消费者，共同监听users主题的v3channel。

./nsq_tail --nsqd-tcp-address 127.0.0.1:4150 --topic=users --channel=v3
2023/03/26 15:57:46 Adding consumer for topic: users
2023/03/26 15:57:46 INF    1 [users/v3] (127.0.0.1:4150) connecting to nsqd
./nsq_tail --nsqd-tcp-address 127.0.0.1:4150 --topic=users --channel=v3
2023/03/26 15:57:40 Adding consumer for topic: users
2023/03/26 15:57:40 INF    1 [users/v3] (127.0.0.1:4150) connecting to nsqd

然后启动生产者向users主题中发布消息:

curl -d "这是一条users主题的消息 004" 'http://127.0.0.1:4151/pub?topic=users'
OK                                                                                                                                           curl -d "这是一条users主题的消息 005" 'http://127.0.0.1:4151/pub?topic=users'
OK                                                                                                                                           curl -d "这是一条users主题的消息 006" 'http://127.0.0.1:4151/pub?topic=users'
OK                                                                                                                                           curl -d "这是一条users主题的消息 007" 'http://127.0.0.1:4151/pub?topic=users'
OK

然后我们观察消费者控制台:

2023/03/26 15:57:46 INF    1 [users/v3] (127.0.0.1:4150) connecting to nsqd
这是一条users主题的消息 005
这是一条users主题的消息 007
2023/03/26 15:57:40 INF    1 [users/v3] (127.0.0.1:4150) connecting to nsqd
这是一条users主题的消息 004
这是一条users主题的消息 006

可以看到4条消息分别被两个消费者轮询消费了。

  "log_level": "info",

  "max_output_buffer_timeout": "30s",
  "max_output_buffer_timeout_on_idle": false,
  "max_tries": 5,
  "default_requeue_delay": "90s",
  "max_requeue_delay": "15m",
  "max_backoff_duration": "5m",
  "max_req_time": "60s",
  "auth_enabled": true,
  "msg_timeout": "60s",
  "output_buffer_size": 16384,
  "output_buffer_timeout": "250ms",
  "max_in_flight": 100,
  "max_attempts": 10,
  "max_backoff_time": "1m",
  "max_wait_time": "5m",
  "deflate": true,
  "deflate_level": 6,
  "snappy": false,
  "sample_rate": 0,
  "client_id": "",
  "hostname": "",
  "user_agent": "",
  "tls_v1": false,
  "tls_v1.1": false,
  "tls_v1.2": true,
  "tls_config": {}
}

下面是各个选项的描述：

  • log_level：指定日志记录的级别。可选值包括 debug、info、warning、error 和 fatal 等，建议设置为 info。
  • log_rotate_interval：日志切割时间间隔。可选值为 24h、1h 等时间格式，建议设置为 24h。
  • log_rotate_size：日志切割的大小。设置为 0 时，禁止切割，建议不开启切割功能。
  • log_rotate_compress：是否压缩切割的日志。建议不开启压缩。
  • tls_cert：TLS 证书的路径。
  • tls_key：TLS 证书的私钥的路径。

  • tls_client_auth_policy：TLS 客户端认证策略，可选值为 require

  • tls_min_version：TLS 最低版本号。建议设置为 tls1.2。

  • tls_curve_preferences：TLS 曲线偏好列表。建议设置为 p384。
  • auth_http_endpoint：认证 HTTP 端点的地址，用于验证访问者的身份。
  • auth_secret：用于认证的秘密值。
  • lookupd_http_addresses：NSQlookupd HTTP 地址列表，用于向 NSQlookupd 发现 NSQD 节点。
  • nsqd_tcp_address：NSQD 监听的 TCP 地址。
  • http_address：NSQD HTTP API 监听的地址。
  • broadcast_address：NSQD 在集群中公开的地址。
  • max_deflate_level：压缩级别，建议设置为 6。
  • max_msg_timeout：消息最大的超时时间，建议设置为 15m。
  • max_req_timeout：最大请求超时时间，建议设置为 10s。
  • max_confirm_win：最大的确认窗口大小，建议设置为 1000。
  • max_conn_rdy_count：最大的连接准备好的数量，建议设置为 2500。
  • max_conn_rdy_idle_timeout：连接准备好的最大空闲时间，建议设置为 5m。
  • max_output_buffer_size：最大的输出缓冲区大小，建议设置为 64。
  • max_output_buffer_timeout：最大的输出缓冲区超时时间，建议设置为 30s。
  • max_output_buffer_timeout_on_idle：是否在空闲时启用最大的输出缓冲区超时时间，建议不开启。
  • max_tries：消息重新投递的最大次数，建议设置为 5。
  • default_requeue_delay：默认的重新投递延迟时间，建议设置为 90s。
  • max_requeue_delay：重新投递的最大延迟时间，建议设置为 15m。
  • max_backoff_duration：最大的指数退避时间，建议设置为 5m。
  • max_req_time：最大请求时间，建议设置为 60s。
  • auth_enabled：是否启用认证，建议启用。
  • msg_timeout：消息的超时时间，建议设置为 60s。
  • output_buffer_size：输出缓冲区大小，建议设置为 16384。
  • output_buffer_timeout：输出缓冲区超时时间，建议设置为 250ms。
  • max_in_flight：最大的并发消息数量，建议设置为 100。
  • max_attempts：最大的尝试次数，建议设置为 10。
  • max_backoff_time：最大的退避时间，建议设置为 1m。
  • max_wait_time：最大的等待时间。

在nsqd启动时可以通过-config来指定配置文件:

nsqd -config=/path/to/nsqd.conf

也可以使用配置文件的同时,使用命令行参数来覆盖配置文件中的参数

nsqd -config-file=/path/to/nsqd.conf -tcp-address=0.0.0.0:4150

PATH: /topic/empty

请求类型: POST

示例

curl -X POST http://127.0.0.1:4151/topic/empty?topic=name

清除channel消息

清空现有通道的所有排队消息（内存和磁盘）

PATH: /channel/empty

请求类型: POST

示例

curl -X POST http://127.0.0.1:4151/channel/empty?topic=name&channel=name

暂停topic

暂停消息流向指定的主题的所有频道（消息将在主题排队）,恢复之后消息依然还会被发送到主题中。

PATH: topic/pause

请求类型: POST

示例

curl -X POST 'http://127.0.0.1:4151/topic/pause?topic=name'

恢复topic

将消息流恢复到现有的、暂停的主题的频道

PATH: topic/unpause

请求类型: POST

示例

curl -X POST 'http://127.0.0.1:4151/topic/unpause?topic=name'

暂停channel

暂停消息流向现有通道的消费者（消息将在通道排队）

PATH: channel/pause

请求类型: POST

示例

curl -X POST http://127.0.0.1:4151/channel/pause?topic=name&channel=name

恢复channel

将消息流恢复到现有的、暂停的频道的消费者。

PATH: channel/unpause

请求类型: POST

示例

curl -X POST http://127.0.0.1:4151/channel/unpause?topic=name&channel=name

9. Go项目集成

我们平常更多的还是通过程序来使用nsq,Go语言与nsq有着天然的集成优势,可谓是嫡系,所以我们以Go语言为例,来集成使用一下nsq。

1.下载依赖

go get github.com/nsqio/go-nsq

2.Producer

创建生产者并且发布消息:

// Instantiate a producer.
config := nsq.NewConfig()
// 创建客户端
producer, err := nsq.NewProducer("127.0.0.1:4150", config)
if err != nil {
	log.Fatal(err)
}

// 消息和主题
messageBody := []byte("hello")
topicName := "topic"

// Synchronously publish a single message to the specified topic.
// Messages can also be sent asynchronously and/or in batches.
// 发布消息
err = producer.Publish(topicName, messageBody)
if err != nil {
	log.Fatal(err)
}

// Gracefully stop the producer when appropriate (e.g. before shutting down the service)
producer.Stop()

3.Consumer

package main
import (
	"log"
	"os/signal"
	"github.com/nsqio/go-nsq"
)

type myMessageHandler struct {}

// HandleMessage implements the Handler interface.
func (h *myMessageHandler) HandleMessage(m *nsq.Message) error {
	if len(m.Body) == 0 {
		// Returning nil will automatically send a FIN command to NSQ to mark the message as processed.
		// In this case, a message with an empty body is simply ignored/discarded.
		return nil
	}

	// do whatever actual message processing is desired
	err := processMessage(m.Body)

	// Returning a non-nil error will automatically send a REQ command to NSQ to re-queue the message.
	return err
}

func main() {
	// Instantiate a consumer that will subscribe to the provided channel.
	config := nsq.NewConfig()
	consumer, err := nsq.NewConsumer("topic", "channel", config)
	if err != nil {
		log.Fatal(err)
	}

	// Set the Handler for messages received by this Consumer. Can be called multiple times.
	// See also AddConcurrentHandlers.
	consumer.AddHandler(&myMessageHandler{})

	// Use nsqlookupd to discover nsqd instances.
	// See also ConnectToNSQD, ConnectToNSQDs, ConnectToNSQLookupds.
	err = consumer.ConnectToNSQLookupd("localhost:4161")
	if err != nil {
		log.Fatal(err)
	}

	// wait for signal to exit
	sigChan := make(chan os.Signal, 1)
	signal.Notify(sigChan, syscall.SIGINT, syscall.SIGTERM)
	<-sigChan

	// Gracefully stop the consumer.
	consumer.Stop()
}
