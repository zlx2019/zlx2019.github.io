Rust所有权模型

前言

Rust的所有权模型是一种内存管理机制，它通过在编译时检查来确保内存安全。该模型强制执行一组规则，确保程序不会出现内存泄漏，使用未初始化的内存或访问已释放的内存等错误。

在Rust中，每个值都有一个被称为所有者的变量。在任何给定时间，每个值只能有一个所有者。当所有者超出其范围时，该值将被销毁并释放其内存。

Rust使用“借用”机制来允许临时访问值的引用，而不将所有权转移给另一个变量。这些引用具有生命周期，在其生命周期结束时，它们将自动失效，这确保了对于值的访问是安全的。

此外，Rust还提供了一些其他功能来支持所有权模型，例如可移动性和所有权转移。这些功能允许值在所有者之间移动，而无需复制整个值。这可以提高性能并避免错误。

简而言之,Rust的无GC机制就是通过所有权模型来实现的,所以理解Rust的所有权模型是作为Rust开发者最核心重要的部分内容.

常见的内存管理模型大致分为三类:

垃圾回收器

通过垃圾回收器对内存进行管理,这是目前较为常见解决方案,大多与编程语言都基于这种方式管理.如Java、Go、C#、Python等语言。可以看到,这些都是目前非常主流的编程语言,它们都选择了这种方案足以证明了这种方式的优秀。优势有如下几点:

  • 减少内存泄漏的可能性：使用垃圾回收机制可以避免因为程序员未及时释放内存而导致的内存泄漏。
  • 简化程序员的工作：程序员无需手动管理内存，可以更加专注于业务逻辑的实现。
  • 提高程序的健壮性：垃圾回收可以帮助程序自动清理不再使用的对象，从而减少因为内存溢出而导致程序崩溃的可能性。

当然,再优秀的方案也会有一定的劣势,如下:

  • 垃圾回收需要遍历内存中的所有对象，因此在某些情况下可能会影响应用程序的性能。
  • 垃圾回收需要维护内存中的对象，因此会占用一定的内存。

手动控制内存

这个就不用多说了吧,如C和C++出了名的难学，原因就是因为要手动管理内存的分配与释放.

这种方案的优势很明显,因为开发者可以完全控制内存的使用,也就意味着可以对内存进行最细粒度的使用,那么就可以实现程序运行效率的最大化。

当然有利也有弊,正确的使用会使程序具有极高的效率,但是如果使用不正确,那也会有非常严重的结果.而且学习难度极高.

所有权模型

所有权模型是结合了前两种的方案的优缺点,所产生的一种新的方案.其实这种方式,本质上比手动控制内存还要复杂,编写的效率甚至不如手动控制内存的效率,因为开发者要经常思考和考虑所有权的转移和分配。而且它的学习门槛会更高,不是空穴来风哦,James Gosling在采访中公开吐槽Rust太难学了~

其理念是: 让开发者在所有权模型和编译器检查的双重监管之下,在编写程序的前期花更多的时间尽可能检查出所有的内存问题。

OwnerShip

Rust中的OwnerShip所有权规则:

  1. 在Rust中的每一个值都会存在一个对应的被称为所有者owner的变量。
  2. 并且在同一时间,一个值只能有一个所有者owner。
  3. 当所有者owner离开当前作用域时,那么值将自动被丢弃。

以上三点规则非常重要,贯彻了整个Rust开发流程,请牢记于心。

如下代码所示:

fn main(){
  
  //1. 创建一个新的作用域
  {
    
    //2. 目前这里text无效,因为还没有声明
    
    let text = "Hello"; //3. 声明一个变量,此时text开始生效,并且text是Hello的所有者.
    
    //...
  
  } // 到这里时,作用域结束,text将不再生效,而Hello会失去所有者,从而导致被丢弃。
}

下面讲述一下关于所有权的行为语义.

1. 所有权-转移

其语义为 move,表示将一个值的所有权,转移到给另一个值身上。然后原来的值就会被丢弃掉,因为它的所有权已经没了。

如下代码所示:

let s1 = String::from("Hello"); // 创建一个s1变量

let s2 = s1; //value moved here 将s1所有权移动给s2,此时s1会失效,无法使用

println!("{}",s1)// borrow of moved value: `s1` 这个时候如果再对s1进行操作,编译器则会检查到.

编译这段代码你会发现,这是不通过的。

error[E0382]: borrow of moved value: `s1`
  --> src/main.rs:24:19
   |
20 |     let s1 = String::from("Hello"); 
   |         -- move occurs because `s1` has type `String`, which does not implement the `Copy` trait
21 | 
22 |     let s2 = s1; 
   |              -- value moved here
23 | 
24 |     println!("{}",s1)
   |                   ^^ value borrowed here after move

borrow of moved value: s1,表示变量s1已经不存在了,无法继续使用。

移动值的时候有两种语义: move和clone。很显然,这个案例就是使用的move语义,在let s2 = s1的时候,s1的所有权就已经转移到了s2变量,所以s1已经被丢弃。

2. 所有权-克隆

如果我们确实需要一个值进行深度复制,但是又不想旧的值失效那该怎么办呢?可以使用一个叫做clone的函数。

如下修改:

let s1 = String::from("Hello"); // 创建一个s1变量

let s2 = s1.clone(); // 将s1的值克隆一份,并且将克隆出来的值赋值给s2

println!("{}",s1);// 这种情况下s1依然有效

//Hello

可以通过clone的方式来进行深度拷贝。

3. 所有权-拷贝

对于只存在于栈上的数据,可以无需clone,直接会将值拷贝到一个新的值上.如下:

let x = 5;
let y = x;
println!("x = {}, y = {}", x, y);

//x = 5, y = 5

但这段代码似乎与我们刚刚学到的内容相矛盾：没有调用 clone，不过 x 依然有效且没有被移动到 y 中。

原因是像整型这样的在编译时已知大小的类型被整个存储在栈上，所以拷贝其实际的值是快速的。这意味着没有理由在创建变量 y 后使 x 无效。换句话说，这里没有深浅拷贝的区别，所以这里调用 clone 并不会与通常的浅拷贝有什么不同，我们可以不用管它。

Rust 有一个叫做 Copy trait 的特殊标注，可以用在类似整型这样的存储在栈上的类型上（第 10 章详细讲解 trait）。如果一个类型实现了 Copy trait，那么一个旧的变量在将其赋值给其他变量后仍然可用。

如下是一些 Copy 的类型：

  • 所有整数类型，比如 u32。
  • 布尔类型，bool，它的值是 true 和 false。
  • 所有浮点数类型，比如 f64。
  • 字符类型，char。
  • 元组，当且仅当其包含的类型也都实现 Copy 的时候。比如，(i32, i32) 实现了 Copy，但 (i32, String) 就没有。

4. 所有权-引用

如果在rust中,我们一个变量只想用一下另一个变量的值,但是又不想将所有权来回转移,那么就可以通过引用的方式来实现.

通过&表示引用,如&String就表示String类型的引用. 如下代码:

// 通过字符串的引用
// 计算一个字符串的长度,并且返回
fn calculate_size(s: &String) -> usize{
    s.len()
}

fn main() {
    let str = String::from("Hi,Hello");

    // 将str的引用传入函数,不需要转移所有权
    let len = calculate_size(&str);
    println!("str: {},长度为: {}",str,len)
}

//str: Hi,Hello,长度为: 8

如上所示,我们将str的引用作为参数传入函数内,并且函数内通过str的引用获取长度并且返回,并且在函数作用域结束之后在main函数内依然可以访问str变量。因为传递引用并不会转移所有权。

注意: 引用默认是不可变的,也就是说我们没办法通过引用直接修改值。如果想通过引用去修改值,那么则需要通过可变引用。

5. 所有权-可变引用

引用正如变量一般,默认是不可变的,所以如果想要通过引用修改变量,则需要通过可变引用。如下所示:

fn calculate_size(s: &mut String) -> usize{
    // 通过可变引用,对值进行追加字符
    s.push_str(",天气真好~");
    s.len()
}

fn main() {
    // 声明一个可变的字符串变量
    let mut str = String::from("Hi,Hello");
    // 传入可变引用作为参数
    let len = calculate_size(&mut str);
    println!("str: {},长度为: {}",str,len)
}

//str: Hi,Hello,天气真好~,长度为: 22

&表示为引用,而&mut表示为可变引用。

通过&mut就可以获取一个变量的可变引用,但是前提是变量必须是mut修饰的。

需要注意的是,可变引用有一些的限制性。

限制一: 在同一时间,一个变量不允许同时存在两个可变引用。

let mut msg = String::from("Hello");
let mut_ref1 = &mut msg;
let mut_ref2 = &mut msg;
println!("{},{}",mut_ref1,mut_ref2)

//error[E0499]: cannot borrow `msg` as mutable more than once at a time

原因: 如果同时存在多个可变引用的话,那就意味着可能存在同时读写操作,从而产生数据安全性问题。所以rust之间从根源上解决此问题。

本质上来说,可变引用可以拥有多个,但是不能同时拥有。如下所示:

fn main() {
    let mut s = String::from("hello");
    {
        let r1 = &mut s;
    } // r1 在这里离开了作用域，所以我们完全可以创建一个新的引用
    // 创建一个新的可变引用
    let r2 = &mut s;
}

限制二: 在某个特定的作用域里,如果已经存在某个变量的引用,那么就不能在该作用域添加该变量的可变引用。

fn main() {
    let mut msg : String = String::from("Hello");
    {
        // 获取msg的引用
        let r1: &String = &msg;
        let r2: &String = &msg;
        // 获取msg的可变引用
        let r3 : &mut String = &mut msg; // 报错: ^^^^^^^^ mutable borrow occurs here

        println!("{},{}",r1,r2)
    }
}

原因: 通常我们对变量的引用都是进行读取操作,编译器通常对变量的引用预期就是不可变的,但是如果在同一个作用域里还存在一个可变引用,那么就会导致变量的行为变得不可预测。rust是禁止这种情况的产生。

但是如果我们稍稍修改下代码,你会发现结果又有所不同,如下:

fn main() {
    let mut msg : String = String::from("Hello");
    {
        // 获取msg的引用
        let r1: &String = &msg;
        let r2: &String = &msg;
        println!("{},{}",r1,r2);
        // 获取msg的可变引用
        let r3 : &mut String = &mut msg;
    }
}
// Hello,Hello

编译通过。因为编译器已经检测到了我们在定义了r3可变引用之后,并没有再次操作r1和r2两个引用.所以编译器认为这是安全的。因为在编译器看来,执行了println!("{},{}",r1,r2);这行代码后,`r1和r2的作用域就已经结束了。

6. 悬挂引用

在具有指针的语言中，很容易通过释放内存时保留指向它的指针而错误地生成一个 悬垂指针（dangling pointer），所谓悬垂指针是其指向的内存可能已经被分配给其它持有者。相比之下，在 Rust 中编译器确保引用永远也不会变成悬垂状态：当你拥有一些数据的引用，编译器确保数据不会在其引用之前离开作用域。

fn dangle() -> &String { // dangle 返回一个字符串的引用

    let s = String::from("hello"); // s 是一个新字符串

    &s // 返回字符串 s 的引用
} // 这里 s 离开作用域并被丢弃。其内存被释放。
  // 危险！

7. 所有权与函数

将值传递给函数在语义上与给变量赋值相似。向函数传递值可能会move或者copy，就像赋值语句一样。

如对函数传入一个String类型的值: 本质上就是对所有权进行了move操作，与直接进行=赋值是一样的。

fn show_message(message: String){
    println!("{}",message)
}

fn main() {
    let message = String::from("Hello");
    show_message(message); // move
    println!("{}",message) //error[E0382]: borrow of moved value: `message`
}

但是如果对函数传入一个值类型如i32的值,那么实际上时copy的语义,这与之前的完全一致:

fn main() {
    let n1 = 10;
    let n2 = 20;
    show_number(n1,n2); //n1=10 n2=20
}

fn show_number(n1: i32,n2 : i32){
    println!("n1={} n2={}",n1,n2)
}

8. 所有权与返回值

函数的返回值,同理也是转移所有权。

fn gives_ownership() -> String {           // gives_ownership 将返回值移动给
                                           // 调用它的函数
  
  let some_string = String::from("yours"); // some_string 进入作用域

  some_string                              // 返回 some_string 并移出给调用的函数
}

fn main() {
  let s1 = gives_ownership();         // gives_ownership 将返回值
}

当gives_ownership()函数执行完之后,返回参数的所有权就被转移到了s1身上.函数内的some_string就被自动丢弃了。

所有权可以转移给函数后,再次返回得到新的所有权:

fn main() {
  let s2 = String::from("hello");     // s2 进入作用域

  let s3 = takes_and_gives_back(s2);  // s2 被移动到
                                      // takes_and_gives_back 中,
                                      // 它也将返回值移给 s3
}

// takes_and_gives_back 将传入字符串并返回该值
fn takes_and_gives_back(a_string: String) -> String { // a_string 进入作用域

  a_string  // 返回 a_string 并移出给调用的函数
}
  1. 如上所示,首先创建了s2变量,然后将所有权转移给了takes_and_gives_back()函数的a_string参数.
  2. 在takes_and_gives_back()函数内部,a_string参数又将所有权作为返回值转移了出来。
  3. a_string的所有权最终转移到了s3身上。

9. 总结

  • 对于特定的变量数据,在同一时间只能存在一个可变引用,或者多个引用。
  • 引用的数据必须是有效的,不要出现悬挂引用的情况。
