Java【类装载器】

发表于2022-04-06|更新于2023-03-22|Java
|字数总计:4.3k|阅读时长:15分钟|阅读量:

前言

顾名思义,它是用来加载Class的。它负责将Class 的字节码形式转换成内存形式的Class对象。字节码可以来自于磁盘文件 .class,也可以是jar包里的 .class，也可以来自远程服务器提供的字节流，字节码的本质就是一个字节数组 []byte,它有特定的复杂的内部格式。

每个 Class 对象的内部都有一个 classLoader 字段来标识自己是由哪个 ClassLoader 加载的。ClassLoader 就像一个容器，里面装了很多已经加载的 Class 对象。

1. 类加载的三个阶段

1.加载

加载阶段其实就是查找后缀为.class文件,并且将这些类的二进制数据 读取到内存中,将其放在方法区中,方法区主要用于存放类加载的一些元信息(类的数据结构),然后再在堆中创建一个java.lang.Class对象,这个对象就是访问该类所存放在方法区的数据结构信息的入口。

2.链接

链接阶段又分为三个小阶段分别为 验证、 准备、 解析

  • 验证

    验证阶段 主要负责确保类文件的正确性。

    1. 文件格式验证

      校验类文件的魔术因子是否正确(.class文件魔术因子: 0XCAFEBABE)、编译版本与本地虚拟机版本是否一致等。

    2. 元数据校验

      主要是对字节码描述的信息进行语义分析。是否有父类、是否继承了不允许被继承的类(final修饰的类)、是否实现了父类或者接口的抽象方法等等。

    3. 字节码验证

      主要是进行数据流和控制流的分析, 如 声明了一个Int类型 但是赋值的值是long类型。

    4. 符合引用验证

      是否可以通过所依赖类的全类名找到对应的类，在指定的类中是否存在所依赖或使用的属性、方法符号引用中的类、字段、方法的可访问性。防止调用了不存在的方法 或者属性。

    验证阶段是一个非必须执行的过程,可以通过 -Xverify:none参数来关闭

  • 准备

    准备阶段主要为类的静态变量分配内存，并且将其初始化为默认值。

    private static int num = 3;

    在该阶段num将会被分配到内存且初始化为 0。

    private static final int num = 3;

    但如果是被final关键字所修饰的静态常量,在该阶段会被直接初始化为3。

  • 解析

    解析阶段主要是将符号引用替换成直接引用。简单点来说就是把我们编写的基础代码编译成虚拟机在运行期间能够直接运行的指令,如下所示

    在一个类中声明一个静态变量

    class User{
    public static final String name = "张三";
}

    然后去调用该变量,我们所写的调用代码就是符号引用

    System.out.println(User.name);//符号引用

    接下来我们通过javap -v查看编译后的类

    3: getstatic #3 // Field com/zero/java8/loader/User.name:Ljava/lang/String;

    编译成字节码后则为直接引用,这就是解析阶段主要责任。

3.初始化

该阶段主要做两件事 为静态变量设置初始化值和执行静态代码块的代码。

依照静态变量和静态代码块的编写顺序,从上到下执行。分别为静态变量设置初始值和执行静态代码块。

如果一个静态变量在静态代码块之后声明,那么只能对该变量进行写操作,无法读取。

如果有父类,那么会先执行父类的静态资源。

这个初始化过程是线程安全的。

2. 类的主动使用

类的主动使用导致Class被加载的场景

  1. 直接通过New关键字去实例化。

    这是最常用也是最常见的方式,整个类都会被实例化。

  2. 调用一个类的静态变量或者对该静态变量进行赋值。

  3. 调用一个接口的静态变量 or 静态方法。

  4. 对一个类进行反射也会触发类加载。

  5. 初始化一个子类,同样会触发父类的加载。但是如果初始化父类,不会触发对子类的初始化

  6. 启动该类的main方法

3. 类的被动引用

类的被动引用,不会导致Class所初始化的场景

  1. 通过子类调用父类的静态变量或者静态方法,只会初始化父类,并不会初始化子类。

    class User{
    public static String name = "张三";
    static {
        System.out.println("User Class load");
    }
    public static void show(){
        System.out.println(name);
    }
}
class Student extends User{
    static {
        System.out.println("Student Class load");
    }
}

public static void main(String[] args) {
  //通过 Student调用User父类静态资源
	System.out.println(Student.name);
  Student.show();
}

    结果只初始化了父类

    User Class load
张三

  2. 通过数组引用,同样也不会触发初始化

    User[] users = new User[1];

  3. 直接调用一个类的静态常量同样也不会触发初始化

    class User{
    public static final String name = "张三";
    static {
        System.out.println("User Class load");
    }
}

public static void main(String[] args) {
  System.out.println(User.name);
}

    结果

    张三

    原因是因为被final修饰的常量在编译的时候就会直接放入常量池。

    但是会有些特殊的场景会有些不同,如下:

    class User{
    public static final int VALUE = new Random().nextInt(100);
    static {
        System.out.println("User Class load");
    }
}
public static void main(String[] args) {
  System.out.println(User.VALUE);
}

    结果

    User Class load
15

    你会发现,同样是被final修饰的常量,但是这种情况则初始化了类

    原因是因为 虽然VALUE是常量,但是赋予它的值不是常量,因为这个值只有在初始化阶段才会计算得出

4. 类加载案例测试

下面我们通过两个小案例来回顾一下以上掌握的知识

public class Singleton {
    public static int A = 0;
    public static int B;
    private static Singleton instance = new Singleton();
    public Singleton(){
        A++;
        B++;
    }
    public static Singleton getInstance(){
        return instance;
    }
    public static void main(String[] args) {
        getInstance();
        System.out.println(Singleton.A);
        System.out.println(Singleton.B);
    }
}

输出结果

1
1

结果和我们所想的一样,那么接下来我们调整一下代码

public class Singleton {
    private static Singleton instance = new Singleton();
    public static int A = 0;
    public static int B;
    public Singleton(){
        A++;
        B++;
    }
    public static Singleton getInstance(){
        return instance;
    }
    public static void main(String[] args) {
        getInstance();
        System.out.println(Singleton.A);
        System.out.println(Singleton.B);
    }
}

再次执行结果

0
1

这次的结果就很有意思了,我们只是调整了代码的书写顺序,其他都没动,但是结果却不同,那么这是为什么呢

我们接下来一步步去解析Class在初始化阶段的步骤,对于此案例我们只需要关注链接阶段的准备阶段与初始化阶段即可。

首先分析下改动代码前的案例

  1. 在准备阶段,我们首先为静态变量分配内存,以及初始化默认值
A=0	//int类型默认值为0
B=0	//int类型默认值为0
instance=null //引用类型默认值为null
  1. 初始化完默认值后,再进行初始化阶段,正确赋值我们指定的初始值
A=0 //这个0 与 准备阶段的0不同,这是我们指定的初始值,而并非int类型的默认值
B=0	//B没有指定初始值,直接使用int类型默认值
instance = new Singleton();//这个时候会实例化Singleton,会执行它的构造函数

//执行构造函数
A = A++
B = B++
//构造函数执行后
A = 1
B = 1

分析后的结果与执行结果一致,接下来分析改动后的案例

  1. 在准备阶段,我们首先为静态变量分配内存,以及初始化默认值
instance=null //引用类型默认值为null
A=0	//int类型默认值为0
B=0	//int类型默认值为0
  1. 初始化完默认值后,再进行初始化阶段,正确赋值我们指定的初始值
instance = new Singleton();//由于我们改动了代码顺序,所以先实例化了Singleton
//执行构造函数
A = A++
B = B++
//构造函数执行后
A = 1
B = 1

//注意,这个时候结果基本与改动前一致,但是我们的A与B此时还没有进入初始化阶段,接下来初始化A、B
A = 0 //将我们指定的初始值赋值到A变量上,这也是问题的所在,我们通过构造函数产生的值被初始值所覆盖了
B = 1 //由于B没有初始值,所以无需初始化,它的值还是为1

通过解析步骤分析,如果都是普通变量或者都是静态变量,代码执行的顺序同样是从上到下执行,但是如果两中变量交互操作的情况下需要慎重书写代码的顺序。

5. 类加载方式有哪些

  1. 本机磁盘加载(加载我们本机的jdk源码)
  2. 内存中直接加载,如动态代理
  3. 通过网络加载
  4. 从zip、jar资源包中加载
  5. 动态编译

6. 类加载的最终产物

经过类加载后最终的东西是位于堆区中的Class<?> 对象 和 存放在方法区中的类元信息

如下所示

User user = new User();

如果User是初次使用,那么经过ClassLoader加载后,在堆中会存在一个Class<User>对象,这个对象只会存在一份,然后这个User有哪些变量属性,方法结构等等这些结构信息都会存储在方法区中。

实例化对象的过程:

会通过Class<User>对象 映射到方法区中,获取到该类的结构元信息,进行创建实例化,实例化后的实例对象会存储在堆中。每实例化一个实例对象,都会存储在堆中。而变量名的引用则会存储在栈贞中的局部变量表中。

User user = new User();
//1. User Class<User>对象 在堆中
//2. user 变量名 存贮在局部变量表中
//3. new User(); 实例对象 存储在堆中

关联关系:

由变量引用 —> 实例对象 —> 类元信息

7. 三大类加载器

  1. BootStrapClassLoad

    根类加载器,有C++语言编写,主要负责加载jer/lib核心类库。

  2. ExtClassLoader

    扩展类加载器,主要加载jre/lib/ext下的扩展类库。

  3. AppClassLoader

    系统类加载器,通常我们自己编写的classpath下的类都由该加载器加载。

8. 自定义类加载器

package com.zero.java8.loader;

import java.io.ByteArrayOutputStream;
import java.io.File;
import java.io.FileInputStream;
import java.io.IOException;

/**
 * 自定义ClassLoader 加载指定的class文件
 *
 * @author Zero.
 * @date 2022/4/7 11:45 上午
 */
public class CustomClassLoader extends ClassLoader {

    //默认加载类的所在目录
    private final static String DEFAULT_PATH = "/usr/local/app";

    //指定加载类所在目录,优先级大于默认所在目录
    private String path = DEFAULT_PATH;

    //该加载器名称
    private String classLoaderName;

    //构造函数
    public CustomClassLoader(){
        super();
    }
    //构造函数,并且指定加载器名称
    public CustomClassLoader(String classLoaderName){
        super();
        this.classLoaderName = classLoaderName;
    }
    //构造函数,指定加载器名称和父类加载器,如果不指定则默认为AppClassLoader
    public CustomClassLoader(String classLoaderName,ClassLoader parent){
        super(parent);
        this.classLoaderName = classLoaderName;
    }
    public void setPath(String path) {
        this.path = path;
    }
    public String getClassLoaderName() {
        return classLoaderName;
    }
    public void setClassLoaderName(String classLoaderName) {
        this.classLoaderName = classLoaderName;
    }

    /**
     * 重写查找类文件方法
     * @param name  类全类名
     * @return      Class<?> 类信息
     * @throws ClassNotFoundException   未找到类文件异常
     */
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        String classPath = name.replace(".","/");
        File classFile = new File(path,classPath + ".class");
        if (!classFile.exists()){
            throw new ClassNotFoundException("class file not found");
        }
        //读取class文件转为 字节数组
        byte[] classBytes = loadClassBytes(classFile);
        if (null == classBytes || classBytes.length == 0){
            throw new ClassNotFoundException("load class file error");
        }
        //加载Class
        return this.defineClass(name,classBytes,0,classBytes.length);
    }

    /**
     * 读取class文件内容
     * @param classFile 要加载的类二进制文件
     * @return  字节数组
     */
    private byte[] loadClassBytes(File classFile) {
        try (ByteArrayOutputStream baos = new ByteArrayOutputStream();FileInputStream fis = new FileInputStream(classFile)){
            byte[] buffer = new byte[1024];
            int len;
            while ((len = fis.read(buffer)) != -1){
                baos.write(buffer,0,len);
            }
            baos.flush();
            return baos.toByteArray();
        }catch (IOException e){
            e.printStackTrace();
            return null;
        }
    }
}

}

结果显示还是系统类加载器所加载该类,那么这是为什么呢?

原因是因为Java的类加载器有一个双亲委派机制,也就是说当一个类加载器接到一个加载请求时,会优先交给自己的父类加载器去加载,依次类推,直到所有的父类加载器都无法在自己所负责加载的内容中找到该类,才会把请求依次返回给子类加载器,目的就是防止我们自己编写的类与核心类库发生冲突,从而破坏了核心类库文件。

所以我们只要删除项目中classpath下编译好的Student.class文件即可。

再次测试

但是并没有触发类的初始化,通过类加载器的loadClass方法并不会触发类的加载,因为这个操作并不属于主动加载。

通过类加载器后,在堆中,会产生一个Class<Studnet> 对象,并且只会生成一个,那么这种说法到底是否正确呢?接下来我们来测试一下。

public static void main(String[] args) throws Exception {
    CustomClassLoader loader = new CustomClassLoader("CustomClassLoader");
    Class<?> student1 = loader.loadClass("com.zero.java8.loader.Student");
    Class<?> student2 = loader.loadClass("com.zero.java8.loader.Student");

    System.out.println(student2.hashCode());
}

哦豁,结果出现了意外,也就是说不同的类加载器加载类后所生成的Class<?>对象 并不是同一个,只有同一个加载器的情况下才只会生成一个Class对象。因为ClassLoader有命名空间的区别。

9. 双亲委派机制

这个机制我们在上一章节有提到,该机制的优点就是能提高系统的安全性, 在此机制下,用户自定义的类加载器不可能加载应该有父类加载器所加载的可靠类,因此可以防止恶意代码代替父加载器的可靠代码。

我们来简单写个案例。

编写一个String类,并且 package为java.lang,与jdk自带的String完全相同

package java.lang;
/**
 * @author Zero.
 * @date 2022/4/7 1:15 下午
 */
public class String {
    static {
        System.out.println("custom string");
    }
}
public class Main {
    public static void main(String[] args) {
        String arg = new String();
        System.out.println(arg);
    }
}

我们什么都不做,简单实例下,然后运行

Error: A JNI error has occurred, please check your installation and try again
Exception in thread "main" java.lang.SecurityException: Prohibited package name: java.lang

编译器直接报错,因为我们定义的String与自带的String发生冲突,这个错误是由编译器发现并且阻拦的,并不能体现出双亲委派机制,所以我们换一种方式,通过反射来创建String

public static void main(String[] args) throws ClassNotFoundException {
  Class<?> aClass = Class.forName("java.lang.String");
  System.out.println(aClass.getClassLoader()); //Result: null
}

我们知道通过反射获取Class是主动加载,所以会导致类的初始化,但是可以看到我们自定义的String的静态代码块并没有被执行,而且它的ClassLoader是null,这就表示,这个String是由根加载器所加载的,并非我们恶意创建的String。这就是该机制的好处,自己定义的类如果与核心类库发生冲突,会被丢弃掉,只会加载优先级更高的核心类库文件。

10. 命名空间

每个类加载器都有属于自己的命名空间,命名空间由该加载器的所有父类加载器组成。在同一个命名空间中,不会出现重复的名字。这也是为什么不同的ClassLoader创建出的Class对象不是同一个的原因所在。

11. 运行时包

父类加载器看不到子类加载器所加载的类。不同命名空间下的类加载器之间,是相互隔离的。

12. 类的卸载

虚拟机中的Class只有满足以下条件才会被GC回收,也就是Class的卸载

  1. 该类的所有实例对象全部被回收掉
  2. 加载该类的ClassLoader实例被回收掉
  3. 该类的Class对象 没有被引用的情况下

核心类库的Class不会被回收,会伴随着虚拟机的整个生命周期
