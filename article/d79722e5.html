GRPC-Protobuf篇章

1. Protobuf是什么

Protocol buffers 提供了一种语言中立、平台中立、可扩展的机制，用于以向前兼容和向后兼容的方式序列化结构化数据。它类似于 JSON，只是它更小更快，并且它生成本地语言绑定。您一旦定义了数据的结构化方式，就可以使用特殊生成的源代码轻松地将结构化数据写入各种数据流并使用各种语言从中读取结构化数据。

Protocol buffers 是目前非常主流的一种用于数据传输的数据结构。它们广泛用于服务器间高性能通信。序列化后的体积要比XML、JSON要小很多，并且它的序列化与反序列化速度也比XML、JSON快很多.是直接把对象和字节数组做转换，而XML和JSON还需要构建成XML或者JSON对象结构。

2. Protobuf语法认识

  • syntax 语法版本

    syntax = "proto3";

    指定当前proto的版本,如果不指定则默认为proto2。

  • package 包名

    package user;

    指定proto的package(包名)。

    注意:这里指定是当前proto的package,而不是编译后的各种语言的package,我们直到很多编程语言里都有package的概念,但这里跟那些没有关系,只是当前proto的package,用来给proto分包和引用使用的。

  • option 配置 

    option go_package = "user;user";

    用来配置一些属性,如编译后的语言的package命名规范等.

    go的package较为复杂,需要如下编写:

    假如我们的mod为com.zero/protobuf-demo

    // option go_package = "包名;别名";
option go_package = "./user;user";

    如果go_package这样指定,编译时--go_out=/usr/local/protobuf-demo/pb.

    那么编译后就会在这个最后的pb目录中创建一个user文件夹,然后产生的go文件就存放在这个文件夹中,并且go文件的package也叫user。

    ├── pb
│   └── user
│       └── user.pb.go

  • message 消息定义

    message UserRequest{

}

    用于定义Proto中消息的关键字,也可以称为实体.类似于Java中的class。Go中的struce。

  • service 服务定义

    service User{

}

    用于定义服务的关键字,服务内定义rpc接口.

  • rpc 接口定义

    service User{
  rpc Users(UserRequest)returns(UserResponse);
}

  • enum 定义

    enum WEEK{
  DEFAULT = 0;
}

    用于定义枚举的关键字。

  • double

  • sint32
  • sint64
  • bool
  • string
  • bytes
  • fixed32
  • fixed64
  • sfixed32
  • sfixed64

还有一些Google公司提供好的常用Proto:

  • Duration 是有符号的、固定长度的时间跨度，例如 42 秒。
  • Timestamp 是独立于任何时区或日历的时间点，例如 2017-01-15T01:30:15.01Z。
  • Interval 是独立于时区或日历的时间间隔，例如2017-01-15T01:30:15.01Z - 2017-01-16T02:30:15.01Z。
  • Date 是一个完整的日历日期，例如 2025-09-19。
  • DayOfWeek 是一周中的某一天，例如星期一。
  • TimeOfDay 是一天中的某个时间，例如 10:42:23。
  • LatLng 是纬度/经度对，例如 37.386051 纬度和 -122.083855 经度。
  • Money 是金额及其货币类型，例如 42 美元。
  • PostalAddress 是邮政地址，例如 1600 Amphitheatre Parkway Mountain View, CA 94043 USA。
  • Color 是 RGBA 颜色空间中的一种颜色。
  • Month 是一年中的一个月，例如四月。

定义案例如下:

// 引入Google提供好的时间类型
import "google/protobuf/timestamp.proto";
message PersonRequest{
  uint64 id = 1;
  string user_name = 2;
  string password = 3;
  bool locked = 4;
  int32 age = 5;
  bytes buf = 6;
  google.protobuf.Timestamp send_time = 7;
}

可以看到,一个消息中的每一个字段都有一个唯一的编号。这些字段编号用于在 消息二进制格式中标识您的字段，一旦您的消息类型被使用，就不应更改

请注意，1 到 15 范围内的字段编号需要一个字节进行编码，包括字段编号和字段类型（您可以在Protocol Buffer Encoding中找到更多相关信息 ）。16 到 2047 范围内的字段编号占用两个字节。因此，您应该为非常频繁出现的消息元素保留数字 1 到 15。请记住为将来可能添加的频繁出现的元素留出一些空间。

您可以指定的最小字段编号为 1，最大字段编号为 2 29 - 1，即 536,870,911。您也不能使用数字 19000 到 19999（FieldDescriptor::kFirstReservedNumber通过 FieldDescriptor::kLastReservedNumber），因为它们是为 Protocol Buffers 实现保留的——如果您在.proto. 同样，您不能使用任何先前保留的字段编号。

指定字段规则

字段的规则有如下几种:

  • singular

    表示该字段可传参也以为Null。使用proto3语法时，当没有为给定字段指定其他字段规则时，这是默认字段规则。

  • optional

    与 相同singular，只是您可以检查该值是否已明确设置。字段optional处于两种可能状态之一：

    • 该字段已设置，并包含一个明确设置或从线路解析的值。它将被序列化到电线。
    • 该字段未设置，并将返回默认值。它不会被序列化到电线。

  • repeated

    表示该字段是一个序列.重复值的顺序将被保留。

    简单来说就是表示这个字段是一个集合或者切片:

    message UserRequest{
 	repeated string name_list = 1;
}

  • map

    表示该字段是一个K-V键值结构.

    message UserRequest{
	map<string,StudentRequest> user_info = 1;
}

    其实与下面这种方案是一致的:

    message Test{
	message Entry{
		optional string key = 1;
		optional int32 value = 2;
	}
	repeated Entry = 1;
}

  • oneof

    表示将消息中的字段进行分组,每一组中有且只有一个字段会有返回值.

    如下所示: 定义一个Proto

    message Request{
	// jpg 和 pug 字段同时只能有一个有返回值
  oneof img_type{
    string jpg = 1;
    string png = 2;
  }
}

    在Go中构造该参数:

    request := &pb.Request{
  // 只能二选一传值
  //ImgType: &pb.Request_Jpg{"JPG格式"},
  ImgType: &pb.Request_Png{"Png格式"},
}

    服务端接受值:

    imgType := ""
jpg := req.GetJpg()
png := req.GetPng()
if jpg != "" {
  imgType = jpg
} else {
  imgType = png
}
fmt.Printf("收到gRPC Client 请求, message: %s  \n", imgType)

  • reserved

    主要用来禁止一些序号不可用.

    message PersonRequest{
  uint64 id = 1;       //编译器报出异常:Field 'id' uses reserved number 1
  string user_name = 2;//Field 'id' uses reserved number 2
  string password = 3;// Field 'id' uses reserved number 3
  bool locked = 4; //Field name 'locked' is reserved
  int32 age = 5;

  reserved 1 to 3; // 1-3之间的数字都不能作为唯一标识
  reserved "locked","test";//禁止使用的字段名
}

    所以只能换成别的序号来使用.

枚举

Protobuf中也是支持枚举的,结构定义如下:

enum SEX{
  UNKNOWN = 0;
  MAN = 1;
  WOMAN = 2;
}

对于枚举，默认值是第一个定义的枚举值，它必须是 0。

4. 牛刀小试-Go

定义一个数据类型较多的Proto,然后编译成Go文件看一下。

syntax = "proto3";
package user;
// 引入Google提供好的时间类型
import "google/protobuf/empty.proto";
import "google/protobuf/timestamp.proto";

option go_package = "./user;user";
// 生成代码时的优化,默认为:SPEED
// SPEED 提供解析与序列化速度
// CODE_SIZE 代码体积大小
// LITE_RUNTIME 优化message运行时
//
option optimize_for = SPEED;

message UserRequest{
  uint64 id = 1;
  string user_name = 2;
  string password = 3;
  bool locked = 4;
  int32 age = 5;
  bytes buf = 6;
  google.protobuf.Timestamp send_time = 7;
  optional string token = 8;
  repeated string name_list = 9;
  map<string,sint64> user_info = 10;
}

然后进行编译:

protoc --go_out=./pb --go-grpc_out=./pb  ./proto/user/*.proto

查看一下编译后的Go结构体

type UserRequest struct {
	state         protoimpl.MessageState
	sizeCache     protoimpl.SizeCache
	unknownFields protoimpl.UnknownFields

	Id       uint64                 `protobuf:"varint,1,opt,name=id,proto3" json:"id,omitempty"`
	UserName string                 `protobuf:"bytes,2,opt,name=user_name,json=userName,proto3" json:"user_name,omitempty"`
	Password string                 `protobuf:"bytes,3,opt,name=password,proto3" json:"password,omitempty"`
	Locked   bool                   `protobuf:"varint,4,opt,name=locked,proto3" json:"locked,omitempty"`
	Age      int32                  `protobuf:"varint,5,opt,name=age,proto3" json:"age,omitempty"`
	Buf      []byte                 `protobuf:"bytes,6,opt,name=buf,proto3" json:"buf,omitempty"`
	SendTime *timestamppb.Timestamp `protobuf:"bytes,7,opt,name=send_time,json=sendTime,proto3" json:"send_time,omitempty"`
	Token    *string                `protobuf:"bytes,8,opt,name=token,proto3,oneof" json:"token,omitempty"`
	NameList []string               `protobuf:"bytes,9,rep,name=name_list,json=nameList,proto3" json:"name_list,omitempty"`
	UserInfo map[string]int64       `protobuf:"bytes,10,rep,name=user_info,json=userInfo,proto3" json:"user_info,omitempty" protobuf_key:"bytes,1,opt,name=key,proto3" protobuf_val:"zigzag64,2,opt,name=value,proto3"`
}

前三个属性忽略即可,重点看我们自己定义的字段和类型.

5. 牛刀小试-Java

syntax = "proto3";
package user;
// 引入Google提供好的时间类型
import "google/protobuf/empty.proto";
import "google/protobuf/timestamp.proto";

// java相关
// java的package路径
option java_package = "com.zero.summer.user";
// 指定java类名
// 该类名不能和message等同名
option java_outer_classname = "UserProto";
// 是否将message enum service产生的文件生成到一个文件内
option java_multiple_files = true;
// 是否生成定义的service服务,如果为false，那怕定义了也不会生成.
option java_generic_services = true;

// 生成代码时的优化,默认为:SPEED
// SPEED 提供解析与序列化速度
// CODE_SIZE 代码体积大小
// LITE_RUNTIME 优化message运行时
option optimize_for = SPEED;

message UserRequest{
  uint64 id = 1;
  string user_name = 2;
  string password = 3;
  bool locked = 4;
  int32 age = 5;
  bytes buf = 6;
  google.protobuf.Timestamp send_time = 7;
  optional string token = 8;
  repeated string name_list = 9;
  map<string,sint64> user_info = 10;
}
service User{
  rpc Users(UserRequest)returns(google.protobuf.Empty);
}

然后进行编译:

protoc --java_out=./pb ./proto/user/*.proto

然后他会在当前目录的pb目录中根据java_package创建如下结构:

6. 编码规范推荐

1. 标准文件格式

  • 将行长度保持在 80 个字符以内。
  • 使用 2 个空格的缩进。
  • 更喜欢对字符串使用双引号。

2. 文件内容结构

  1. License header (if applicable)
  2. File overview
  3. Syntax 指定语法版本
  4. Package 包名
  5. Imports 导入第三方包
  6. options 配置
  7. Everything else 其他业务代码

3. Package

包名应该是小写的。包名称应该具有基于项目名称的唯一名称，并且可能基于包含协议缓冲区类型定义的文件的路径。

4. Message

消息结构命名应当为CamelCase（首字母大写）例如 UserInfoRequest。

消息的字段名应当使用小写+_的规范来命名,例如 user_name。

message UserInfoRequest{
	optional string user_name = 1;
}

如以上这种,在Java中生成的代码就会如以下所示:

public String getUserName() { ... }
public Builder setUserName(String v) { ... }

5. repeated Field

序列字段应当使用复数表示

message UserRequest{
	repeated string names = 1;
}

6. Enum

枚举类型名称使用 CamelCase（首字母大写），值名称使用 CAPITALS_WITH_UNDERSCORES：

enum FooBar {
  FOO_BAR_UNSPECIFIED = 0;
  FOO_BAR_FIRST_VALUE = 1;
  FOO_BAR_SECOND_VALUE = 2;
}

7. Service

如果您.proto定义了一个 RPC 服务，您应该为服务名称和任何 RPC 方法名称使用 CamelCase（首字母大写）：

service FooService {
  rpc GetSomething(GetSomethingRequest) returns (GetSomethingResponse);
  rpc ListSomething(ListSomethingRequest) returns (ListSomethingResponse);
}
