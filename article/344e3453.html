响应式编程

响应式编程（Reactive Programming）是一种新的编程范式，中文称响应式（或反应式）编程，是一种高性能应用的编程方式。其最早是由微软提出并引入到 .NET 平台中，随后 ES6 也引入了类似的技术。在 Java 平台上，较早采用反应式编程技术的是 Netflix 公司开源的 RxJava 框架。现在大家比较熟知的 Hystrix 就是以 RxJava 为基础开发的。

响应式编程来源于数据流和变化的传播，意味着由底层的执行模型负责通过数据流来自动传播变化。

比如求值一个简单的表达式 c=a+b，当 a 或者 b 的值发生变化时，传统的编程范式需要对 a+b 进行重新计算来得到 c 的值。如果使用反应式编程，当 a 或者 b 的值发生变化时，c 的值会自动更新。

在传统的编程范式中，我们一般通过迭代器（Iterator）模式来遍历一个序列。这种遍历方式是由调用者来控制节奏的，采用的是拉的方式。每次由调用者通过 next()方法来获取序列中的下一个值。使用反应式流时采用的则是推的方式，即常见的发布者-订阅者模式。当发布者有新的数据产生时，这些数据会被推送到订阅者来进行处理。在反应式流上可以添加各种不同的操作来对数据进行处理，形成数据处理链。这个以声明式的方式添加的处理链只在订阅者进行订阅操作时才会真正执行。

所需基础

响应式编程的基础：

  • 数据流的概念
  • Observable 类和它的各种操作
  • 通过工厂方法创建静态和动态的Observable 对象

Observable 是事件的源头，Observer 提供了一组简单的接口，并通过订阅事件源来消费 Observable 的事件。Observable 通过 onNext 向 Observer 通知事件的到达，后面可能会跟上 onError 或 onComplete 来表示事件的结束。

反应式编程其实并不神秘，通过与我们熟悉的迭代器模式对比便可了解其基本思想：

event Iterable (pull) Observable (push)
retrieve data T next() onNext(T)
discover error throws Exception onError(Exception)
complete !hasNext() onCompleted()

上面表格的中的 Observable 那一列便代表反应式编程的 API 使用方式。可见，它就是常见的观察者模式的一种延伸。如果将迭代器看作是拉模式，那观测者模式便是推模式。被订阅者（Publisher）主动的推送数据给订阅者（Subscriber），触发 onNext 方法。异常和完成时触发另外两个方法。

回压（Backpressure）

反应式流中第一个重要概念是回压（backpressure）。在基本的消息推送模式中，当消息发布者产生数据的速度过快时，会使得消息订阅者的处理速度无法跟上产生的速度，从而给订阅者造成很大的压力。当压力过大时，有可能造成订阅者本身的奔溃，所产生的级联效应甚至可能造成整个系统的瘫痪。负压的作用在于提供一种从订阅者到生产者的反馈渠道。订阅者可以通过 request()方法来声明其一次所能处理的消息数量，而生产者就只会产生相应数量的消息，直到下一次 request()方法调用。这实际上变成了推拉结合的模式。

回压是 RS 规范和 Reactor 主要关注点之一（如果还有其它关注点的话）。回压的原理是说，在一个推送场景里，生产者的生产速度比消费者的消费速度快，消费者会向生产者发出信号说“嘿，慢一点，我处理不过来了。”生产者可以借机控制数据生成的速度，而不是抛弃数据或者冒着产生级联错误的风险继续生成数据。

你也许会想，在 Mono 里为什么也需要回压：什么样的消费者会被一个单独的触发事件压垮？答案是“应该不会有这样的消费者”。不过，在 Mono 和 CompletableFuture 工作原理之间仍然有一个关键的不同点。后者只有推送：如果你持有一个 Future 的引用，那么说明一个异步任务已经在执行了。另一方面，回压的 Flux 或 Mono 会启动延迟的拉取 - 推送迭代：

  1. 延迟是因为在调用 subscribe() 方法之前不会发生任何事情
  2. 拉取是因为在订阅和发出请求时，Subscriber 会向上游发出信号，准备拉取下一个数据块
  3. 接下来生产者向消费者推送数据，这些数据在消费者的请求范围之内

对 Mono 来说，subscribe() 方法就相当于一个按钮，按下它就等于说“我准备好接收数据了”。Flux 也有一个类似的按钮，不过它是 request(n) 方法，这个方法是 subscribe() 的一般化用法。

核心概念

Flux 和 Mono 是 Reactor 中的两个基本概念。

  • Flux表示的是包含 0 到 N 个元素的异步序列。在该序列中可以包含三种不同类型的消息通知：正常的包含元素的消息、序列结束的消息和序列出错的消息。当消息通知产生时，订阅者中对应的方法 onNext(), onComplete()和 onError()会被调用。
  • Mono 表示的是包含 0 或者 1 个元素的异步序列。该序列中同样可以包含与 Flux 相同的三种类型的消息通知。

Flux 和 Mono 之间可以进行转换：

  • 对一个 Flux 序列进行计数操作，得到的结果是一个 Mono对象
  • 把两个 Mono 序列合并在一起，得到的是一个 Flux 对象。

简而言之,响应式编程对于开发者来说,就是通过Mono与Flux实现我们以往正常的开发过程.所以只要掌握好这两个基础异步组件的使用,那么就简单多了.

创建Flux

Flux类提供了大量通过由工厂创建的静态方法:

  • just(T… data): 通过指定的全部元素,构建出一个序列。如果在已知元素内容及数量的情况下,使用此方式最简单。
  • from(Publisher source): 根据一个已有的数据源(如Flux、Mono),创建出一个新的Flux序列。
  • fromArray(T[] array): 根据一个数组对象构建一个Flux序列。
  • fromStream(Stream s):根据一个Stream流构建一个Flux序列。
  • fromIterable(Iterable it):根据一个迭代器构建一个Flux序列。
  • range(int start, int count): 快速生成一个整数类型的数据流。元素从start开始递增,直到(start+count-1)元素停止递增。
  • interval(Duration delay, Duration period): 快速生成一个从0开始递增的Long类型序列,并且可以指定延迟多长时间后开始发布,以及间隔多长时间发布一次。
  • empty(): 创建一个不包含任何元素的序列,只发布结束消息的Flux序列。
  • error(Throwable err):创建一个只包含错误通知的Flux序列。
  • never(): 创建一个不包含任何消息通知的序列。

API使用

// just
Flux.just(1001, 1002, 1003).subscribe(System.out::println);

//from
// 根据一个已有的Flux 构建出一个新的Flux
Flux.from(Mono.just("lll")).subscribe(System.out::println);
// 将Mono转换成Flux
Flux.from(Flux.just(1,3,4)).subscribe(System.out::println);
// 根据一个数组创建一个Flux
Flux.fromArray(new String[]{"1","2","3"});
// 根据一个流创建一个Flux
Flux.fromStream(Stream.of(1,2,3));
// 根据一个可迭代的数据结构创建一个Flux
Flux.fromIterable(List.of(1,2,3,4));

// 根据Integer类型Flux序列,元素为从10开始递增,递增到29(start+count-1)
Flux<Integer> range = Flux.range(10, 20);

// 创建一个从0开始递增Long类型的Flux序列,每隔一秒发布一次.
Flux.interval(Duration.ofSeconds(1));
// 延迟五秒后开始发布,其后每隔一秒发布一次
Flux.interval(Duration.ofSeconds(5),Duration.ofSeconds(1));
// 创建一个空的Flux
Flux.empty();
// 创建一个异常元素Flux
Flux.error(new RuntimeException(""));
// 创建一个不包含任何消息通知的序列。
Flux.never();

以上是Flux的静态创建方法.下面介绍更灵活自主的创建方式:

  • generate(Consumer> generator)

    此方法通过传入一个入参为SynchronousSink的Consumer函数,序列的产生就是通过SynchronousSink来完成的.

    SynchronousSink方法如下:

    • next(T data): 发出非空元素，生成 onNext 信号。
    • complete(): 成功终止序列，生成 onComplete 信号。如果不调用该方法,将会死循环向序列中添加元素,永不停止。

    • error(Throwable e): 使序列失败，生成 onError 信号。

      方法通过同步和逐一的方式循环来产生 Flux 序列。

    在执行complete()方法之前,next()方法有且只能被调用一次。

    如果只调用了next()方法设置值,但没有调用complete()方法终止序列停止生成,则会永无休止的向序列中添加这个值。

    如下生成单个元素序列:

    // 将数字 1 添加到序列中
Flux.generate(sink->{
  sink.next(1);
  sink.complete();
}).subscribe(System.out::println);

  • generate(Callable[S] stateSupplier, BiFunction generator)

    通过一些状态条件生成指定数量的元素,第一个参数stateSupplier就是用来获取状态条件的函数,最终会作为参数传入BiFunction函数中作为条件,如下使用:

    // 创建一个元素从 1-5的序列
Flux.generate(()-> 1, (number,sink)->{
  // 将值设置到序列中
  sink.next(number);
  if (number == 5){
    // 如果值==5了 则停止生成
    sink.complete();
  }
  // 返回的是下一轮生成时的number值
  return ++number;
}).subscribe(System.out::println);

    如上所示,我们引入了一个初始值为1的number值,来作为停止生成元素的信号阈值,一旦number == 5则会触发complete()方法来终止.

    在BiFunction函数体内每次返回的值,就是下一次循环时的第一个参数值。

  • create(Consumer<? super FluxSink> emitter)

    create()方法与generate()方法不同之处在于使用的是FluxSink对象生成元素.

    FluxSink支持同步和异步的元素生成,并且可以一次调用中,产生多个元素。

    FluxSink方法如下:

    • next(T t): 发出非空元素，生成 onNext 信号。
    • complete(): 成功终止序列，生成 onComplete 信号。
    • error(Throwable e): 使序列失败，生成 onError 信号。
    • contextView(): 返回当前订阅者的上下文作为 ContextView 检查。
    • isCancelled(): 如果下游取消了序列，则返回 true。
// 创建元素1-10的Flux序列
Flux.create(sink -> {
  for (int i = 1; i <= 10; i++) {
    sink.next(i);
  }
  sink.complete();
}).subscribe(System.out::println);

创建Mono

Mono 最多只触发一个事件，它跟 RxJava 的 Single 和 Maybe 类似，所以可以把 Mono用于在异步任务完成时发出通知。

Mono 是一个发出(emit)0-1个元素的Publisher<T>,可以被onComplete信号或者onError信号所终止。

对于Mono而言,可以认为它是一种特别的Flux,所以它们有跟多相似的静态创建方法.如just()、empty()、error()、never()。这些方法含义与Flux中的一模一样。除此之外Mono还有一些独有的静态创建方法。

  • justOrEmpty(T data): 根据值创建一个Mono,如果值为Null则响应onComplete信号,反之将值作为元素发送。
  • justOrEmpty(Optional data): 与上边一致。
  • fromSupplier(Supplier supplier): 通过一个Supplier函数的返回值作为元素创建Mono
  • fromFuture(CompletableFuture future):通过一个Future返回值作为元素构建Mono
  • fromRunnable(Runnable runnable): 通过一个Runnable接口构建Mono
  • fromCallable(Callable supplier): 通过一个Callable接口构建Mono
  • fromCompletionStage(CompletionStage completionStage)： 通过一个CompletionStage构建Mono
  • create(Consumer> callback): 通过自定义方式构建Mono。
  • delay(Duration duration)和 delayMillis(long duration)：创建一个 Mono 序列，在指定的延迟时间之后，产生数字 0 作为唯一值。
  • ignoreElements(Publisher source)：创建一个 Mono 序列，忽略作为源的 Publisher 中的所有元素，只产生结束消息。

订阅消息

上面创建Flux和Mono都属于发布，只有订阅了数据才会开始发布.接下来详解一下如何对其进行订阅.

订阅是通用的哦~

  • subscribe(): 最简单的订阅消息,会忽略所有的消息通知.相当于直接丢弃。

  • subscribe(Consumer consumer)

    订阅所有的消息,它将请求无限需求 （Long.MAX_VALUE）。对每个来自于onNext()方法的通知的值进行消费。但是不处理onError()和OnComplete()的通知。

    next()发布 —> onNext()收到通知 —>subscribe()消费。

  • subscribe(Consumer consumer, Consumer< Throwable> errorConsumer)

    在上一个重载方法的基础上,增加对onError()通知的处理。

  • subscribe(Consumer consumer,Consumer< Throwable> errorConsumer,Runnable completeConsumer)

    在上一个重载方法的基础上,增加对OnComplete()通知的处理。

使用如下:

Flux.just(1,2,3,4,5)
  .subscribe(v->{
    System.out.println("消费数据: " +v);
  },err->{
    System.out.println("发送异常 :" + err.getMessage());
  },()->{
    System.out.println("数据消费完成");
  });

消费数据: 1
消费数据: 2
消费数据: 3
消费数据: 4
消费数据: 5
数据消费完成
public static void main(String[] args) {
  Flux.error(new RuntimeException("模拟异常"))
    .subscribe(v->{
      System.out.println("消费数据: " +v);
    },err->{
      System.out.println("发生异常 :" + err.getMessage());
    },()->{
      System.out.println("数据消费完成");
    });
}

//发生异常 :模拟异常

  • subscribe(Consumer consumer,Consumer< Throwable> errorConsumer,Runnable completeConsumer,Consumer< Subscription> subscriptionConsumer)

    在上一个重载方法上,提供了下游订阅者自己指定想要消费的消息数量或者停止上游服务发送消息,如下使用

    Flux.just(1,2,3,4,5)
  .subscribe(v->{
    System.out.println("消费数据: " +v);
  },err->{
    System.out.println("发生异常 :" + err.getMessage());
  },()->{
    System.out.println("数据消费完成");
  },(subscription)->{
    subscription.request(1);//只需要一个消息消费
    // or
    //subscription.cancel();//要求上游服务停止发送消息
  });

// 1

  • subscribe(Subscriber< T > actual)

    最常用的方式,可以对所有通知任意操作。

    Flux.just(1,2,3,4,5)
  .subscribe(new Subscriber<Integer>() {
    @Override
    public void onSubscribe(Subscription s) {
      // 向上游服务要求5条消息
      s.request(5);
    }
    @Override
    public void onNext(Integer integer) {
      System.out.println("消费数据: " + integer);
    }
    @Override
    public void onError(Throwable t) {
      System.out.println("发生异常: " + t.getMessage());
    }
    @Override
    public void onComplete() {
      System.out.println("消费结束!");
    }
  });

//消费数据: 1
//消费数据: 2
//消费数据: 3
//消费数据: 4
//消费数据: 5
//消费结束!

转换操作符

map(转换)

可以将序列中的元素转换为另一种元素。

// 将integer序列转换为String序列
Flux<String> flux = Flux.range(1, 10).map(Object::toString);

flatMap(转换)

可以将序列中的元素,映射为一个新的流,然后将流中的元素合并为一个新的流。

有些复杂,简单举个栗子: 我们根据一组userIds,获取所有的用户名称(一个人可能有多个名字),然后将这些名字合并为一个Flux序列.如下所示:

// 根据三个id,获取所有的名字,并且合并到一个序列中
Flux<String> map = Flux.just("1001", "1002", "1003")
  .flatMap(id -> Flux.just(id + "的第一个名字",id + "的第二个名字"));
List<String> names = map.toStream().collect(Collectors.toList());
System.out.println(names);


//[1001的第一个名字, 1001的第二个名字, 1002的第一个名字, 1002的第二个名字, 1003的第一个名字, 1003的第二个名字]

take(long n, boolean limitRequest)

如果 limitRequest == false 此运算符不传播请求的背压量。相反，它会发出一个无限的请求，并在发出 N 个元素后取消。如果 n 为零，则订阅源但立即取消，则运算符完成。
在这种模式下，尽管取消了，但源可能会产生许多无关的元素。如果该行为不可取，并且您不拥有来自下游的请求（例如预取运算符），请考虑改用 limitRequest = true 。

limitRequest – true 更密切地跟踪下游请求，并将上游请求限制为 n. false 从上游请求无限金额。

Flux.just(1,2,3,4,5).take(2,true).subscribe(System.out::println);

从Flux序列中按照指定的条件（continuePredicate）选取元素，直到条件不满足，就停止选取。它会返回一个Flux类型的对象，其中存放着满足条件的元素。

当该操作符的上游操作符都执行完之后,才会发布元素。一般用于做一些同步操作.

Flux.just("1001","1002")
  .flatMap(data->{
    // 将数据存储到DB

    // 存储缓存
    Mono<Void> saveDataCahce = SaveCache(data);

    return Mono.when(saveDataDB);
  });

工具操作符

subscribe(Consumer)

订阅序列元素最通用的方式。

我们也可以自己实现Subscriber或者BaseSubscriber来重写它的处理函数.

如下,通过实现BaseSubscriber类,实现主动拉取式消费元素。

Flux.range(0,5).subscribe(new BaseSubscriber<Integer>() {
  //初始化钩子函数 实现此方法以 request(long) 作为初始请求调
  @Override
  protected void hookOnSubscribe(Subscription subscription) {
    System.out.println("init");
    // 请求一个元素,否则无法触发OnNext()方法
    request(1);
  }
  @Override
  protected void hookOnNext(Integer value) {
    // 这里每次消费完元素后,再次请求1个元素,直到消费完毕
    System.out.println("onNext: " + value);
    request(1);
  }
});

在指定的时间内,如果没有产生元素,则抛出异常.
