Go核心篇章-Mutex原理

发表于2023-07-26|更新于2023-08-01|Go
|字数总计:7.1k|阅读时长:24分钟|阅读量:

前言

在Channel原理篇中,我们知道Go推崇使用通信来共享内存，但是Go也提供了传统的共享内存方式,那就是使用Mutex锁机制。

Go内置了以下两种锁:

  • Mutex 互斥锁/独占锁。
  • RWMutex 读写锁/共享锁。

本章主要针对这两种锁的实现原理、概念、源码等进行分享,基础用法就不过多描述。

核心机制

原理

顾名思义,既然是一把锁,那么就会存在两种状态: 已上锁/未上锁。

Mutex通过一个state状态值标识锁的状态。值为0表示未上锁,1表示已上锁。

  • 加锁: 将state由0设置为1;
  • 解锁: 将state由1设置为0;

当执行加锁时,如果state已经为1,则加锁失败,需要等待被解锁。Mutex的加锁与解锁的过程,是通过CAS指令,依靠CPU原语来保证原子性。

阻塞与自旋

在Java中,原生内置的Synchronized同步锁在JKD6以后,引入了锁的升级的概念: 偏向锁 -> 轻量级锁 -> 重量级锁,本质上就是由自旋模式升级为阻塞模式,如果不了解可以看一下这篇文章 Java锁。

Mutex同样也是有这样一个升级过程,当尝试加锁时,发现锁已经被其他人占有,Mutex会有如下两种策略:

  • 阻塞模式

    将当前的协程挂起,置为阻塞态(Gwaiting),直到锁被释放后,以回调的方式再次唤醒该协程(Grunnable),然后进行锁的再次争夺。

  • 自旋模式

    使用CAS+自旋的方式,不断尝试获取锁,直到获取到锁。

这两种策略,各有优劣,并且适用于不同的场景:

锁竞争方案 优势 劣势 适用场景
阻塞/唤醒 精准打击，不浪费 CPU 时间片 需要挂起协程，进行上下文切换，操作较重 并发竞争激烈的场景
自旋+CAS 无需阻塞协程，短期来看操作较轻 长时间争而不得，会浪费 CPU 时间片 并发竞争强度低的场景

Mutex结合这两种策略,会有一个锁升级过程,会根据当前的并发环境决定使用的策略。锁默认刚开始都是自旋模式(乐观的)

  • 首先保持乐观,采用CAS+自旋的策略不断抢夺锁;

  • 当尝试抢锁失败到达一定的条件后,判定当前并发过于激烈,则由自旋模式转为阻塞模式。

触发转为阻塞模式的条件如下:

  1. 自旋4次都未抢到锁;
  2. 系统CPU单核或者仅创建了单个P调度器;(如果只有一个P不停的自旋,其他G压根没有可用的资源来释放锁,所以此情况只能用阻塞模式)
  3. 当前P的执行队列内还有G;(为了避免自旋影响到当前P的调度效率,所以此情况也会转为阻塞模式)

饥饿模式与正常模式

饥饿模式与正常模式你可以理解为是 非公平模式和公平模式。

Mutex运行过程中存在两种模式: 饥饿模式和正常模式。

正常模式(非公平模式)

这是Mutex默认的模式,当Goroutine从阻塞队列被唤醒后,会和此时正要抢夺锁的新的Goroutine开始争夺,如果抢锁失败,则会重新回到阻塞队列头部。

什么是饥饿?

饥饿在这里是指一个Goroutine由于长时间得不到锁,从而陷入饥荒。

假设现在有一把锁被持有着,此时会有两类Goroutine对这把锁蠢蠢欲动:

  • 正处于自旋方式不断尝试获取锁的新的Goroutine;
  • 被阻塞住并处于锁的阻塞队列中等待被唤醒的旧Goroutine;

如果此时锁被释放，那么阻塞队列中刚被唤醒的Goroutine很难抢到锁的执行权，因为它是从Gwaiting状态转为Grunnable状态后再等待被调度，所以它的执行权是远远不如一直在尝试抢锁的新Goroutine，所以可能刚被唤醒又会由于抢锁失败而进入阻塞队列中。为了解决这种情况，Go提供了饥饿模式，也就是所谓的公平模式。

饥饿模式(公平模式)

这是Mutex为了解决饥饿而采用的机制: 饥饿模式下，锁被释放后，锁的持有权会按照阻塞队列的顺序依次传递下去,新的Goroutine不能直接抢占锁,而是会进入阻塞队列尾部等待。

两个模式之间的动态转换条件

  • 正常模式 -> 饥饿模式: 当阻塞队列中有Goroutine等待时间超过1ms,则会进入饥饿模式。
  • 饥饿模式 -> 正常模式: 当阻塞队列为空,或者当前最新抢到锁的Goroutine抢锁时间已经低于1ms时,则回退到正常模式。

总结: 正常模式灵活机动且性能较好，而饥饿模式性能较差但是可以保证资源分配的公平性。这两种模式的切换体现了Mutex为适应环境而变化,在公平与性能之间作出的调整与平衡。

源码分析

数据结构

sync.Mutex源码结构如下:

type Mutex struct {
  state int32 // 锁的状态标识,每个bit位表示不同的状态标识。
  sema  uint32 // 阻塞队列信号量
}

  1. state: 锁中最核心的状态字段。共占32个bit位,不同的bit位分别存储了不同的状态标识。如mutexLocked、mutexWoken、mutexStarving等标识。

  2. sema: 用于阻塞和唤醒goroutine的信号量。

标识状态常量

以下常量分别代表了各个状态在state中的bit位。

const (
  // 值为1。用于标识state右侧第一位bit位为1.表示已上锁。
  // state的第一个bit位,标识了锁是否被上锁: 1-已加锁 0-未加锁
  // 
  // 如果state为0,二进制则为`...0000` 表示未上锁;
  // 如果state为1,二进制则为`...0001` 表示已上锁;
  mutexLocked = 1 << iota // 1

  // 值为2。用于标识state右侧第二位bit位为1,表示有协程正在活跃抢锁。
  // state的第二个bit位,标识了当前锁是否有协程正在抢锁。1-是 0-否
  // 
  // 如果state为2,二进制则为...0010,表示当前有协程在抢锁;
  // 如果state为3,二进制则为...0011,表示当前有协程在抢锁,但是锁已经被持有;
  // 1-是 0-否
  mutexWoken // 2

  // 值为4，用于标识state右侧第三位bit位为1,表示处于饥饿模式。
  // state的第三个bit位,标识了当前锁是否处于饥饿模式。1-饥饿模式 0-正常模式
  // 
  // 如果state为4,二进制则为...0100,表示当前锁处于饥饿模式。
  // 如果state为5,二进制则为...0101,表示当前如果state为5,二进制则为...0101,表示当前锁处于饥饿模式,并且锁被占有着。锁处于饥饿模式,并且锁被占有着。
  mutexStarving // 4

  // 该常量不具有意义,仅仅表示state最右侧的3个bit位代表状态标识
  mutexWaiterShift = iota // 3

  // 正常模式转换为饥饿模式的 等待时间条件阈值
  starvationThresholdNs = 1e6
)
  1. mutexLocked: 值为1,表示已经上锁。
  2. mutexWoken: 值为2,表示当前有协程正在活跃抢锁。
  3. mutexStarving: 值为4,表示当前处于饥饿模式。
  4. mutexWaiterShift: 值为3,不具有具体含义,用于通过位运算获取state右侧状态位或左侧数据位。
  5. starvationThresholdNs: 正常模式转换为饥饿模式的 等待时间条件阈值,1ms。

state的bit位描述

0|0|0|0
阻塞队列的G的数量|是否饥饿模式|是否有G在抢锁|是否上锁
阻塞队列的G的数量|mutexStarving|mutexWoken|mutexLocked

state的值可能产生的情况举例(后三位):

值 = 二进制位 = 表示含义
0 = 000 = 表示未上锁。
1 = 001 = 已上锁。
2 = 010 = 正在尝试抢锁。
3 = 011 = 有协程正在尝试抢锁,但是锁已经被持有了。
4 = 100 = 当前锁处于饥饿模式
5 = 101 = 当前处于饥饿模式,并且锁已经被持有了。
6 = 110 = 当前处于饥饿模式,并且有G正在尝试抢锁
7 = 111 = 当前处于饥饿模式,有协程正在尝试抢锁,但是锁已经被持有了。

state后29位用于存储阻塞队列中的协程数量
每增加一个协程,则state需要+8(1<<3),以此类推。每唤醒一个协程,则state需要-8(1<<3)。

8 = ...1|000 = 表示有1个协程在等待。
16 = ..10|000 = 表示有0个协程在等待。

结构图解

低 3 位分别标识 mutexLocked（是否上锁）、mutexWoken（是否有协程在抢锁）、mutexStarving（是否处于饥饿模式），高 29 位的值聚合为一个范围为 0~2^29-1 的整数，表示在阻塞队列中等待的协程个数.

在源码中,Go使用了大量的位运算来计算和修改锁的状态,如下几种运算:

  • state & mutexLocked: 判断是否已经上锁;
  • state | mutexLocked: 加锁;
  • state & mutexWoken: 判断是否有协程正在抢锁;
  • state | mutexWoken：更新状态，标识存在抢锁的协程;
  • state &^ mutexWoken：更新状态，标识不存在抢锁的协程;
  • state & mutexStarving：判断是否处于饥饿模式；
  • state | mutexStarving：置为饥饿模式；
  • state >> mutexWaiterShif：获取阻塞等待的协程数；
  • state += 1 << mutexWaiterShif(3)：阻塞等待的协程数 + 1.
  • state -= 1 << mutexWaiterShif(3)：阻塞等待的协程 - 1.

Mutex

Lock()

Mutex加锁的流程如下:

  1. 首先通过CAS尝试一次加锁,如果加锁成功则直接返回。
  2. 如果加锁失败,则步入lockSlow()方法,该方法通过自旋和阻塞两种策略核心实现。
    1. 通过CAS+自旋的方式,尝试加锁4次(当前必须为非饥饿模式,否则会跳过这一步)。
    2. 如果4次自旋加锁都未成功。
    3. 自旋策略结束后会继续进行CAS加锁,如果还是失败,则被陷入阻塞,添加到阻塞队列中，等待被唤醒。
    4. 被唤醒之后。
      1. 计算被阻塞了多少时间,从而决定锁是否需要转换为饥饿模式或者退回正常模式。
      2. 如果是饥饿模式;
        1. 根据阻塞时间,判断是否转换为正常模式。
        2. 加锁成功。
      3. 如果是正常模式,则会继续尝试抢锁。

源码解读流程如下:

// Mutex 阻塞式加锁方法,如果锁正在被占用,则"阻塞"等待锁释放,直到获取到锁
func (m *Mutex) Lock() {
  // 1.尝试直接加锁
  // 通过CAS,尝试将state从0设置为1(mutexLocked) = ...001
  if atomic.CompareAndSwapInt32(&m.state, 0, mutexLocked) {
    if race.Enabled {
      race.Acquire(unsafe.Pointer(m))
    }
    // 只有state为初始化状态0 = (...0|0|0|0),所有bit位均为0才会成立
    // (阻塞队列等待数量 && 非饥饿模式 && 没有协程在抢锁 && 锁没有被持有) 只有此情况才会步入到这里,直接加锁成功并且返回。
    // 加锁成功,直接返回
    return
  }
  // 第一次加锁失败,使用自旋策略和阻塞策略获取锁
  m.lockSlow()
}


func (m *Mutex) lockSlow() {
  // 2. 通过自旋或者阻塞的方式进行加锁,直到获取锁。

  // 声明一个int64类型,用于记录当前G如果需要阻塞,那么阻塞的开始时间戳(纳秒)
  // 只有当前G要被阻塞并且挂起时初始化值
  // 用来计算当前G被阻塞了多长时间,然后决定锁的模式之间的转换
  // 并且使用该时间判断是否是被唤醒的G,如果是被唤醒后又要阻塞,则回退到阻塞队列头部
  var waitStartTime int64

  // 是否需要转换为饥饿模式，默认为false，当前G被阻塞过后又被唤醒后才会用到该属性。
  // 被唤醒后会根据waitStartTime计算一共阻塞了多长时间,如果大于1ms,此值会被值为true。
  starving := false

  // 当前是否有G在取锁,默认为false。
  awoke := false

  // 当前G自旋获取锁失败的次数
  iter := 0

  // 开始自旋或者阻塞之前,获取一次锁的状态,作为预期值.
  old := m.state
  // 循环不断加锁,直到获取到锁.
  for {
    // 2.1 自旋式加锁(乐观)处理
    // 判断是否进行自旋加锁，只有((锁已经被持有 && 不处于饥饿模式) && 还有可自旋次数)成立才会进行自旋
    if old&(mutexLocked|mutexStarving) == mutexLocked && runtime_canSpin(iter) {
      // 步入这里,表示锁已经被持有,并且不处于饥饿模式,但是还有可自旋次数

      // 如果当前没有G在抢锁,并且锁的阻塞队列中有G在等待唤醒。
      // 此时为了优化效率,将mutexWoken设置为1,表示有协程正在抢锁,就不要从队列中唤醒G了,直接让正在抢锁的新G获取。
      // 避免再有其他协程和自己抢锁。
      if !awoke && old&mutexWoken == 0 && old>>mutexWaiterShift != 0 && atomic.CompareAndSwapInt32(&m.state, old, old|mutexWoken) {
        // 将mutexWoken标识位设置为1
        // 表示当前存在协程正在获取锁
        awoke = true
      }
      // 进行自旋
      runtime_doSpin()
      // 自旋次数+1
      iter++
      // 再次获取锁的信息
      old = m.state
      // 继续下一次自旋获取锁
      continue
    }
    // 2.2 如果不满足自旋条件,或者自旋次数用尽后处理 (处于饥饿模式 || 没有可自选次数)

    // 再次获取一次加锁之前的state状态信息,作为加锁成功后的状态信息,如果失败此数据则会被无效丢弃。
    new := old
    // 判断锁是否已经是正常模式,如果是则直接加锁
    if old&mutexStarving == 0 {
      // 加锁,注意这里并非百分百加锁成功,需要后续的校验
      new |= mutexLocked
    }

    // 如果锁处于已上锁状态,或者当前锁是饥饿模式,那么就进入阻塞队列
    if old&(mutexLocked|mutexStarving) != 0 {
      // 阻塞等待的协程数量+1
      new += 1 << mutexWaiterShift
    }
    // 判断当前锁是否需要设置为饥饿模式,如果需要则设置。
    // 当一个G被唤醒之后,重新抢锁才会执行到这里.
    if starving && old&mutexLocked != 0 {
      // 设置为饥饿模式
      new |= mutexStarving
    }
    // 当前协程抢锁失败,即将要被挂起,所以需要将mutexWoken重置为0,表示没有协程在抢锁。
    if awoke {
      // The goroutine has been woken from sleep,
      // so we need to reset the flag in either case.
      if new&mutexWoken == 0 {
        throw("sync: inconsistent mutex state")
      }
      // 当前G被唤醒,无论是否抢到锁,我们都需要将mutexWoken恢复成0,表示当前没有协程在抢锁
      // 当任何数与这个进行&^操作时，都会被置为0,如果mutexWoken为1，那么就会被清空为0，如果mutexWoken为0,那么与左侧保持一致，为0
      new &^= mutexWoken
    }
    // 2.3
    // 通过CAS,将加锁产生的新锁信息，更新到state中
    // 如果更新失败,则加锁失败
    // 注意,如果更新成功,并不一定是加锁成功,可能是在这过程中,并没有其他G进来抢锁,
    // 具体是否加锁成功,需要进一步判断
    if atomic.CompareAndSwapInt32(&m.state, old, new) {

      // 判断是否加锁成功
      // 如果锁状态标识为0(未加锁) 并且是正常模式则加锁成功。
      if old&(mutexLocked|mutexStarving) == 0 {
        // 加锁成功,退出循环
        break // locked the mutex with CAS
      }

      // 判断是否是从队列头部所唤醒的G,如果是G则放入阻塞队列的头部
      queueLifo := waitStartTime != 0
      // 判断当前G开始阻塞时间戳是否为0
      // 如果是0,则表示第一次循环
      if waitStartTime == 0 {
        // 获取当前时间纳秒,作为开始阻塞挂起的时间
        waitStartTime = runtime_nanotime()
      }
      // 将当前G添加到阻塞队列,当前G会一直阻塞到这一步,直到被唤醒...
      runtime_SemacquireMutex(&m.sema, queueLifo, 1)

      // 4.从阻塞队列被唤醒

      // 4.1 判断模式转换,计算当前G被阻塞了多长时间(当前时间戳 - 开始阻塞时的时间戳 > 1ms)
      //	   如果时长大于1ms就转换为饥饿模式,反之如果小于1ms,则转换为正常模式
      starving = starving || runtime_nanotime()-waitStartTime > starvationThresholdNs
      old = m.state

      // 4.2 被唤醒后,如果锁之前就是饥饿模式(公平模式),表示当前G会直接获取锁的持有权。
      if old&mutexStarving != 0 {
        // 当前G被从阻塞队列中唤醒,已经是饥饿模式,那也就意味着本次肯定会抢到锁
        if old&(mutexLocked|mutexWoken) != 0 || old>>mutexWaiterShift == 0 {
          throw("sync: inconsistent mutex state")
        }
        // 创建一个新的锁的状态信息
        // 并且将锁标识位设置为1,并且阻塞队列的G数量-1。
        delta := int32(mutexLocked - 1<<mutexWaiterShift)

        // 如果当前阻塞队列已经为空,或者阻塞时间小于1ms,则转换为正常模式
        if !starving || old>>mutexWaiterShift == 1 {
          // 退出饥饿模式,转换为正常模式
          delta -= mutexStarving
        }
        // 直接加锁,通过原子相加的方式,更新state状态
        atomic.AddInt32(&m.state, delta)
        // 加锁成功 退出
        break
      }

      // 4.3 从队列中被唤醒,正常模式下,继续尝试抢锁、自旋、阻塞这个流程.
      awoke = true // 标识有协程正在抢锁
      iter = 0     // 重置自旋次数
    } else {
      // 如果CAS失败,标识没有成功获取到锁,锁可能被别的Goroutine获得，或者一直没有被释放
      // 那么就刷新状态,重新开始执行加锁流程。
      old = m.state
    }
  }
  if race.Enabled {
    race.Acquire(unsafe.Pointer(m))
  }
}

TryLock()

非阻塞式加锁,会直接通过CAS进行一次加锁,成功或失败都会直接返回,该方法较为简单,流程如下:

  1. 如果当前锁已被持有,或者处于饥饿模式,直接返回false;
  2. 通过CAS,将锁的状态标识由0改为1,如果失败直接返回false;
  3. 加锁成功;

源码如下:

func (m *Mutex) TryLock() bool {
  // 获取锁的状态信息
  old := m.state
  // 1. 如果当前锁已被持有,或者处于饥饿模式,直接返回false
  if old&(mutexLocked|mutexStarving) != 0 {
    return false
  }
  // 2. 通过CAS,将锁的状态标识由0改为1,如果失败直接返回false
  if !atomic.CompareAndSwapInt32(&m.state, old, old|mutexLocked) {
    return false
  }
  if race.Enabled {
    race.Acquire(unsafe.Pointer(m))
  }
  // 获取锁
  return true
}

Unlock()

Mutex解锁流程:

  1. 通过原子操作,直接进行解锁。
  2. 如果解锁后的state为0(0000),则表示无需唤醒操作,直接解锁成功。
  3. 如果state值不为0,则需要步入到unlockSlow()方法,唤醒阻塞的协程;
    1. 校验锁是否是上锁状态,如果未上锁就解锁会产生panic。
    2. 正常模式解锁。
      1. 判断是否需要从阻塞队列中唤醒等待的协程，如果不需要直接退出，满足如下条件则可以直接退出:
        • 阻塞队列为空。没有阻塞等待的协程,当然不需要唤醒。
        • 锁被释放之后又被其他协程占有。已经被持有了,所以也没必要唤醒阻塞的协程了。
        • 当前有正在自旋尝试获取锁的协程。已经有协程再尝试获取锁,也没必要唤醒阻塞的协程,避免没有意义的竞争。
      2. 循环尝试唤醒阻塞队列头部的等待协程，直到唤醒一个协程后退出。
    3. 饥饿模式解锁
      1. 如果是饥饿模式,直接从阻塞队列头部唤醒协程即可。

流程图:

源码如下:

// Mutex 解锁方法
func (m *Mutex) Unlock() {
  if race.Enabled {
    _ = m.state
    race.Release(unsafe.Pointer(m))
  }
  // 1.直接通过原子操作,尝试解锁
  //   将state值-1,并且获取值。
  new := atomic.AddInt32(&m.state, -mutexLocked)

  // 3. 判断操作后的值是否为0,如果为0,则表示解锁成功,并且没有可唤醒的协程,直接退出。
  // 只有state为 1 == 0001 == (阻塞队列为空 && 非饥饿模式 && 锁已被抢占)才会直接解锁成功
  if new != 0 {
    // 如果值不为0,则需要尝试唤醒阻塞队列中的协程。
    m.unlockSlow(new)
  }
  // 2. 直接退出
}

func (m *Mutex) unlockSlow(new int32) {
  // 3.1 校验锁是否是出于加锁状态,如果未加锁情况下进行释放锁,则panic
  if (new+mutexLocked)&mutexLocked == 0 {
    fatal("sync: unlock of unlocked mutex")
  }
  // 根据正常模式和饥饿模式进行不同解锁处理

  // 3.2 如果是正常模式
  if new&mutexStarving == 0 {
    old := new
    // 循环尝试唤醒阻塞队列头部的协程,直到唤醒后结束
    for {
      // 3.2.2 判断解锁完成后是否需要从阻塞队列中唤醒阻塞的协程,当满足如下条件之一时,则不需要唤醒
      // (1) 阻塞队列为空。没有阻塞等待的协程,当然不需要唤醒。
      // (2) 锁被释放之后又被其他协程占有。已经被持有了,所以也没必要唤醒阻塞的协程了。
      // (3) 当前有正在自旋尝试获取锁的协程。已经有协程再尝试获取锁,也没必要唤醒阻塞的协程,避免没有意义的竞争。
      if old>>mutexWaiterShift == 0 || old&(mutexLocked|mutexWoken|mutexStarving) != 0 {
        // 无需唤醒,直接结束.
        return
      }
      // 需要唤醒
      // 将阻塞队列中的数量-1,并且将锁标识为当前有协程正在抢夺锁.
      new = (old - 1<<mutexWaiterShift) | mutexWoken
      // 更新锁的状态
      if atomic.CompareAndSwapInt32(&m.state, old, new) {
        // 更新成功,从队列头部唤醒等待的协程
        runtime_Semrelease(&m.sema, false, 1)
        return
      }
      // 更新失败,由其他协程介入,更新一下锁的信息,继续尝试唤醒
      old = m.state
    }
  } else {
    // 3.3 饥饿模式
    // 如果是饥饿模式,直接从阻塞队列头部唤醒协程即可。
    runtime_Semrelease(&m.sema, true, 1)
  }
}

RWMutex

sync.RWMutex 读写锁,本身也是基于sync.Mutex来实现的。

读写锁的核心机制:

  • 写锁具有严格的排他性，当被占有时，其他尝试获取写锁或读锁的协程均会被阻塞。
  • 读锁具有有限的共享性,当被占有时，尝试获取写锁的协程会被阻塞,而尝试获取读锁的协程不会阻塞而会共享。

在一些读多写少的场景下,使用读写吞吐量会更高，倘若所有的操作都是写操作,那么RWMutex则会退化为Mutex。

核心结构

// 可同时获取读锁的最大协程数量 值为1073741824
const rwmutexMaxReaders = 1 << 30

type RWMutex struct {
  // Mutex锁,主要保证写锁与写锁之间的互斥
  w Mutex
  // 用于阻塞写操作的协程的阻塞队列,只会阻塞一个协程。多余的写协程会阻塞在w锁中的阻塞队列里。
  writerSem uint32
  // 用于阻塞读操作的协程的阻塞队列
  readerSem uint32
  
  // 标识当前有多少协程获取了读锁
  // 当有协程获取了一把读锁,则该值+=1,反之有协程释放了读锁,该值则-=1。
  // 获取写锁后,该值会 -= rwmutexMaxReaders。
  readerCount int32
  
  // 记录当前协程获取写锁之前,还需要等待多少个读锁释放
  readerWait int32
}

RWMutex中的readerCount是一个非常核心的属性,主要用该属性标识当前是否可以加读锁或者写锁:

readerCount < 0,则表示当前写锁被占有,无法获取读锁或者再次获取写锁。

readerCount > 0，则表示当前可以加读锁，但是不可以加写锁。

readerCount = 0，则表示当前可以获取写锁也可以获取读锁。

RLock()

获取读锁流程:

  1. 通过原子操作，将readerCount递增1，表示当前协程取走一把读锁。
  2. 此时如果readerCount小于0，则表示当前存在协程获取着写锁，需要阻塞当前协程放入读阻塞队列,等待写锁释放。
  3. 此时如果readerCount大于0,则表示获取读锁成功，直接返回。

源码如下:

// 获取读锁流程
func (rw *RWMutex) RLock() {
  // 1. 通过原子操作,将readerCount值递增1,表示当前来了一个读操作协程
  if atomic.AddInt32(&rw.readerCount, 1) < 0 {
    // 2. 如果递增后的值依然小于0,则表示当前存在协程获取了写锁,则需要挂起当前协程
    // 将当前协程阻塞,放入读阻塞队列
    runtime_SemacquireMutex(&rw.readerSem, false, 0)
  }
  // 3. 如果readerCount大于0,则表示获取读锁成功，直接返回。
}

TryRLock()

非阻塞式获取读锁流程:

  1. 直接判断readerCount是否小于0,如果小于,则表示目前写锁被持有,获取读锁失败。
  2. 如果大于0,则将readerCount递增1,获取读锁成功。

源码如下:

// 非阻塞式获取读锁
func (rw *RWMutex) TryRLock() bool {
  if race.Enabled {
    _ = rw.w.state
    race.Disable()
  }
  for {
    // 1.获取当前readerCount,如果readerCount小于0,则表示写锁被持有,无法获取读锁
    c := atomic.LoadInt32(&rw.readerCount)
    if c < 0 {
      if race.Enabled {
        race.Enable()
      }
      return false
    }
    //2. readerCount大于0，表示可以获取读锁,将readerCount += 1,获取读锁成功
    if atomic.CompareAndSwapInt32(&rw.readerCount, c, c+1) {
      if race.Enabled {
        race.Enable()
        race.Acquire(unsafe.Pointer(&rw.readerSem))
      }
      return true
    }
  }
}

RUnlock()

释放读锁流程:

  1. 通过原子操作,将readerCount递减1,表示释放了一把读锁。
  2. 判断是否存在阻塞等待的写协程需要被唤醒。
  3. 如果此时readerCount小于0,则表示存在。
    1. 校验是否有可以释放的读锁,如果没有则panic。
    2. 判断当前释放的是否是最后一把读锁,如果是则唤醒阻塞等待的写协程。
  4. 不存在则直接解锁完成。

源码如下:

// 释放读锁流程
func (rw *RWMutex) RUnlock() {
  // 1. 通过原子操作,将readerCount递减1,表示释放了一把读锁
  if r := atomic.AddInt32(&rw.readerCount, -1); r < 0 {
    // 2. 判断是否存在阻塞等待的写协程
    // 如果此时readerCount小于0,则表示存在
    rw.rUnlockSlow(r)
  }
}

// 释放读锁,并且可能需要唤醒写协程
func (rw *RWMutex) rUnlockSlow(r int32) {
  // 校验当前是否可以释放读锁
  // 如果在释放读锁之前,readerCount为0，则表示没有读锁可以释放,直接panic
  if r+1 == 0 || r+1 == -rwmutexMaxReaders {
    race.Enable()
    fatal("sync: RUnlock of unlocked RWMutex")
  }
  // 判断当前释放的是否是最后一把读锁,如果是则唤醒阻塞等待的写协程
  if atomic.AddInt32(&rw.readerWait, -1) == 0 {
    // 从写阻塞队列头部唤醒等待的协程
    runtime_Semrelease(&rw.writerSem, false, 1)
  }
}

Lock()

获取写锁流程:

  1. 通过Mutex直接加锁,让所有获取写锁的协程串行化.
    1. 这里加锁只是保证写与写之间的互斥,并非获取到了读写锁中的写锁。
  2. 将readerCount - rwmutexMaxReaders,表示获取写锁。
  3. 判断当前是否有读锁被持有。
    1. 判断readerCount减去rwmutexMaxReaders之前是否!=0,表示目前还有读锁被持有,将会阻塞当前协程,放入写阻塞队列中,等待读锁的唤醒。
    2. 如果是0,则表示可以获取写锁,加锁成功。

源码如下:

// 获取写锁
func (rw *RWMutex) Lock() {
  // 1. 通过Mutex直接加锁,让所有获取写锁的协程串行化.
  // 这里也体现了,如果全部是写操作,会直接退化为一把Mutex互斥锁.
  rw.w.Lock()

  // 2. 根据地址将当前锁的readerCount值-1073741824,标识获取写锁
  //    并且获取readerCount-1073741824操作之前的值,也就是获取目前被持有了多少把读锁
  r := atomic.AddInt32(&rw.readerCount, -rwmutexMaxReaders) + rwmutexMaxReaders

  // 3.1 如果r != 0,则表示当前是有读锁被持有,无法获取写锁,阻塞当前协程，放入到写的阻塞队列中,等待读锁唤醒
  if r != 0 && atomic.AddInt32(&rw.readerWait, r) != 0 {
    runtime_SemacquireMutex(&rw.writerSem, false, 0)
  }
  // 3.2 如果r==0,则表示当前没有读锁被持有,直接获取写锁成功
}

Unlock()

解写锁流程:

  1. 将readerCount+=readerCount,表示没有写锁被持有了。
  2. 唤醒所有的读等待协程。
  3. 释放Mutex互斥锁,让其他写操作也可以执行。

源码如下:

func (rw *RWMutex) Unlock() {
	// 1. 将readerCount+=readerCount,表示没有写锁被持有了
	r := atomic.AddInt32(&rw.readerCount, rwmutexMaxReaders)

	//  判断是否已经获取了写锁,如果没有获取过则panic
  //  如果读协程已经达到上限,也会产生panic
	if r >= rwmutexMaxReaders {
		race.Enable()
		fatal("sync: Unlock of unlocked RWMutex")
	}
	// 2. 唤醒所有的读等待协程
	for i := 0; i < int(r); i++ {
		runtime_Semrelease(&rw.readerSem, false, 0)
	}
	// 3. 释放Mutex互斥锁,让其他写操作也可以执行
	rw.w.Unlock()
	if race.Enabled {
		race.Enable()
	}
}
