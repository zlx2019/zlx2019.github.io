Git的使用

发表于2023-01-10|更新于2023-03-22|Git
|字数总计:10.8k|阅读时长:39分钟|阅读量:

前言

一直没有对Git做深入的学习,平常有问题了也是直接百度,以至于每次都会创建个仓库等简单琐事耽误大量时间,所以记录下这次学习笔记。

Git是什么

Git是目前世界上最流行最先进的分布式版本控制系统。

Git的诞生

Git的作者就是大名鼎鼎的Linus,没错就是Linux系统的开源者。

Linus在1991年创建了Linux,从此Linux系统迅速发展,但是Linux不只是Linus自己一个人不断完善,而是世界各地的开源参与者不断贡献代码。

在2002年以前，世界各地的志愿者把源代码文件通过diff的方式发给Linus，然后由Linus本人通过手工方式合并代码.额…看到这里你可能会难以想象,这是多么大的工作量。

但是尽管如此Linus也不想使用当时的一些版本控制系统,如 CVS、SVN等。觉得这些系统效率太差,而且收费，不符合开源精神。随着代码量的不断增加,Linus已经很难通过手工方式进行管理,然后Linus花费了两周写了一个分布式版本管理系统,这就是Git!

基本使用流程

1. 创建版本库

版本库又被称为仓库(Repository),用来存储我们所要管理的多个文件。你可以理解为一个目录,这个目录里的所有文件都可以被Git管理起来,每个文件的修改、删除，Git都能跟踪，以便任何时刻都可以追踪历史，或者在将来某个时刻可以“还原”。

如何创建呢?很简单,找到我们想作为仓库的目录。

mkdir repo01
cd repo01

通过git init命令将当前目录变成Git可管理的仓库。

% git init
Initialized empty Git repository in /usr/local/myproject/git-project/repo01/.git/

这样就把仓库创建好了,提示你目前还是一个空的仓库empty Git repository。

然后我们来看下目录的变化:

% ls -a
.	..	.git

注意: 你会发现多了一个.git隐藏目录,这个是用来跟踪Git版本库的,千万不要乱动该目录的文件,很容易破坏掉仓库。

2. 添加到暂存区

当我们对仓库内的文件做了调整之后,需要把它们添加到暂存区中,如下创建一个新的文件a.txt

touch a.txt

并且写入以下内容:

Hello Git!

然后就可以通过git add a.txt命令将a.txt文件添加到暂存区

git add a.txt

执行上面的命令，没有任何显示，这就对了，Unix的哲学是“没有消息就是好消息”，说明添加成功。

3. 提交到版本库

当我们把调整后的文件添加到暂存区并且确认没问题后,就可以通过git commit命令进一步提交到版本库。

% git commit -m 'add first file'

[main (root-commit) a1c80ed] add first file
 1 file changed, 1 insertion(+)
 create mode 100644 a.txt

-m表示后面的文本是本次提交的说明,可以任意输入内容,注意:这很重要,通常在公司协同开发时,提交的说明有很多规范,对自己对同事阅读都很重要。

1 file changed: 有一个文件被改动,也就是我们新添加了一个a.txt文件。

1 insertion(+): 插入了一行内容,也就是Hello Git!。

为什么Git添加文件需要add，commit一共两步呢？因为commit可以一次提交很多文件，所以你可以多次add不同的文件，比如：

git add 101.txt
git add 102.txt 103.txt

git commit -m 'add 3 files.'

不过我们后续会使用一些开发工具,可以直接一步直接git commit掉。

通过git add .命令可以一键将当前目录下所修改过的文件全部添加到暂存区。

仓库状态查看

通过git status命令可以查看当前仓库的状态:

% git status  

On branch main
nothing to commit, working tree clean

这表示没有任何文件进行修改。

现在我们对一个文件进行修改,然后再次查看:

% git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   test01/101.txt

no changes added to commit (use "git add" and/or "git commit -a")

modified: test01/101.txt指出了我们所修改但是还未提交到暂存区的文件。

虽然Git告诉我们101.txt被修改了，但如果能看看具体修改了什么内容，自然是很好的。比如你休假两周从国外回来，，已经记不清上次怎么修改的101.txt，所以，需要用git diff这个命令看看：

zero@Zero repo01 % git diff
diff --git a/test01/101.txt b/test01/101.txt
index 72943a1..92b5f31 100644
--- a/test01/101.txt
+++ b/test01/101.txt
@@ -1 +1,2 @@
 aaa
+添加一句话!
\ No newline at end of file

可以看到,这个文件内的改动+添加一句话!。

然后我们将修改后的文件提交到暂存区

git add 101.txt

然后再次查看状态

% git status     
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	modified:   101.txt

表示暂存区里可以提交的文件有101.txt。接下来我们就可以git commit了

% git commit -m 'test add'
[main e2fef7c] test add
 1 file changed, 1 insertion(+)

提交成功后,我们在使用git status查看一下：

% git status
On branch main
nothing to commit, working tree clean

Git告诉我们当前没有需要提交的修改，而且，工作目录是干净（working tree clean）的。

小结：

  • 通过git status命令 查看仓库状态
  • 通过git diff命令查看被修改过文件的改动内容

时间回溯

1. 版本回退

到目前为止,我们已经学会如何创建仓库、如何将文件放入暂存区,然后再提交到版本库。每一次的commit你可以理解为一个版本或者快照,这些版本Git都会进行记录和跟踪,一旦你哪一步把文件改乱了,或者误删了什么东西,还可以直接回退到一个commmit的版本。那么该如何操作?

现在我们再次创建一个仓库,如何创建就不过多赘述,可以翻看之前的教程.然后在仓库内创建一个文件:

touch README.md

模拟第一次工作提交:

# 将文本追加到README.md 文件
echo "2023年01月01日第一次提交" >> README.md

# 添加到暂存区
% git add README.md 
# 确认提交大版本库
% git commit -m '第一次提交'
[main (root-commit) 6d02308] 第一次提交
 1 file changed, 1 insertion(+)
 create mode 100644 README.md

模拟第二次工作提交:

% echo "2023年01月05日第二次提交" >> README.md 
% git add README.md 
% git commit -m '第二次提交'
[main 1c2c2fa] 第二次提交
 1 file changed, 1 insertion(+)

模拟第三次提交:

% echo "2023年01月08日第三次提交" >> README.md
% git add README.md                           
% git commit -m '第三次提交'                  
[main 1f3fc92] 第三次提交
 1 file changed, 1 insertion(+)

通过git log命令可以查看仓库内的commit命令提交历史记录。

% git log

commit 1f3fc924d51a3a4d0e314f7e826c38f4a8ecf887 (HEAD -> main)
Author: Zero <15133024857@163.com>
Date:   Wed Jan 11 14:50:07 2023 +0800

    第三次提交

commit 1c2c2fac60534cc44cdae3ed5016b80bc2434941
Author: Zero <15133024857@163.com>
Date:   Wed Jan 11 14:49:03 2023 +0800

    第二次提交

commit 6d02308ab464740274741fd9cb165fe7c44442d6
Author: Zero <15133024857@163.com>
Date:   Wed Jan 11 14:47:15 2023 +0800

    第一次提交

git log命令显示提交时间由近到远的记录。我们可以看到一共有三次commit,最近的一次也就是最上面14:50:07提交的那次.

信息内容为:commit记录的id,然后是提交者的名字其次是提交时间。

如果觉得内容显示的太多有些眼花缭乱,那么可以加上--pretty=oneline参数进行查看:

% git log --pretty=oneline

1f3fc924d51a3a4d0e314f7e826c38f4a8ecf887 (HEAD -> main) 第三次提交
1c2c2fac60534cc44cdae3ed5016b80bc2434941 第二次提交
6d02308ab464740274741fd9cb165fe7c44442d6 第一次提交

这样只会显示commitid和提交时的-m备注参数。

在Git中HEAD表示当前版本,也就是最近一次的提交版本,上一个版本可以用HEAD^表示,上上个版本则为HEAD^^,显然一个^符表示向上翻一个版本,当然如果版本过多的话,也可以简写,比如前100个版本: HEAD~100。

那么先的版本的id尾号为f887 并且内容为:

% cat README.md 

2023年01月01日第一次提交
2023年01月05日第二次提交
2023年01月08日第三次提交

接下来我们让版本回退到上一个版本,也就是id尾号为4941的第二次提交版本:

通过git reset --hard 回退的版本命令

% git reset --hard HEAD^
HEAD is now at 1c2c2fa 第二次提交

回退完成,现在让我们再次查看下提交历史记录和文件内容:

% git log --pretty=oneline

1c2c2fac60534cc44cdae3ed5016b80bc2434941 (HEAD -> main) 第二次提交
6d02308ab464740274741fd9cb165fe7c44442d6 第一次提交

% cat README.md 
2023年01月01日第一次提交
2023年01月05日第二次提交

可以看到第三次提交的记录和提交的内容已经全部不见,完全回退到了第二次提交记录时的阶段。

但是…如果想再次回到第三次提交可以不可以呢？其实还是可以的只要你记录着第三次提交的commitid就可以：

版本号没必要写全，前几位就可以了，Git会自动去找。当然也不能只写前一两位，因为Git可能会找到多个版本号，就无法确定是哪一个了。

% git reset --hard 1f3f
HEAD is now at 1f3fc92 第三次提交

然后再次查看历史记录和文件内容:

% git log --pretty=oneline
1f3fc924d51a3a4d0e314f7e826c38f4a8ecf887 (HEAD -> main) 第三次提交
1c2c2fac60534cc44cdae3ed5016b80bc2434941 第二次提交
6d02308ab464740274741fd9cb165fe7c44442d6 第一次提交

% cat README.md 
2023年01月01日第一次提交
2023年01月05日第二次提交
2023年01月08日第三次提交

果然,又滚回来了…在版本之间来回横跳,有这样强大的回退功能,再也不需要担心代码出现问题。第一时间回滚到上一次没有问题的版本即可。

假如现在回退到了第一个版本:

% git reset --hard HEAD^^
HEAD is now at 6d02308 第一次提交

然后我们几天后又想回退到第三个版本,但是已经找不到commit-id了那该如何?

在Git中，总是有后悔药可以吃的。Git提供了一个命令git reflog用来记录你的每一次命令：

% git reflog

6d02308 (HEAD -> main) HEAD@{0}: reset: moving to HEAD^^
1f3fc92 HEAD@{1}: reset: moving to 1f3f
1c2c2fa HEAD@{2}: reset: moving to HEAD^
1f3fc92 HEAD@{3}: commit: 第三次提交
1c2c2fa HEAD@{4}: commit: 第二次提交
6d02308 (HEAD -> main) HEAD@{5}: commit (initial): 第一次提交

该命令记录了你所有的commit和reset命令记录。可以从这里找到第三次commit的id为1f3fc92,然后就可以再次回退到第三个提交版本:

% git reset --hard 1f3fc92
HEAD is now at 1f3fc92 第三次提交

% git log --pretty=oneline
1f3fc924d51a3a4d0e314f7e826c38f4a8ecf887 (HEAD -> main) 第三次提交
1c2c2fac60534cc44cdae3ed5016b80bc2434941 第二次提交
6d02308ab464740274741fd9cb165fe7c44442d6 第一次提交

% cat README.md 
2023年01月01日第一次提交
2023年01月05日第二次提交
2023年01月08日第三次提交

小结:

  • HEAD指向的版本就是当前版本，因此，Git允许我们在版本的历史之间穿梭，使用命令git reset --hard commit_id。
  • 穿梭前，用git log可以查看提交历史，以便确定要回退到哪个版本。
  • 要重返未来，用git reflog查看命令历史，以便确定要回到未来的哪个版本。

2. 工作区撤销

假如我们对一个文件进行了操作,如下:

% echo "修改了文件内容...." >> README.md 

% cat README.md 
2023年01月01日第一次提交
2023年01月05日第二次提交
2023年01月08日第三次提交
修改了文件内容....

但是我们并没有通过add命令添加到暂存区中,现在想恢复操作之前该如何?

当然,如果是一些简单的改动,你自然可以手动恢复,但是如果你做了大量的改动,导致你自己都无法恢复,这个时候可以通过Git进行撤销操作。

通过git checkout --要撤销的文件命令,将该文件在工作区内的所有改动进行撤销。

# 进行撤销
% git checkout -- README.md 

# 查看仓库状态
% git status
On branch main
nothing to commit, working tree clean
# 查看文件内容
% cat README.md 
2023年01月01日第一次提交
2023年01月05日第二次提交
2023年01月08日第三次提交

或者通过git checkout -- .撤销所有文件在工作空间的改动。

新版本的Git也可以使用如下命令完成同样的操作:

# 撤销文件工作区内的改动
git restore file

# 撤销所有文件工作区内的改动
git restore .

3. 暂存区撤销

如果我们对文件进行了修改,而且还添加到了暂存区:

% echo '测试撤销暂存区' >> README.md 

% git add README.md 

% cat README.md 
2023年01月01日第一次提交
2023年01月05日第二次提交
2023年01月08日第三次提交
测试撤销暂存区

想要丢弃修改的话需要如下操作:

  1. 使用命令撤销暂存区:git reset HEAD README.md
  2. 然后使用撤销工作区的命令: git checkout -- README.md

这样就完成了暂存区的撤销。

或者使用另一种方法,通过git restore

  1. 撤销暂存区的修改: git restore --staged README.md
  2. 撤销工作区的修改: git restore READMO.md

restore命令的解释

git restore --worktree readme.txt

表示把当前工作区恢复到上一次添加到暂存区时的样子。

通俗点来讲就是: 将本文件上一次add添加暂存区之后的所有修改全部撤销。


git restore --staged readme.txt

表示把当前暂存区恢复到当前分支的版本。

通俗点来讲就是: 将本文件上一次commit之后所有的add到暂存区的内容全部撤销。


git restore --source=HEAD --staged --worktree readme.txt

同时撤销暂存区和工作区的变动。

其实就是分别执行了上面两条命令。

小结:

撤销工作区命令:

  • git restore --worktree <file> 简写: git restore <file>

  • git checkout -- <file>

    效果一样。

撤销暂存区命令:

  • git restore --staged <file>

  • git reset HEAD <file>

    效果一样。

4. 删除文件

删除文件很简单,就正常删除后,然后commit即可.

问题是如果你误删了文件,那么也可以通过撤销的方式恢复回来

% rm -rf README.md

% git status
On branch main
Changes not staged for commit:
  (use "git add/rm <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	deleted:    README.md

no changes added to commit (use "git add" and/or "git commit -a")

恢复文件:

% git restore README.md
% ls

README.md

注意：从来没有被添加到版本库就被删除的文件，是无法恢复的！

远程仓库

Git是支持托管到远程仓库的,这也是Git的杀手锏之一。通常我们都会托管到 Github这个神奇的地方。

如何注册Github账号就不多赘述,请自行解决。

由于本地的Git与Github的传输时通过SSH加密的,所以我们要生成SSH公私秘钥,并且把公钥设置到Github中。

  1. 生成SSH公私秘钥

    进入当前用户目录下,看看有没有.ssh目录,如果有在查看是否目录内有id_rsa和id_rsa.pub这两个文件。如果有就证明已经生成过,直接跳到下一步即可。

    如果没有,请打开终端,输入一下指令,创建SSH Key:

    ssh-keygen -t rsa -C "zero9501@example.com"

    将邮箱换成你自己的邮箱,然后一路回车使用默认值就可以,也无需设置密码。

    执行完之后你会在用户目录下看到.ssh目录,以及目录内的id_rsa和id_rsa.pub文件。id_rsa是私钥，不能泄露出去，id_rsa.pub是公钥，可以放心地告诉任何人。

  2. 将公钥设置到Github

    登录Github 打开”Settings”设置中的 “SSH and GPG keys” 选择项。

    然后点击 “New SSH key” 创建一个新的SSH Key。

Title随意填写,然后将刚刚生成的id_rsa.pub文件的内容复制到”Key”文本框中 点击”Add SSH key”,有可能需要输入密码验证一下,输入即可。

为什么GitHub需要SSH Key呢？因为GitHub需要识别出你推送的提交确实是你推送的，而不是别人冒充的，而Git支持SSH协议，所以，GitHub只要知道了你的公钥，就可以确认只有你自己才能推送。

当然，GitHub允许你添加多个Key。假定你有若干电脑，你一会儿在公司提交，一会儿在家里提交，只要把每台电脑的Key都添加到GitHub，就可以在每台电脑上往GitHub推送了。

最后友情提示，在GitHub上免费托管的Git仓库，任何人都可以看到喔（但只有你自己才能改）。所以，不要把敏感信息放进去。

如果你不想让别人看到Git库，有两个办法，一个是交点保护费，让GitHub把公开的仓库变成私有的，这样别人就看不见了（不可读更不可写）。另一个办法是自己动手，搭一个Git服务器，因为是你自己的Git服务器，所以别人也是看不见的。这个方法我们后面会讲到的，相当简单，公司内部开发必备。

确保你拥有一个GitHub账号后，我们就即将开始远程仓库的学习。

1. 绑定远程仓库

通常我们会在本地创建一个Git仓库后,然后再Github创建一个远程仓库,并且把它们关联起来进行远程同步,这样Github的远程仓库既可以作为备份使用,也支持其他人协同工作。

首先我们在Github中创建一个远程仓库:

创建完成后,你会得到一个创建地址: https://github.com/****/first_repo.git或者git@github.com:****/first_repo.git

目前这个仓库是空的,一般空的仓库Github会给出对应的提示,如何绑定到你的本地仓库。

现在我们在本地创建一个仓库,并且对它进行初始化:

% mkdir first_repo
% cd first_repo
% git init

Initialized empty Git repository in /usr/local/myproject/git-project/first_repo/.git/

然后通过git remote add命令绑定到远程仓库中:

git remote add origin git@github.com:****/first_repo.git

origin表示远程库的名字,因为一个本地库可以绑定多个远程库,然后分别推送到不同的远程库中。origin是Git默认远程库的叫法。

然后我们可以通过git remote -v命令查看当前已绑定的远程库有哪些:

% git remote -v

origin	git@github.com:****/first_repo.git (fetch)
origin	git@github.com:****/first_repo.git (push)

接下来,就可以通过git push命令把本地库的推送到远程库之中:

git push -u origin main

该命令会将本地仓库的main分支的所有内容推送到origin这个远程仓库的main分支上。

因为是第一次推送所以要加上-u参数，表示不但推送内容,而且会把本地库main分支和远程库的main所关联起来,只有关联起来才可以推送。

执行该命令后可能出现报错:

% git push -u origin main

error: src refspec main does not match any
error: failed to push some refs to 'https://github.com/zlx2019/first_repo.git'

这个错误的解决方案非常简单，首先尝试在该目录中创建一个文件，然后提交并推送到远程服务器。

% touch a.txt
% git add a.txt 
% git commit -m 'init'

# 再次推送即可
% git push -u origin main

Enumerating objects: 3, done.
Counting objects: 100% (3/3), done.
Writing objects: 100% (3/3), 197 bytes | 197.00 KiB/s, done.
Total 3 (delta 0), reused 0 (delta 0), pack-reused 0
To github.com:zlx2019/first_repo.git
 * [new branch]      main -> main
Branch 'main' set up to track remote branch 'main' from 'origin'.

注意:这里仓库地址优先推送使用ssh方式,如果选用https方式需要输入用户名和密码,但是用户名和密码Github已不再支持.

第一次推送关联完成之后,以后commit之后只需要直接执行如下命令进行推送即可:

git push origin main

然后我们可以看一下目前的本地分支情况:

% git branch -v

* main c861a79 init

可以看到，推送的同时已经为我们自动创建了main分支。

查看本地和远程分支:

% git branch -a

* main
  remotes/origin/main

本地和远程分别有一个main分支。

以上情况属于本地库和远程库都是空的情况下,如果本地库和远程仓库都有各自的文件,这样推送是无法直接推送的。如下所示:

% git push -u origin main                

To github.com:zlx2019/first_repo.git
 ! [rejected]        main -> main (fetch first)
error: failed to push some refs to 'github.com:zlx2019/first_repo.git'
hint: Updates were rejected because the remote contains work that you do
hint: not have locally. This is usually caused by another repository pushing
hint: to the same ref. You may want to first integrate the remote changes
hint: (e.g., 'git pull ...') before pushing again.
hint: See the 'Note about fast-forwards' in 'git push --help' for details.

这种情况下,我们先需要通过git pull命令拉取远程库的内容:

# 拉取 origin远程库 的main分支
git pull origin main

然后你会发现,依然会报错

remote: Enumerating objects: 3, done.
remote: Counting objects: 100% (3/3), done.
remote: Total 3 (delta 0), reused 3 (delta 0), pack-reused 0
Unpacking objects: 100% (3/3), 177 bytes | 88.00 KiB/s, done.
From github.com:zlx2019/first_repo
 * branch            main       -> FETCH_HEAD
 * [new branch]      main       -> origin/main
fatal: refusing to merge unrelated histories

这表示github 的仓库和本地的没有一个共同的 commit 所以 git 不让提交.需要添加 --allow-unrelated-histories 告诉 git 允许不相关历史合并。

git pull origin main --allow-unrelated-histories

这样就先把远程库的内容拉取了下来,然后就可以push上去了.

% git push origin main

Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Delta compression using up to 8 threads
Compressing objects: 100% (3/3), done.
Writing objects: 100% (4/4), 444 bytes | 444.00 KiB/s, done.
Total 4 (delta 0), reused 0 (delta 0), pack-reused 0
To github.com:zlx2019/first_repo.git
   c861a79..46bf665  main -> main

2. 移除远程仓库

如果添加错误了仓库地址,也可以删除。

建议先查看绑定了哪些远程仓库:

% git remote -v

origin	git@github.com:****/first_repo.git (fetch)
origin	git@github.com:****/first_repo.git (push)

然后就可以根据仓库名进行删除:

git remote rm origin

3. 修改远程仓库

通过如下命令可以修改已绑定的仓库地址:

git remote set-url <repo-name> <url>

例如:

% git remote set-url origin git@github.com:zlx2019/****.git

4. 克隆仓库

通常我们团队协作开发,建立一个新的项目时,会先建立远程仓库,然后所有参与人员把仓库克隆下来。

假设你现在建立好了远程仓库,只需要通过如下命令就可以克隆下来:

git clone git@github.com:zlx2019/****.git

Git支持多种协议，包括https，但ssh协议速度最快。

小结:

  • 绑定一个远程仓库: git remote add <origin> <git-url>关联一个远程库时必须给远程库指定一个名字，origin是默认习惯命名。
  • 关联后，使用命令git push -u origin main第一次推送main分支的所有内容。
  • 此后，每次本地提交后，只要有必要，就可以使用命令git push origin main推送最新修改。
  • 要克隆一个仓库，首先必须知道仓库的地址，然后使用git clone命令克隆。

分支管理

分支是Git中最为重要最为核心的一个概念,一个仓库可以有多个分支,每个分支都是相互独立的,但是分支之间是可以进行合并的。

在具体的工作过程中,分支通常会这样使用:

分支在实际中有什么用呢？假设你准备开发一个新功能，但是需要两周才能完成，第一周你写了50%的代码，如果立刻提交，由于代码还没写完，不完整的代码库会导致别人不能干活了。如果等代码全部写完再一次提交，又存在丢失每天进度的巨大风险。

现在有了分支，就不用怕了。你创建了一个属于你自己的分支，别人看不到，还继续在原来的分支上正常工作，而你在自己的分支上干活，想提交就提交，直到开发完毕后，再一次性合并到原来的分支上，这样，既安全，又不影响别人工作。

1. 创建合并分支

我们先创建一个仓库:

% mkdir three
% cd three 
% git init

Initialized empty Git repository in /usr/local/myproject/git-project/three/.git/

然后新建一个文本文件,并且写入内容:

% touch title.txt
% echo 'main分支新建的文件' >> title.txt 
% cat title.txt 

main分支新建的文件

注意: 这个时候仓库还没有任何分支,是无法创建新的分支的,我们需要先生成默认的main分支才可以,只需要进行一次commit指令,就会默认生成一个主分支。

% git add title.txt 
% git commit -m 'init branch'

然后查看目前的分支信息:

% git branch 
 
* main

可以看到,目前只有一个main分支,也就是主分支。*表示当前处于的分支。

接下来我们以当前分支为基准,新建一个新的分支(新建的分支将拥有当前分支的所有内容),并且切换到新的分支上:

% git branch dev
% git checkout dev

Switched to branch 'dev'

该命令可以简写为:

git checkout -b dev

然后再次查看下分支信息:

% git branch    

* dev
  main

这样就创建好了一个dev分支,并且切换到了该分支上。不但如此,该分支还持有main分支的所有内容:

% ls
title.txt

% cat title.txt 
main分支新建的文件

然后我们就可以在dev分支上正常工作,如创建一个新的文件,并且写入内容:

% touch body.txt
% echo 'dev分支新建的文件' >> body.txt 
% cat body.txt 

dev分支新建的文件

然后进行提交

% git add body.txt 
% git commit -m 'init dev'

现在dev分支上的工作完成,我们就可以切回到main分支上:

% git checkout main
Switched to branch 'main'

% git branch 
  dev
* main

% ls
title.txt

切换回来后我们发现,刚刚在dev上新建的body.txt文件不见了,这是因为那个提交是在dev分支上,与main分支此刻的提交并没有改变。

现在我们把dev分支合并到当前所在的main分支上:

% git merge dev

Updating cf0982a..96be53c
Fast-forward
 body.txt | 1 +
 1 file changed, 1 insertion(+)
 create mode 100644 body.txt

git merge <branch_name> 命令可以将指定的分支合并到当前所在的分支中。

Fast-forward表示此次合并为 快进模式,因为没有冲突,所以直接把main指向dev的当前提交。

合并完我们就可以查看一下当前main分支的内容:

% ls 
body.txt	title.txt

% cat body.txt 
dev分支新建的文件

然后我们就可以放心删除dev分支:

% git branch -d dev
Deleted branch dev (was 96be53c).

% git branch 
* main

删除完在查看,发现只剩下main分支。

注意: 如果要删除一个没有被合并过的分支,如果删除，将丢失掉修改，如果要强行删除，需要使用大写的-D参数。

git branch -D dev

在Git新版本中提供了新命令来创建切换分支,因为checkout与之前的撤销命令有些冲突,使用起来有些迷惑。

创建新的分支并且切换到新的分支上,可以使用:

% git switch -c dev

Switched to a new branch 'dev'

切换到已有分支上,可以使用:

git switch main

Switched to branch 'main'

使用新的git switch命令，比git checkout要更容易理解。

小结:

  • 查看分支：git branch
  • 创建分支：git branch <name>
  • 切换分支：git checkout <name>或者git switch <name>
  • 创建+切换分支：git checkout -b <name>或者git switch -c <name>
  • 合并某分支到当前分支：git merge <name>
  • 删除分支：git branch -d <name>

2. 解决冲突

关于解决冲突,是合并分支时经常遇到的问题,因为尽管工作时是按照功能或者模块进行划分,但是也避免不了多个人同时修改了同一个文件的内容,这个时候就需要手动解决一下冲突,将两个人修改的内容手动合并到一起,或者删除一方的修改内容。

接下来我们通过一起模拟一个案例来学习如何使用最基本的Git终端命令解决冲突。

首先我们在main分支创建一个main.go文件.

package main

func main(){
    
}

然后进行提交

% git add main.go 
% git commit -m 'main create main.go'

[main bde9db7] main create main.go
 1 file changed, 5 insertions(+)
 create mode 100644 main.go

接下来我们建立一个新的分支dev。

% git switch -c dev

Switched to a new branch 'dev'

在dev分支上修改main.go文件为如下:

package main

func main(){
  name := "张三"
}

然后在dev分支上进行提交。

% git add main.go 
% git commit -m 'dev edit main.go'

[dev 0c33e2d] dev edit main.go
 1 file changed, 1 insertion(+), 1 deletion(-)

提交完成后,切回main分支。

% git switch main
Switched to branch 'main'

注意: 先不要进行合并,假设出于某些原因,在你建立dev分支之后,你又对main分支的main.go文件进行了修改,修改内容如下:

package main

func main(){
  age := 18
}

然后在main分支进行提交:

% git add main.go 
% git commit -m 'main edit main.go'

[main 7a8c1a2] main edit main.go
 1 file changed, 1 insertion(+), 1 deletion(-)

现在dev分支和main分支同时对main.go文件进行了修改.

如果这种情况下把dev合并到main分支,则会产生冲突,如下:

% git merge dev

Auto-merging main.go
CONFLICT (content): Merge conflict in main.go
Automatic merge failed; fix conflicts and then commit the result.

Git提示我们：main.go文件自动合并失败,必须手动解决冲突后再提交。

此时,通过git status也可以提示我们需要解决冲突的文件。

% git status

On branch main
You have unmerged paths.
  (fix conflicts and run "git commit")
  (use "git merge --abort" to abort the merge)

Unmerged paths:
  (use "git add <file>..." to mark resolution)
	both modified:   main.go

no changes added to commit (use "git add" and/or "git commit -a")

接下来我们开始手动解决冲突,打开要解决冲突的文件:

package main

func main(){
<<<<<<< HEAD
  age := 18
=======
  name := "张三"
>>>>>>> dev
}

每一处冲突的地方,<<<<<<< HEAD到=======中间的部分都为本分支的冲突区域.

而=======到>>>>>>> dev中间的部分都为要合并过来分支的冲突区域.

这里我们是想要保留两个分支的修改内容,所以修改如下即可:

package main

func main(){
  age := 18
  name := "张三"
}

然后再次进行提交:

% git add main.go 
% git commit -m '解决冲突完成.'

[main 9d3c354] 解决冲突完成.

当然这是最原生的解决冲突方案,如果修改的范围非常小,自然没问题,但是我们通常冲突的范围都很广,可能涉及到几十上百个文件,这种情况使用这种方案会累死人的…

一般都会使用一些工具进行解决冲突,如本人最喜欢使用 IntelliJ IDEA解决冲突.下面会介绍一下这种方式。

3. IDEA解决冲突

当通过IDEA合并时发生冲突后,会弹出如下界面:

中间的文本框内的文件是需要解决冲突的文件,需要一个一个把冲突手动处理掉。

Accept Yours:就是直接选取本地的代码，覆盖掉远程仓库的

Accept Theirs:是直接选取远程仓库的，覆盖掉自己本地的

Merge:是表示手动合并,这也是我们最常使用的方式。

选中要解决冲突的文件,然后直接点击Merge,开始手动合并,会看到如下界面:

左边部分就是本地仓库代码or当前分支代码。

右边部分就是远程仓库代码or被合并分支代码。

中间部分就是我们手动合并后的结果。

左下角的Accept Left 和Accept Right其实就相当于是之前的 Accept Yours 和 Accept Theirs。

修改完成后，点击Apply，冲突就解决了。

如果取消合并后,代码就会变成原生Git解决冲突那种

修改完成后，对冲突的文件进行 git add 操作，之后 git commit，最后 git push 到远程，这样冲突就解决了。

4. stash储存工作区

当你接到一个序号102的Bug修复任务时,你不自觉的就像以往一样,打算创建一个issue-101来修复它,但是你突然发现目前正在dev分支进行开发,并且任务已经完成了一半,由于才进行到一半还无法提交,但是Bug要求两个小时内修复,这该如何?

zero@Zero codes % git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   mian.go

no changes added to commit (use "git add" and/or "git commit -a")

幸好，Git还提供了一个stash功能，可以把当前工作现场“储藏”起来，等以后恢复现场后继续工作：

zero@Zero codes % git stash 
Saved working directory and index state WIP on main: 357a36b init

现在我们在查看工作空间,发现是干净的.

zero@Zero codes % git status
On branch main
nothing to commit, working tree clean

刚才的工作区哪去了? 这个时候我们通过git stash list可以看看:

zero@Zero codes % git stash list
stash@{0}: WIP on main: 357a36b init

可以看到,这就是我们刚刚的工作区状态。

假如我们现在修复完bug了,只需要把状态恢复如初即可:

zero@Zero codes % git stash apply 

On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   mian.go

no changes added to commit (use "git add" and/or "git commit -a")

这样工作区就恢复了,可以继续正常工作了.

当然我们要删除一下之前存储的工作区:

zero@Zero codes % git stash drop
Dropped refs/stash@{0} (8cf18604954a0d7ff1d4a8967b3f5ace3ab3eb19)

git stash list

再用git stash list查看，就看不到任何stash内容了。

也可以简写为git stash pop,恢复的同时也会把内容直接删除。

5. cherry-pick

假如我们从main分支开辟一个dev分支,几天后突然在main分支发现一个bug,然后由其他工作人员建立了一个新的issue分支修复了,但是仔细想想我们的的dev分支也是由main分支出来的,那么显然我们的分支也存在这个bug,那该如何处理呢?

如果main分支只是修补了这个bug也好,直接合并过来即可,但是如果这几天内,有一些其他分支也都合并到main分支后并且上线了,那么显然直接合并过来是行不通的。

为了方便操作，Git专门提供了一个cherry-pick命令，让我们能复制一个特定的提交到当前分支：

git cherry-pick <commit_id>

在master分支上修复的bug，想要合并到当前dev分支，可以用git cherry-pick <commit>命令，把bug提交的修改“复制”到当前分支，避免重复劳动。

6. 协同开发

当你从一个远程库克隆到本地时,Git会先把目录创建为本地仓库,再创建一个默认分支main并且和远程仓库的main分支关联起来,并且默认的远程库名称为origin。

查看远程库信息:

% git remote 
origin

或者使用更详细的命令查看:

% git remote -v

origin	git@github.com:zlx2019/codes.git (fetch)
origin	git@github.com:zlx2019/codes.git (push)

显示持有origin远程库的拉取(pull)和推送(push)权限。如果没有推送权限则看不到push地址。

推送分支

推送分支，就是把该分支上的所有本地提交推送到远程库。推送时，要指定本地分支，这样，Git就会把该分支推送到远程库对应的远程分支上：

git push origin main

或者新建一个分支,直接推送到远程仓库中:

% git switch -c dev
Switched to a new branch 'dev'


% git push origin dev
Total 0 (delta 0), reused 0 (delta 0), pack-reused 0
remote: 
remote: Create a pull request for 'dev' on GitHub by visiting:
remote:      https://github.com/****/codes/pull/new/dev
remote: 
To github.com:****/codes.git
 * [new branch]      dev -> dev

但是请注意,并不是所有的分支都一定要推送到远程仓库,如果把一些不必要的分支也推送上去,到头来还是要一个一个删除,否则仓库会显得很杂乱。

  • main分支是主分支,这个不用多说肯定要实时同步的。
  • test测试分支一般也是需要实时同步的,如果你不完整同步的话,你的同事很可能因为你的代码推送不完整而无法使用测试环境。
  • dev分支是开发分支，团队所有成员都需要在上面工作，所以也需要与远程同步；
  • 一般修复bug的分支,除了一些非必要情况,一般不会推到远程库。
  • feat分支是否推送,取决于你和同事之间的需求。

抓取远程分支

当你从一个远程库克隆下来后,默认情况下,你只能看到main分支,如果需要dev分支,则需要执行如下命令:

% git switch -c dev origin/dev

Branch 'dev' set up to track remote branch 'dev' from 'origin'.
Switched to a new branch 'dev'

抓取到远程的dev分支。

然后就可以正常开发、提交、推送了。

注意: 推送之前先pull一下代码,看有没有小伙伴提交了新的代码,然后解决下冲突再push。

标签管理

发布一个版本时，我们通常先在版本库中打一个标签（tag），这样，就唯一确定了打标签时刻的版本。将来无论什么时候，取某个标签的版本，就是把那个打标签的时刻的历史版本取出来。所以，标签也是版本库的一个快照。

1. 创建标签

在Git中打一个tag很简单,只要切换到要打tag的分支上,执行如下命令即可:

git tag <版本号>

% git tag v1.0

这样一个版本为v1.0的标签就打好了。下面查看一下标签:

% git tag   
v1.0

标签默认打包的是,当前分支最新的commit上的,当然也可以根据指定的commit的id进行打标签

通过git log指令找到你要打标签的commit-id

% git log --pretty=oneline --abbrev-commit

0f8fa7b (HEAD -> dev, tag: v1.0, origin/dev) aaa
23b20a4 (origin/main, origin/HEAD, main) Initial commit

比如根据23b20a4这个commit进行打包:

% git tag v0.9 23b20a4
% git tag
v0.9
v1.0

然后可以使用git show <tagName>查看一个标签的具体信息:

% git show v1.0

commit 0f8fa7bda229e5f5799fd2650ec56ddf29e0ccf3 (HEAD -> dev, tag: v1.0, origin/dev)
Author: Zero <15133024857@163.com>
Date:   Sat Jan 14 19:14:57 2023 +0800

    aaa

diff --git a/README.md b/README.md
index 8c583d4..edcd17c 100644
--- a/README.md
+++ b/README.md
@@ -1 +1,2 @@
-# codes
\ No newline at end of file
+# codes
+aaa

标签自然也可以带有说明,如下创建:

git tag -a v2.0 -m 'tag_desc' 23b20a4

-a 表示标签名称

-m 表示标签说明

2. 推送标签

推送某个tag到远程库:

git push origin v1.0

一次性将全部未推送的tag推送上去:

git push origin --tags

3. 删除标签

如果标签打错了,也可以进行删除:

git tag -d v1.0

删除远程仓库的标签:

1.先删除本地的标签

git tag -d v0.9
Deleted tag 'v0.9' (was 23b20a4)
  1. 然后推送
% git push origin :refs/tags/v0.9
To github.com:zlx2019/codes.git
 - [deleted]         v0.9

小结:

  • 命令git tag <tagname>用于新建一个标签，默认为HEAD，也可以指定一个commit id；
  • 命令git tag -a <tagname> -m "blablabla..."可以指定标签信息；

  • 命令git tag可以查看所有标签。

  • 命令git push origin <tagname>可以推送一个本地标签；

  • 命令git push origin --tags可以推送全部未推送过的本地标签；
  • 命令git tag -d <tagname>可以删除一个本地标签；
  • 命令git push origin :refs/tags/<tagname>可以删除一个远程标签。

强行关联远程与本地已有分支

假如我们已经写了部分代码,这个时候又想创建远程仓库(生成一些文件,如README.md)并且推送上去,这个时候就需要强制关联一下本地与远程分支,以main分值为例。

  1. 在项目目录创建本地仓库

    git init
git add xxx
git commit -m 'project init'.

  2. 绑定刚创建好的远程仓库

    git remote add origin git@github.com:***/***.git

  3. 关联分支

    git branch --set-upstream-to=origin/main main

    出现错误:

    error: the requested upstream branch 'origin/main' does not exist

    找不到上游分支,执行如下命令,先对main远程分支进行pull。

    可能会进入Vim命令界面,直接wq保存即可.

    git pull origin main --allow-unrelated-histories

    然后再次绑定

    % git branch --set-upstream-to=origin/main main
Branch 'main' set up to track remote branch 'main' from 'origin'.

zero@Zero gtask % git branch -vv
* main 220ec0b Merge branch 'main' of github.com:zlx2019/gtask

  4. 推送代码

    git push origin main

分支命名规范

  • main（主分支，永远是可用的稳定版本，不能直接在该分支上开发
  • test (测试分支,只有在本分支测试通过后才可以合并到main分支)
  • develop（开发主分支，所有新功能以这个分支来创建自己的开发分支，该分支只做只合并操作，不能直接在该分支上开发）
  • feature-xxx(新功能开发分支,在develop上创建分支，以自己开发功能模块命名，功能测试正常后合并到develop分支)
  • Hotfix-xxx(紧急线上bug修复分支,从main分支创建,修复完后再合并到main分支)

提交命名规范

<type> <scope>: <subject>
// 空一行
<info>

  • type用于表示本次提交的类别:

    feature : 新功能。
    fix : 修复bug。
    docs : 文档改变。
    style : 代码格式改变,不影响代码运行的变动。
    refactor : 某个已有功能重构
    perf : 性能优化
    test : 增加测试
    build : 改变了build工具 如 grunt换成了 npm
    revert : 撤销上一次的 commit
    chore : 构建过程或辅助工具的变动

  • scope用于说明 commit 影响的范围，比如数据层、控制层、视图层等等，视项目不同而不同。

  • subject是 commit 目的的简短描述，不超过50个字符。
  • info是对本次提交的详细描述。
