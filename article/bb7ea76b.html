Go核心篇章-Channel原理

发表于2023-07-25|更新于2023-07-27|Go
|字数总计:7.4k|阅读时长:27分钟|阅读量:

前言

Do not communicate by sharing memory; instead, share memory by communicating.

“不要通过共享内存进行通信，而是通过通信来共享内存”,这句话是GO社区中非常经典的一句话。
而本文的主角Channel,就是Go语言中对这句话的完美展现。

Channel是什么?

Channel是Go中一种内置的数据类型,Go推崇使用Channel通信的方式来进行内存共享,它内部封装了共享内存和锁机制,借此抽象出一层更高级的同步机制。

对于传统的共享内存方式,它有如下特点:

  • 解耦
  • 高效
  • 支持实现协程同步
  • …

本章只讲解Channel的原理及源码,一些基础用法等相关使用直接跳过.

核心数据结构

Channel相关的的源码实现位于runtime/chan.go文件。

chan

chan的核心结构如下:

type hchan struct {
  // 当前缓冲区队列的元素数量
  qcount   uint
  // 缓冲区队列的容量
  dataqsiz uint
  // 缓冲区队列(环形队列)
  buf      unsafe.Pointer
  // 元素类型大小
  elemsize uint16
  // 该通道是否关闭。1表示已关闭,0表示未关闭。
  closed   uint32
  // 元素类型
  elemtype *_type
  // 缓冲区队列尾部索引(下一个可以被写入的位置的索引)
  sendx    uint
  // 缓冲区队列头部索引(下一个可以被读取的元素的索引)
  recvx    uint
  // 阻塞队列,用于存放由于读操作而阻塞的G
  recvq    waitq
  // 阻塞队列,用于存放由于写操作而阻塞的G
  sendq    waitq
  // 锁,用于保证并发读写安全性
  lock mutex
}

(1) qcount: 当前缓冲区内的元素数量;

(2) dataqsiz: 缓冲容量大小,默认为0,可以通过make()初始化时指定,指定后无法修改。

(3) buf: 缓冲区,这是一个基于数组实现的环形队列。

(4) elemsize: 通道元素类型大小。

(5) closed: 通道是否被close。如果被重复close,会产生panic。1表示已关闭,0表示未关闭。

(7) elemtype: 通道的元素类型。

(8) sendx: 缓冲区队列头部索引。下一个要入队的元素索引位置。

(9) recvx: 缓冲区队列尾部索引。下一个要出队的元素索引位置。

(10) recvq: 阻塞队列,用于存放由于读操作而阻塞的协程。

(11) sendq: 阻塞队列,用于存放由于写操作而阻塞的协程。

(12) lock: 锁。用于保证并发读写安全性。

waitq

waitq就是上述chan结构中使用到的阻塞队列,用于存放由于读或者写操作而阻塞的协程,它是以链表的结构而存在。

// 阻塞队列
type waitq struct {
  // 队列头部节点
  first *sudog
  // 队列尾部节点
  last  *sudog
}

(1) first: 队列头部节点。

(2) last: 队列尾部节点。

sudog

sudog是waitq中的节点元素。用于标识阻塞的Goroutine(sudog是suspended goroutine的缩写),该结构是针对协程G进行了进一步包装。

源码位于runtime/runtime2.go文件。

type sudog struct {
  // 被阻塞的协程
  g *g
  // 指向下一个被阻塞的协程
  next *sudog
  // 指向上一个被阻塞的协程
  prev *sudog
  // 要从通道中读取或者写入的元素
  elem unsafe.Pointer
  // 开始阻塞的时间戳
  acquiretime int64
  // 被释放阻塞的时间戳
  releasetime int64
  ticket      uint32

  // 当前G,是否参与select语句
  // 用于控制是阻塞模式还是非阻塞模式
  // 如果是true,表示是select语句,那么这种情况是非阻塞模式.
  isSelect bool

  // 表示是否通信成功,如果是true,说明G正常唤醒(值读取到了|值发送完成)
  // 如果是false,表示唤醒是因为通道被close了
  success bool

  // chan指针,指向当前sudog阻塞于哪个chan
  c        *hchan // channel
}

(1) g: 当前被阻塞的具体协程。

(2) next: 下一个节点。

(3) prev: 上一个节点。

(4) elem: 要从通道中读取或者写入的元素。

(4) isSelect: 当前G阻塞是否是因为处于select语句,如果是,那么将会采用非阻塞模式。

源码流程分析

下面通过源码走读的方式来论证一些流程理论。

构造流程

当我们在代码中编写make(chan int)语句时,经过编译后,会执行runtime/chan.go文件中的makechan()函数。

创建流程如下:

  1. 通过runtime/chan.go的makechan()创建。

  2. 计算缓冲区所需要的内存大小mem(mem = 元素类型所占大小 * 缓冲区容量)

  3. 根据不同的情况使用不同方式分配内存。

  • 无内存使用

    如果mem为0,则表示不需要分配缓冲区内存,只需要分配hchanSize(96)内存大小即可,hchanSize是一个Chan所需的固定内存大小。

    那么什么情况下mem会为0呢? 只有两种可能性:

    1. 为无缓冲区通道,如: make(chan any)。
    2. 元素类型是struct{},如make(chan struct{},n),struct{}不占任何内存。

  • 元素类型非指针

    如果元素类型是值类型,那么就需要额外的内存空间来存储,因为值类型会进行值拷贝。所以需要的内存为 缓冲区内存 +hchanSize(96)。

  • 元素类型为指针类型

    如果元素为指针类型,那么会进行两次的内存分配,也就是chan.buf的内存和chan的内存是隔离开的。

  1. chan的值初始化。

  2. 构造完成。

源码细节如下:

// 分配一个Chan
// 参数一: Chan元素类型
// 参数而: 缓冲区大小
func makechan(t *chantype, size int) *hchan {
  // 获取元素类型
  elem := t.elem

  //...

  // 计算缓冲区所需的内存大小
  // mem = 元素类型字节大小 * 缓冲区大小
  mem, overflow := math.MulUintptr(elem.size, uintptr(size))
  // 判断所需内存是否超出上限
  if overflow || mem > maxAlloc-hchanSize || size < 0 {
    panic(plainError("makechan: size out of range"))
  }
  var c *hchan
  // 判断chan是什么类型,采用不同的方式处理
  switch {
    case mem == 0:
    // mem如果为0，那么只会存在两种可能
    // 1.是无缓冲通道 c := make(chan type)
    // 2.元素类型是struct{}  c := make(chan struct{})
    // 编译器对struct{} 这种类型做了优化处理,它不会占用任何内存,表示的就是一个单纯的信号
    // 这两种情况下,是不需要额外的内存

    // 所以这种情况下,我们只需要分配一个固定内存值即可 hchanSize = 96
    // 这个内存值是一个chan排除掉缓冲区后的所需内存大小
    c = (*hchan)(mallocgc(hchanSize, nil, true))
    c.buf = c.raceaddr()
    case elem.ptrdata == 0:
    // 如果元素类型不是一个指针类型
    // 分配内存 缓冲区所需内存大小 + chan本身需要的内存大小
    c = (*hchan)(mallocgc(hchanSize+mem, nil, true))
    // 通过偏移量的方式 为chan的缓冲区设置内存
    c.buf = add(unsafe.Pointer(c), hchanSize)
    default:
    // Elements contain pointers.
    // 如果元素类型是指针类型
    // 由于元素是指针,存放的是地址的指向,所需可以将buf和chan的内存隔离开,分两次分配
    // 分配chan内存
    c = new(hchan)
    // 分配buf的内存
    c.buf = mallocgc(mem, elem, true)
  }
  //设置元素类型大小
  c.elemsize = uint16(elem.size)
  // 设置元素类型
  c.elemtype = elem
  // 设置缓冲区大小
  c.dataqsiz = uint(size)
  lockInit(&c.lock, lockRankHchan)
  if debugChan {
    print("makechan: chan=", c, "; elemsize=", elem.size, "; dataqsiz=", size, "\n")
  }
  return c
}

写操作流程

当通过chan <- data向通道内写入一个数据时,底层的流程是怎样的? chan <- data会编译为runtime/chan.go中的chansend1()函数.

产生错误的操作

写操作会产生错误的几种情况:

  1. 如果通道未初始化,还是nil时向通道写入数据,会产生两种结果:

    • 阻塞模式: 当前协程会通过gopark()被挂起,陷入死锁。

    • 非阻塞模式:会直接返回false。比如在select语句中向通道内写入数据。

  2. 如果通道已经被close了,这时再写入数据同样会有两种结果:

    • 阻塞模式: 会发生panic。
    • 非阻塞模式: 直接返回false。

    源码如下:

    // 判断chan是否初始化
if c == nil {
  //  如果未初始化,并且是非阻塞模式下,直接返回false。
  if !block {
    return false
  }
  //  如果未初始化,是阻塞模式下,则会挂起当前G,并且这个G永远都不会被唤醒,直接陷入死锁。
  gopark(nil, nil, waitReasonChanSendNilChan, traceEvGoStop, 2)
  throw("unreachable")
}
//  如果chan已经被close,并且是非阻塞模式,直接返回false。
if !block && c.closed == 0 && full(c) {
  return false
}
// 如果是阻塞模式,并且被close了,直接panic
if c.closed != 0 {
  unlock(&c.lock)
  panic(plainError("send on closed channel"))
}

情景1: 存在等待读取的协程

写入流程:

  1. 如果写入时,发现recvq(读阻塞队列)中有处于等待协程,那么表示此时的缓冲区是空的。
  2. 直接绕过缓冲区,获取recvq.first(队列头部节点)的等待协程,通过memmove()将元素拷贝给它,并且通过goready()唤醒该协程。

源码如下:

// 从读取的阻塞队列中获取一个读取G(头部),如果有表示当前缓冲区是空的。
// 然后直接绕过缓冲区,将要写入的元素直接发送给这个读取G。
if sg := c.recvq.dequeue(); sg != nil {
  // 将要写入的元素,拷贝给读取等待协程,并且唤醒这个等待协程
  send(c, sg, ep, func() { unlock(&c.lock) }, 3)
  return true
}
// 将一个元素,拷贝给一个等待协程
func send(c *hchan, sg *sudog, ep unsafe.Pointer, unlockf func(), skip int) {
  //...
  if sg.elem != nil {
    // 将元素拷贝给目标等待协程
    // (元素类型,目标协程,元素)
    sendDirect(c.elemtype, sg, ep)
    sg.elem = nil
  }
  // 获取等待协程
  gp := sg.g
  unlockf()
  gp.param = unsafe.Pointer(sg)
  // 设置读取元素成功并且有效的标识
  sg.success = true
  if sg.releasetime != 0 {
    sg.releasetime = cputicks()
  }
  // 唤醒目标协程
  goready(gp, skip+1)
}

func sendDirect(t *_type, sg *sudog, src unsafe.Pointer) {
  dst := sg.elem
  typeBitsBulkBarrier(t, uintptr(dst), uintptr(src), t.size)
  memmove(dst, src, t.size)
}

情景2: 不存在等待读取的协程

  1. 如果当前没有等待的读协程,那就将元素拷贝到缓冲区。

源码细节如下:

// 如果缓冲元素数量 < 缓冲区容量
// 表示当前缓冲区有元素,并且还有可放空间
if c.qcount < c.dataqsiz {
  // Space is available in the channel buffer. Enqueue the element to send.
  // 获取下一个可放入元素的位置索引
  qp := chanbuf(c, c.sendx)
  if raceenabled {
    racenotify(c, c.sendx, nil)
  }
  // 将元素放入指定的位置
  typedmemmove(c.elemtype, qp, ep)
  // 将可放入元素的索引位置后移一位
  c.sendx++
  // 如果sendx已经是缓冲区最大容量,那么就设置为0,成为一个环形
  if c.sendx == c.dataqsiz {
    c.sendx = 0
  }
  // 元素数量+1
  c.qcount++
  // 解锁
  unlock(&c.lock)
  return true
}

情景3: 缓冲区已满,并且没有等待读取的协程

  1. 如果当前缓冲区已经没有可以写入的空间,并且recvq(读阻塞队列)也为空。
  2. 首先,这种情况下如果是非阻塞模式,也就是处于select{}中,会直接返回false。
  3. 正常模式就将当前写协程,放入到sendq写阻塞队列中,直到被其他读协程唤醒,或者由于chan被close而唤醒。
  4. 如果是正常唤醒,则释放资源正常结束,如果是被close唤醒,则发生panic。

源码如下:

// 走到这里就表示,缓冲区已经满了,并且也没有读协程
// 如果是非阻塞模式下,比如在select{}中,则直接返回fasle
if !block {
  unlock(&c.lock)
  return false
}

gp := getg()
// 将当前协程g,包装成等待协程sudog
mysg := acquireSudog()

//...

// 设置要写入的元素
mysg.elem = ep
mysg.waitlink = nil
// 设置g协程
mysg.g = gp
// 设置阻塞模式
mysg.isSelect = false
// 关联通道
mysg.c = c
gp.waiting = mysg
gp.param = nil
// 将包装好的等待协程,放入写阻塞队列
c.sendq.enqueue(mysg)
atomic.Store8(&gp.parkingOnChan, 1)
// 通过gopack()挂起当前的等待协程,当前协程会完全阻塞在这里,直到被其他协程从阻塞队列中取出,并且唤醒....
gopark(chanparkcommit, unsafe.Pointer(&c.lock), waitReasonChanSend, traceEvGoBlockSend, 2)

// 阻塞中....

// 被唤醒,并且要写入的元素已经被读取完毕,直接释放资源即可
gp.waiting = nil
gp.activeStackChans = false
// 获取写操作是否成功标识.
closed := !mysg.success
gp.param = nil
if mysg.releasetime > 0 {
  blockevent(mysg.releasetime-t0, 2)
}
mysg.c = nil
releaseSudog(mysg)
// 如果本次写操作失败处理
if closed {
  if c.closed == 0 {
    // 通道未关闭,退出程序
    throw("chansend: spurious wakeup")
  }
  //如果是因为通道close而操作失败,从而被唤醒,则直接panic。
  panic(plainError("send on closed channel"))
}
return true

总结写流程

插入元素时完整流程如下:

  1. 校验chan是否初始化,如果未初始化,阻塞模式会陷入死锁,而非阻塞模式会返回false。

  2. 校验chan是否已经关闭,如果已经关闭,阻塞模式会panic,而非阻塞模式会返回false。

  3. 对当前chan进行加锁。

  4. recvq(读的阻塞队列)中有sudog(等待协程),表示当前缓冲区是空的,直接跳过缓冲区,将元素复制给读的等待协程。

    1. 将元素通过memmove()拷贝给等待协程。
    2. 通过goready()唤醒等待协程。

  5. buf缓冲区内有元素,并且还没有占满,表示当前没有读的sudog,直接将元素放入缓冲区。

  6. 如果buf缓冲区已满。

    1. 如果是非阻塞模式下会直接返回false。
    2. 阻塞模式会将当前协程,放入sendq(写阻塞队列)中,然后当前协程就会一直阻塞,直到被唤醒。
    3. 写等待协程被唤醒只有两种可能: 一种是正常被读协程唤醒,另一种是由于chan被close而唤醒。
      1. 正常唤醒会返回true正常退出。
      2. 被close唤醒则会发生panic。

整体链路源码如下:

// c <- x 编译后的方法入口
// 向一个通道中发送一个元素
//
//go:nosplit
func chansend1(c *hchan, elem unsafe.Pointer) {
  chansend(c, elem, true, getcallerpc())
}


// 向通道内发送元素
// c     指定的通道
// ep 	  表示元素值
// block 是否为阻塞模式
func chansend(c *hchan, ep unsafe.Pointer, block bool, callerpc uintptr) bool {
  // 1. 判断chan是否初始化
  if c == nil {
    // 1.1 如果未初始化,并且是非阻塞模式下,直接返回false。
    if !block {
      return false
    }
    // 1.2 如果未初始化,是阻塞模式下,则会挂起当前G,并且这个G永远都不会被唤醒,直接陷入死锁。
    gopark(nil, nil, waitReasonChanSendNilChan, traceEvGoStop, 2)
    throw("unreachable")
  }

  //...

  // 2. 如果chan已经被close,并且是非阻塞模式,直接返回false。
  if !block && c.closed == 0 && full(c) {
    return false
  }
  //...

  // 3. 加锁
  lock(&c.lock)
  // 2.1 如果是阻塞模式,并且被close了,直接panic
  if c.closed != 0 {
    unlock(&c.lock)
    panic(plainError("send on closed channel"))
  }

  // 4. 从读取的阻塞队列中获取一个读取G(头部),如果有表示当前缓冲区是空的。
  // 然后直接绕过缓冲区,将要写入的元素直接发送给这个读取G。
  if sg := c.recvq.dequeue(); sg != nil {
    // 4.1 将要写入的元素,拷贝给读取等待协程,并且唤醒这个等待协程
    send(c, sg, ep, func() { unlock(&c.lock) }, 3)
    return true
  }

  // 5. 如果缓冲元素数量 < 缓冲区容量
  // 表示当前缓冲区有元素,并且还有可放空间
  if c.qcount < c.dataqsiz {
    // 获取下一个可放入元素的位置索引
    qp := chanbuf(c, c.sendx)
    if raceenabled {
      racenotify(c, c.sendx, nil)
    }
    // 将元素放入指定的位置
    typedmemmove(c.elemtype, qp, ep)
    // 将可放入元素的索引位置后移一位
    c.sendx++
    // 如果sendx已经是缓冲区最大容量,那么就设置为0,成为一个环形
    if c.sendx == c.dataqsiz {
      c.sendx = 0
    }
    // 元素数量+1
    c.qcount++
    // 解锁
    unlock(&c.lock)
    return true
  }
  // 6.1 走到这里就表示,缓冲区已经满了,并且也没有读协程
  // 		 如果是非阻塞模式下,比如在select{}中,则直接返回fasle
  if !block {
    unlock(&c.lock)
    return false
  }

  gp := getg()
  // 6.2 将当前协程g,包装成等待协程sudog
  mysg := acquireSudog()
  mysg.releasetime = 0
  if t0 != 0 {
    mysg.releasetime = -1
  }
  // 设置要写入的元素
  mysg.elem = ep
  mysg.waitlink = nil
  // 设置g协程
  mysg.g = gp
  // 设置阻塞模式
  mysg.isSelect = false
  // 关联通道
  mysg.c = c
  gp.waiting = mysg
  gp.param = nil
  // 将等待协程,放入写阻塞队列
  c.sendq.enqueue(mysg)
  atomic.Store8(&gp.parkingOnChan, 1)
  // 通过gopack()挂起当前的等待协程
  gopark(chanparkcommit, unsafe.Pointer(&c.lock), waitReasonChanSend, traceEvGoBlockSend, 2)

  // 当前协程会完全阻塞在这里,直到被其他协程从阻塞队列中取出,并且唤醒....


  // 被唤醒,并且要写入的元素已经被读取完毕,直接释放资源即可
  KeepAlive(ep)
  if mysg != gp.waiting {
    throw("G waiting list is corrupted")
  }
  gp.waiting = nil
  gp.activeStackChans = false
  // 获取写操作是否成功标识.
  closed := !mysg.success
  gp.param = nil
  if mysg.releasetime > 0 {
    blockevent(mysg.releasetime-t0, 2)
  }
  mysg.c = nil
  releaseSudog(mysg)
  // 如果本次写操作失败处理
  if closed {
    if c.closed == 0 {
      // 通道未关闭,退出程序
      throw("chansend: spurious wakeup")
    }
    //如果是因为通道close而操作失败,从而被唤醒,则直接panic。
    panic(plainError("send on closed channel"))
  }
  return true

  // 将一个元素,拷贝给一个等待协程
  func send(c *hchan, sg *sudog, ep unsafe.Pointer, unlockf func(), skip int) {
    if raceenabled {
      if c.dataqsiz == 0 {
        racesync(c, sg)
      } else {
        // Pretend we go through the buffer, even though
        // we copy directly. Note that we need to increment
        // the head/tail locations only when raceenabled.
        racenotify(c, c.recvx, nil)
        racenotify(c, c.recvx, sg)
        c.recvx++
        if c.recvx == c.dataqsiz {
          c.recvx = 0
        }
        c.sendx = c.recvx // c.sendx = (c.sendx+1) % c.dataqsiz
      }
    }
    if sg.elem != nil {
      // 将元素拷贝给目标等待协程
      // (元素类型,目标协程,元素)
      sendDirect(c.elemtype, sg, ep)
      sg.elem = nil
    }
    // 获取等待协程
    gp := sg.g
    unlockf()
    gp.param = unsafe.Pointer(sg)
    // 设置读取数据成功，并且有效的标识
    sg.success = true
    if sg.releasetime != 0 {
      sg.releasetime = cputicks()
    }
    // 唤醒目标协程
    goready(gp, skip+1)
  }

读操作流程

下面来看一下读流程的源码细节,源码入口位于runtime/chan.go的chanrecv1()函数。

读取元素完整流程如下:

  1. 校验chan是否初始化,如果为nil,阻塞模式下当前协程会被挂起从而陷入死锁,非阻塞模式将会返回false,false。
  2. 如果chan已经被关闭,并且缓冲区内也没有元素,那么会返回元素类型的零值和false,来表示获取失败的操作。
  3. 加锁,保证数据读写安全性。
  4. 先判断当前是否有写协程在阻塞队列中,如果有写等待协程,则直接取出头部sudog。意味着目前缓冲区内已经满了,否则不可能会有写等待协程。
    1. 如果是无缓冲通道,直接将取出的写协程中的元素拷贝给当前协程。
    2. 如果是有缓冲通道。
      1. 读取缓冲区头部元素,拷贝给当前协程。
      2. 将写协程中的元素拷贝到缓冲区的尾部。
    3. 然后唤醒取出的写协程。
  5. 当前没有写协程在队列,并且缓冲区内有可读元素,直接从缓冲区读取元素即可。
  6. 当前没有写协程在队列,并且缓冲区也为空。
    1. 非阻塞模式下直接返回,读取元素失败。
    2. 阻塞模式下,会将当前读协程放入到读阻塞队列中,挂起该协程,直到被其他写协程唤醒。

源码如下:

// 通道读取元素函数
//
//go:nosplit
func chanrecv1(c *hchan, elem unsafe.Pointer) {
  chanrecv(c, elem, true)
}

// 读取的核心逻辑函数
// c: 要读取的通道
// ep: 元素读取到后放置的容器地址
// block: 是否是阻塞模式
//
// selected: 是否读取成功。
// received: 读取到的数据是否有效,而非通道close后而产生的零值。
func chanrecv(c *hchan, ep unsafe.Pointer, block bool) (selected, received bool) {
  //...

  // 1. 判断chan是否未初始化
  if c == nil {
    // 1.1 如果是非阻塞模式,直接返回false,false,表示读取失败
    if !block {
      return
    }
    // 1.2 如果是阻塞模式,那么直接通过gopark()将当前协程挂起,从而陷入死锁
    gopark(nil, nil, waitReasonChanReceiveNilChan, traceEvGoStop, 2)
    throw("unreachable")
  }

  // Fast path: check for failed non-blocking operation without acquiring the lock.
  // 2. 判断chan是否已经被close (非阻塞模式)
  // 如果是非阻塞模式,并且chan缓存区为空
  if !block && empty(c) {
    // 2.1 如果chan还未关闭,直接返回读取失败
    if atomic.Load(&c.closed) == 0 {
      return
    }
    // 2.2 通道已经关闭,并且缓冲区为空
    if empty(c) {
      //...
      if ep != nil {
        // 将元素类型的零值,拷贝给读协程
        typedmemclr(c.elemtype, ep)
      }
      // 2.3 返回true,false 表示元素为由于chan关闭而产生的零值
      return true, false
    }
  }

  //...

  // 3 加锁
  lock(&c.lock)

  // 2. 判断chan是否已经被close(阻塞模式)

  // 如果通道已关闭
  if c.closed != 0 {
    // 并且通道缓冲区为空
    if c.qcount == 0 {
      if raceenabled {
        raceacquire(c.raceaddr())
      }
      unlock(&c.lock)
      if ep != nil {
        typedmemclr(c.elemtype, ep)
      }
      // 返回元素零值,false.
      return true, false
    }
  } else {
    // 4. 从写阻塞队列中获取协程,如果有那么表示当前缓冲区已经满了
    if sg := c.sendq.dequeue(); sg != nil {
      // 4.1 从缓冲区读取队头元素,然后将写等待协程要写入的元素,拷贝到缓冲区尾部,并且唤醒该协程。
      recv(c, sg, ep, func() { unlock(&c.lock) }, 3)
      return true, true
    }
  }

  // 5. 缓冲区内有元素,但是没有写等待协程
  if c.qcount > 0 {
    // 直接从缓冲区内读取元素

    // 获取缓冲区队头元素
    qp := chanbuf(c, c.recvx)
    if raceenabled {
      racenotify(c, c.recvx, nil)
    }
    // 将元素qp 拷贝给当前读协程
    if ep != nil {
      typedmemmove(c.elemtype, ep, qp)
    }
    // 清除缓冲区头部元素位置
    typedmemclr(c.elemtype, qp)
    // 缓冲区头部索引后移
    c.recvx++
    if c.recvx == c.dataqsiz {
      c.recvx = 0
    }
    // 元素数量-1
    c.qcount--
    // 解锁
    unlock(&c.lock)
    return true, true
  }
  //6. 走到这里表示 缓冲区和写阻塞队列都是空

  //6.1 如果是非阻塞模式,直接返回,表示读取数据失败
  if !block {
    // 解锁
    unlock(&c.lock)
    return false, false
  }
  //6.2 阻塞模式,将当前协程挂起,放入读阻塞队列,直到被唤醒
  // no sender available: block on this channel.
  gp := getg()
  // 将当前协程g 包装成等待协程sudog
  mysg := acquireSudog()
  mysg.releasetime = 0
  if t0 != 0 {
    mysg.releasetime = -1
  }
  // 设置存储元素的指针
  mysg.elem = ep
  mysg.waitlink = nil
  gp.waiting = mysg
  mysg.g = gp
  // 设置为阻塞模式
  mysg.isSelect = false
  // 关联chan
  mysg.c = c
  gp.param = nil
  // 放入读阻塞队列
  c.recvq.enqueue(mysg)
  atomic.Store8(&gp.parkingOnChan, 1)
  // 挂起当前协程
  gopark(chanparkcommit, unsafe.Pointer(&c.lock), waitReasonChanReceive, traceEvGoBlockRecv, 2)

  // 阻塞中....

  // 被唤醒
  // someone woke us up
  if mysg != gp.waiting {
    throw("G waiting list is corrupted")
  }
  gp.waiting = nil
  gp.activeStackChans = false
  if mysg.releasetime > 0 {
    blockevent(mysg.releasetime-t0, 2)
  }
  // 标识读取到的元素是否是有效的
  success := mysg.success
  gp.param = nil
  mysg.c = nil
  releaseSudog(mysg)
  return true, success
}


// 将一个等待协程中元素,拷贝到指定的指针。
func recv(c *hchan, sg *sudog, ep unsafe.Pointer, unlockf func(), skip int) {
  if c.dataqsiz == 0 {
    // 4.1 如果是无缓冲通道
    if raceenabled {
      racesync(c, sg)
    }
    if ep != nil {
      // 直接将写阻塞协程中的元素,拷贝到自己身上
      // copy data from sender
      recvDirect(c.elemtype, sg, ep)
    }
  } else {
    // 4.2 缓冲通道处理
    // 获取缓冲区头部元素,也就是这次要出队的元素
    qp := chanbuf(c, c.recvx)
    if raceenabled {
      racenotify(c, c.recvx, nil)
      racenotify(c, c.recvx, sg)
    }
    if ep != nil {
      // 4.2.1 将要读取的元素qp,拷贝给当前协程ep。
      typedmemmove(c.elemtype, ep, qp)
    }
    // 4.2.2 将写等待协程中的元素,拷贝到缓冲区头部
    typedmemmove(c.elemtype, qp, sg.elem)
    // (非常重要)---将缓冲区头部元素索引后移一位,也就是下一次要出队的元素索引
    // 如果不后移,下次读取到的就是刚刚入队的元素,会破坏队列的特性
    // 移动后,缓冲区头部就会变成尾部
    c.recvx++

    // 回环设置
    if c.recvx == c.dataqsiz {
      c.recvx = 0
    }
    c.sendx = c.recvx // c.sendx = (c.sendx+1) % c.dataqsiz
  }
  // 设置写等待协程相关参数
  // 清除元素
  sg.elem = nil
  gp := sg.g
  unlockf()
  gp.param = unsafe.Pointer(sg)
  // 设置写入元素成功并且有效的标识
  sg.success = true
  if sg.releasetime != 0 {
    sg.releasetime = cputicks()
  }
  // 4.2.3 唤醒写协程
  goready(gp, skip+1)
}

阻塞模式与非阻塞模式

上上面源码流程分析中,阻塞模式与非阻塞模式频繁被我提到,这个概念其实是我们不同的使用方式而产生的。

简单点说,只有在select case语句中使用的时候,chan才是非阻塞模式的，它需要被立即响应并且返回。如下:

c := make(chan int)
select {
  case c <- 1:
  }

当我们手动对chan发起插入或者读取操作时,默认都是阻塞模式。

在select中读取或者插入时,会被编译为selectnbsend()和selectnbrecv函数,它们底层同样也是使用chansend()和chanrecv(),但是第三个参数block被设置为false表示是非阻塞模式。

func selectnbsend(c *hchan, elem unsafe.Pointer) (selected bool) {
  return chansend(c, elem, false, getcallerpc())
}

func selectnbrecv(elem unsafe.Pointer, c *hchan) (selected, received bool) {
  return chanrecv(c, elem, false)
}

读取返回协议

读取 channel 时，可以根据第二个 bool 型的返回值用以判断当前 channel 是否已处于关闭状态：

c := make(chan struct{},2)
v1 := <- c
v2,ok := <- c

实现上述功能的原因是，两种格式下，读 channel 操作会被汇编成不同的方法：

// 通道读取元素函数
func chanrecv1(c *hchan, elem unsafe.Pointer) {
	chanrecv(c, elem, true)
}

// 通道读取元素函数,received表示读取到的值是否有效。
func chanrecv2(c *hchan, elem unsafe.Pointer) (received bool) {
	_, received = chanrecv(c, elem, true)
	return
}

关闭流程

当我们对一个通道关闭时,流程如下:

  1. 判断通道是否未初始化,未初始化则panic。
  2. 加锁。
  3. 通道如果已经被关闭,也会发生panic。(通道只能被关闭一次)
  4. 设置通道为已关闭状态。
  5. 遍历读阻塞队列中的所有等待协程,放到待唤醒协程列表。
  6. 遍历写阻塞队列中的所有等待协程,放到待唤醒协程列表。(由于close而被唤醒的所有写协程,被唤醒后会全部panic)
  7. 解锁。
  8. 唤醒所有的等待协程。

思考: 通常5和6不可能都执行,因为读和写本来就是互斥的,如果存在读等待协程那就意味着缓冲区是空的,反之如果存在写等待协程,那就意味着缓冲区已经满了。

源码流程如下:

// 关闭一个通道
func closechan(c *hchan) {
  // 1. 如果通道未初始化,直接panic
  if c == nil {
    panic(plainError("close of nil channel"))
  }
  // 2. 加锁
  lock(&c.lock)
  // 3. 如果通道已经关闭了,同样会panic.只能被close一次。
  if c.closed != 0 {
    unlock(&c.lock)
    panic(plainError("close of closed channel"))
  }

  if raceenabled {
    callerpc := getcallerpc()
    racewritepc(c.raceaddr(), callerpc, abi.FuncPCABIInternal(closechan))
    racerelease(c.raceaddr())
  }
  //4. 将通道设置为已关闭状态
  c.closed = 1
  // 待唤醒的等待协程列表
  var glist gList
  // 5. 遍历读阻塞队列中的所有等待协程,放到待唤醒协程列表
  for {
    // 弹出协程
    sg := c.recvq.dequeue()
    if sg == nil {
      break
    }
    // 清空元素值
    if sg.elem != nil {
      typedmemclr(c.elemtype, sg.elem)
      sg.elem = nil
    }
    if sg.releasetime != 0 {
      sg.releasetime = cputicks()
    }
    gp := sg.g
    gp.param = unsafe.Pointer(sg)
    // 设置读取操作失败标识
    sg.success = false
    if raceenabled {
      raceacquireg(gp, c.raceaddr())
    }
    // 放入唤醒列表
    glist.push(gp)
  }
  // 6. 遍历写阻塞队列中的所有等待协程,放到待唤醒协程列表
  for {
    sg := c.sendq.dequeue()
    if sg == nil {
      break
    }
    sg.elem = nil
    if sg.releasetime != 0 {
      sg.releasetime = cputicks()
    }
    gp := sg.g
    gp.param = unsafe.Pointer(sg)
    // 设置写入失败操作标识
    // 注意这项标识会让被唤醒的写协程被唤醒后产生panic。
    sg.success = false
    if raceenabled {
      raceacquireg(gp, c.raceaddr())
    }
    // 放入唤醒列表
    glist.push(gp)
  }
  // 7. 解锁
  unlock(&c.lock)

  // 8. 唤醒所有等待协程
  for !glist.empty() {
    gp := glist.pop()
    gp.schedlink = 0
    goready(gp, 3)
  }
}

面试题

要求实现一个Map结构,需要满足如下特性:

  • 支持高并发安全
  • 只存在插入和查询操作o(1)
  • 查询时,如果key存在则直接返回,如果不存在则阻塞等待key插入后获取,若等待超时后返回超时错误
  • 不能死锁或者panic

该题可以利用channel的特性去实现,实现如下:

type ConcurrentMap[K comparable, V any] struct {
  // 互斥锁,保证并发安全
  sync.Mutex
  // 元素K-V容器
  container map[K]V
  // 用于通知key被插入的通道容器
  keyTiggerMap map[K]chan struct{}
}

// 构造函数
func NewConcurrentMap[K comparable, V any]() *ConcurrentMap[K, V] {
  return &ConcurrentMap[K, V]{
    container:    make(map[K]V),
    keyTiggerMap: make(map[K]chan struct{}),
  }
}

// 插入
func (c *ConcurrentMap[K, V]) Put(key K, value V) {
  c.Lock()
  defer c.Unlock()
  // 插入元素
  c.container[key] = value
  // 如果有协程等待获取这个key,就唤醒
  tigger, ok := c.keyTiggerMap[key]
  if ok && tigger != nil {
    //tigger <- struct{}{}
    // 关闭通道,从而使得 所有的读协程被唤醒
    close(tigger)
    // 删除key的chan,防止被重复close
    delete(c.keyTiggerMap, key)
  }
}

// 查询
func (c *ConcurrentMap[K, V]) Get(key K, timeout time.Duration) (V, error) {
  c.Lock()
  // 查询key是否存在
  value, ok := c.container[key]
  if ok {
    // key存在
    c.Unlock()
    return value, nil
  }

  // 检查是否已经有协程在阻塞等待这个key
  tigger, ok := c.keyTiggerMap[key]
  if !ok {
    // 不存在则创建新的chan,放入通道map中
    tigger = make(chan struct{})
    c.keyTiggerMap[key] = tigger
  }
  // 如果已经存在,那么直接复用即可
  // 阻塞前先释放锁
  c.Unlock()
  select {
    case <-tigger:
    // 然后阻塞等待这个key被添加到容器
    case <-time.After(timeout):
    // 如果超时,返回err
    return value, errors.New("time out")
  }
  // 再次获取value
  c.Lock()
  value = c.container[key]
  c.Unlock()
  return value, nil
}
