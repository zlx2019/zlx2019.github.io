Vertx基础入门

Vertx 是什么?

Vertx是款用于构建高性能、可伸缩、异步应用程序的开源框架。它使用基于事件驱动的、非阻塞IO模型。响应式应用既可以随着工作负载的增长而扩展，又可以在出现故障时 弹性回复。响应式应用是即时响应的，因为它能有效利用系统资源并保护自身免受错误影响，从而使延迟可控。

Vert.x 背靠庞大的响应式模块生态系统，具有编写现代服务所需的一切：全面的Web技术栈，响应式数据库驱动程序、消息队列、事件流、集群支持、指标监控、分布式跟踪等等。

简而言之,Vert.x是一款功能强大、易于使用的开源框架，适用于构建各种类型的应用程序，包括Web应用程序、实时数据处理应用程序、微服务等。

Get Started

下面我们使用Vert搭建一个简单的HTTP服务.

  1. 打开start.vertx.io网页,构建一个新的项目。

这与SpringBoot项目差不多,就不多赘述了。这里选择了Web依赖。

  1. 然后点击Generate Project下载项目压缩文件,并且进行解压。

  2. 通过Maven下载完所需要的依赖。

  3. 打开主文件io/zero/MainVerticle.java,并作出如下修改:

    public class MainVerticle extends AbstractVerticle {

  /**
   * Vertx 程序启动入口
   */
  @Override
  public void start(Promise<Void> startPromise) throws Exception {
    // 创建一个Router路由
    Router router = Router.router(vertx);

    // 为所有的请求挂载一个handler处理器,类似于过滤器
    router.route().handler(context->{
      // 获取到连接的请求
      HttpServerRequest request = context.request();
      // 获取请求的客户端地址信息
      String address = request.connection().remoteAddress().toString();

      // 获取请求的name参数
      MultiMap params = context.queryParams();
      String name = params.contains("name") ? params.get("name") : "unknown";
      //以json格式响应回客户端
      context.json(new JsonObject()
        .put("name",name)
        .put("address",address)
        .put("message","Hello Vertx"));
    });
    // 创建HTTP服务
    vertx.createHttpServer()
        // 使用定义好的路由
        .requestHandler(router)
        // 监听的端口
        .listen(8080)
        // 服务启动成功后的回调
        .onSuccess(server -> {
          System.out.println("HTTP Server Started On Port " + server.actualPort());
        });
  }
}

  4. 然后通过mvn package进行打包

  5. 运行应用

    mvn exec:java

HTTP Server Started On Port 8080...

  6. 测试

    curl -v 'http://localhost:8080?name=zhangsan'

    响应结果:

    {
    "name": "张三",
    "address": "127.0.0.1:62803",
    "message": "Hello Vertx"
}

Vertx 核心模块

Vertx的核心模块被称为Vertx Core

Vertx Core提供下列功能项:

  • 编写TCP服务端与客户端
  • 编写支持WebSocket的HTTP客户端和服务端
  • Event bus 事件总线
  • 共享数据 —— 本地的Map和分布式集群Map
  • 周期性、延迟性动作
  • 部署和撤销 Verticle 实例
  • 数据报套接字
  • 编写DNS客户端
  • 编写文件系统操作
  • 高可用性
  • 本地传输
  • 集群

Vert.x Core 中的功能相当底层，不包含诸如数据库访问、授权或高层 Web 应用的功能。 您可以在 Vert.x ext （扩展包）（译者注：Vert.x的扩展包是Vert.x的子项目集合，类似 Web、Web Client、Databases等）中找到这些功能。

Vert.x Core 小巧而轻便，您可以只使用您需要的部分， 它可整体嵌入现存应用中。 Vert.x没有强制要求使用特定的方式构造应用。

Vert.x 也支持在其他语言中使用Vert.x Core,而且在使用诸如 JavaScript或Ruby等语言编写Vert.x代码时，无需直接调用 Java的API；毕竟不同的语言有不同的代码风格，若强行让 Ruby开发人员遵循Java的代码风格会很怪异， 所以我们根据 Java API 自动生成了适应不同语言代码风格的 API。

接下来我们详细学习Vertx Core模块中的内容.请确保项目已经导入该依赖:

<dependency>
 <groupId>io.vertx</groupId>
 <artifactId>vertx-core</artifactId>
 <version>4.4.0</version>
</dependency>
dependencies {
 compile 'io.vertx:vertx-core:4.4.0'
}

Vertx

使用Vert.x框架开发离不开Vertx核心对象,它是Vert.x的控制中心,也是实现一切事情的基石,包括创建各种客户端、服务端,获取事件总线、定时器等等。

可以通过如下代码创建一个Vertx实例:

Vertx instance = Vertx.vertx();

注意
大部分应用将只会需要一个Vert.x实例，但如果您有需要也可创建多个Vert.x实例， 如：隔离的事件总线或不同组的客户端和服务器。

Vertx配置项

可在创建 Vertx 对象的同时指定配置项：

// 指定工作线程池的大小, 默认是20
Vertx vertx = Vertx.vertx(new VertxOptions().setWorkerPoolSize(40));

Vertx集群

如果您想创建一个 集群模式的 Vert.x 对象（参考 event bus章节了解更多事件总线集群细节），那么通常情况下您将需要使用另一种异步的方式来创建 Vertx 对象。

这是因为让不同的 Vert.x 实例组成一个集群需要一些时间（也许是几秒钟）。在这段时间内，我们不想去阻塞调用线程，所以我们将结果异步返回给您.

示例:

// 创建集群配置项
VertxOptions options = new VertxOptions();
// 创建Vertx集群,注册一个方法,集群创建成功后会回调
Vertx.clusteredVertx(options,res->{
  if (res.succeeded()){
    System.out.println("集群创建成功");
    // 获取集群模式下的Vertx实例
    Vertx result = res.result();
    //TODO 可以做一些后置处理
  }else {
    // 创建集群失败,可能是集群管理器出现了问题
  }
});

流式风格

request.response().putHeader("Content-Type", "text/plain").end("some text");

这是贯穿Vert.xAPI 中的一个通用模式，所以请适应这种代码风格。

流式调用可以让代码更为简洁。 当然，Vert.x并 不强制 您用这种方式书写代码，如果您更倾向于用以下非流式编码， 您可以忽略它：

HttpServerResponse response = request.response();
response.putHeader("Content-Type", "text/plain");
response.write("some text");
response.end();

关于编码风格,可以根据个人喜好。

Vertx: 不要打电话给我,有需要我会打给你

Vert.x的API大部分都是事件驱动 的。这意味着当您感兴趣的事情发生时,会以事件的形式发送给您。

这与Java中的CompositeFuture类似,全部基于事件回调来实现异步操作。

如下这些例子:

  • 注册一个定时处理器
// 注册一个定时器, 每秒执行一次
vertx.setPeriodic(1000,id-> {
  System.out.println(id);
});
  • 注册一个HTTP请求处理器
// 服务器每次收到一个HTTP请求时这个处理器将被调用
server.requestHandler(request -> {
  request.response().end("hello world!");
});

当这些对应的事件被触发后,它会异步地调用这些处理器。

请不要阻塞我!

Vertx中几乎所有API都是异步的,除了个别特例的API(如以Sync结尾的某些文件操作API)。

如果是同步API会立刻返回,反之异步的需要提供一个处理器(Handler),来处理稍后回调的事件。

因为Vert.x API不会阻塞线程， 所以通过Vert.x您可以只使用少量的线程来处理大量的并发。

当使用传统的阻塞式API做以下操作时，调用线程可能会被阻塞：

  • 从 Socket 中读取数据
  • 写数据到磁盘
  • 发送消息给接收者并等待回复
  • 其他很多情况

这种情况下,线程在等待阻塞处理结果时不能做任何事,这意味着如果使用阻塞式API处理大量并发操作,需要更多的线程来防止整个系统停止运作

而这些线程使用的内存（例如它们的栈）和线程上下文切换开销很可观。

这意味着，阻塞式的方式对于现代应用程序所需要的并发级别来说是难于扩展的。

Reactor && Multi-Reactor模式

我们前边提过 Vert.x 的 API 都是事件驱动的，当有事件时 Vert.x 会将事件传给处理器来处理。

在多数情况下，Vert.x使用被称为 Event Loop 的线程来调用您的处理器。

由于Vert.x或应用程序的代码块中没有阻塞代码，Event Loop 可以在事件到达时快速地分发到不同的处理器中。

由于没有阻塞代码，Event Loop 可在短时间内分发大量的事件。 例如，一个单独的 Event Loop 可以非常迅速地处理数千个 HTTP 请求。

这种被称之为Reactor 模式(反应器模式),或许你听说过它,例如Python实现协程的asyncio模块也是基于这种模式,Node.js也实现了该模式。

在一个标准的Reactor实现中，有 一个独立的 Event Loop 会循环执行， 处理所有到达的事件并传递给处理器处理。

Vert.x的工作方式有所不同。每个 Vertx 实例维护的是 多个Event Loop 线程。 默认情况下，我们会根据机器上可用的核数量来设置 Event Loop 的数量，您亦可自行设置。

我们将这种模式称为 Multi-Reactor 模式（多反应器模式），区别于单线程的 Reactor 模式（反应器模式）。

注意:
即使一个 Vertx 实例维护了多个 Event Loop，任何一个特定的处理器永远不会被并发执行。
大部分情况下（除了 Worker Verticle 以外） 它们总是在同一个 Event Loop 线程中被调用。

不要阻塞Event Loop!

尽管Vert.x 的API都是非阻塞式的，且不会阻塞 Event Loop， 但是用户编写的处理器中可能会阻塞Event Loop。

如果这样做，该Event Loop在被阻塞时就不能做任何事情；如果您阻塞了Vertx实例中的所有Event Loop，那么您的应用就会完全停止！

导致Event Loop阻塞的原因可能有如下几种:

  • Thead.sleep()
  • 等待一个锁
  • 等待一个互斥信号或监视器（例如同步的代码块）
  • 执行一个长时间数据库操作并等待其结果
  • 执行一个复杂的计算，占用了可感知的时长
  • 在循环语句中长时间逗留

如果上述任何一种情况停止了 Event Loop 并占用了 显著执行时间 ， 那您应该去面壁（译者注：原文此处为 Naughy Step，英国父母会在家里选择一个角落作为小孩罚站或静坐的地方，被称为 naughty corner 或 naughty step），等待下一步的指示。

所以，什么是显著执行时间 ？您要等多久？它取决于您的应用程序和所需的并发数量。

如果您只有单个Event Loop，而且您希望每秒处理10000个 HTTP 请求， 很明显的是每一个请求处理时间不可以超过0.1毫秒，所以您不能阻塞任何过多（大于0.1毫秒）的时间。

如果您的应用程序没有响应，可能这是一个迹象，表明您在某个地方阻塞了Event Loop。 为了帮助您诊断类似问题，若 Vert.x 检测到 Event Loop 有一段时间没有响应，将会自动记录这种警告。 若您在日志中看到类似警告，那么您需要检查您的代码。比如：

Thread vertx-eventloop-thread-3 has been blocked for 20458 ms

Vert.x 还将提供堆栈跟踪，以精确定位发生阻塞的位置。

Future

Vert.x4使用Future对象作为异步API的结果对象。这里的Future可不是指的Java中的Future哦,是Vert.x自己的io.vertx.core.Future。

Future对象包含了成功或失败的异步结果。

我们不能直接操作future的异步结果，而应该设置future的handler；当future执行完毕，结果可用时，会调用handler进行处理。

如下:

// 创建一个vertx实例
Vertx instance = Vertx.vertx();
// 获取vert封装了文件操作的工具
FileSystem fs = instance.fileSystem();
// 得到一个Future异步结果
Future<FileProps> future = fs.props("/admin.txt");

// 注册成功的回调处理器
future.onSuccess(fileProps -> {
  System.out.println("操作成功");
});
// 注册失败的回调处理器
future.onFailure(e -> {
  System.out.println("操作失败: " + e.getMessage());
});

通过注册onSuccess()和onFailure()回调来处理结果。也可以将两种结果处理写到一个处理器,注册onComplete()回调:

// 注册API完成的回调处理器
future.onComplete(result -> {
  if (result.succeeded()){
    System.out.println("操作成功");
  }else {
    System.out.println("操作失败: " + result.cause().getMessage());
  }
});

注意:

Vert.x 3的API只提供了回调模式；为了减少从Vert.x 3迁移到Vert.x 4的工作量，Vert.x 4为每个异步方法都保留了回调版本。 如上面样例代码的 props 方法，提供了带回调参数的版本 props


注意:

请勿混淆 futures 与 promises 。

如果 futures 代表的是异步结果的 “读取端”，那么 promise 就是 “写入端”。 它们允许您推迟获取结果的操作。

在大多数情况下，您不需要在 Vert.x 应用程序中自行创建 promise。 Future组合 和 Future协作 为您提供了转换和合并异步结果的工具。

但是，如果，在您的代码库中，有使用回调的历史遗留方法，那么您可以利用 promise 来扩展 io.vertx.core.Handler<io.vertx.core.AsyncResult>

Promise<String> promise = Promise.promise();
legacyGreetAsync(promise);
Future<String> greeting = promise.future();

诸如 onSuccess、onFailure 和 onComplete 之类的终端操作并不能保证回调的调用顺序。

考虑注册 2 个回调的 future：

future.onComplete(ar -> {
  // 做些什么
});
future.onComplete(ar -> {
  // 可能先被调用
});

第二个回调完全有可能在第一个回调之前被调用。

如您需要保证顺序，考虑将 Future组合 与andThen一起使用。
