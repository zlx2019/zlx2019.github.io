是什么是锁?

编程语言中的锁,是一种抽象的概念,主要控制多线程操作共享数据之间的安全问题。在做操作共享数据之前,加上一把锁,就能独自享有操作权,其他线程无法进行操作。

锁的概念分类

悲观锁

悲观锁是一种悲观的思想,它总是认为会有其他线程来争抢同一资源,所以无论怎样,都要对资源进行加锁。这样其他线程想要请求这个资源的时候就会阻塞，直到等到悲观锁把资源释放为止。传统的关系型数据库里边就用到了很多这种锁机制，比如行锁，表锁等，读锁，写锁等，都是在做操作之前先上锁。

乐观锁

乐观锁的思想与悲观锁相反,它默认是不会进行加锁,但是在乐观锁进行数据修改时,会通过一种原子操作来检查数据是否已经被修改过。只有数据没有被修改过,才会继续执行下去。一般都是通过CAS进行乐观锁的实现。

同步锁/互斥锁

同一时间,只能被一个线程所访问。

重入锁

被同一个线程多次获取锁,不需要频繁的加锁与释放锁。

公平锁

是按照通过CLH等待线程按照先来先得的规则，线程依次排队，公平的获取锁，是互斥锁的一种。Java的ReetrantLock就实现了该锁: new ReetrantLock(true)。

非公平锁

是当线程要获取锁时，它会无视CLH等待队列而直接获取锁。ReetrantLock默认为非公平锁，或——Lock lock = new ReetrantLock(false)。

读锁/共享锁

能被多个线程同时获取、共享的锁。即多个线程都可以获取该锁，对该锁对象进行处理。典型的就是读锁ReentrantReadWriteLock.ReadLock。即多个线程都可以读它，而且不影响其他线程对它的读，但是大家都不能修改它。

本章主要讲一下Java原生支持的锁: synchronized。

Java对象结构

首先了解一下,Java中的Object结构,这很重要。

主要分为三部分:

  • 对象头

  • 实例数据

    实例数据就是初始化该对象时,设置的属性和状态的内容。

  • 对齐填充字节

​ 填充字节是为了满足Java对象大小必须是8bit的倍数,如果不是,则填补一些无意义的字节。

而对象头则是本章的重点,它存放了一些对象本身的运行时信息。相对于实例数据,对象头属于一些额外的存储开销,所以它设计的极小。

对象头包含两部分:

  • klass Point

    klassPoint是一个指针,它指向了当前对象类型所在方法区中的类型元信息。

  • Mark Word

    对象的运行时信息全都存在于MarkWord中。如下图所示:

    可以看出来,MarkWord所占内存极小,所有标志位加起来只有32bit。不同的bit位,表示不同的数据状态。

    synchronized加锁后的信息,全部存在这里。

synchronized

我们都知道,在Java中synchronized关键字是用来加锁的关键字,被编译后会生成 monitorenter和monitorexit两条指令,分别对应加锁与解锁。

接下来我们通过一个案例,来验证一下

public class Main {
    public static void main(String[] args)  {
        synchronized (Main.class){
        }
    }
}

然后对该文件进行编译,编译为字节码文件.

javac Main.java

javap -c Main.class

monitor通常被翻译会监视器或者管程,而线程获取一个对象的锁其实就是获取对象的monitor,一旦有一个线程获取到了一个对象的monitor那么其他线程则无法获取。当一个线程获取到monitor后,由于在执行过程中,进入了Waiting状态,则会释放锁,进入的waitSet中,等待其他线程调用该对象的notfiy或者notifyAll进行唤醒。

monitor是依赖于操作系统的mutex lock实现,由于Java线程是映射的操作系统的线程,所以每当挂起或者唤醒一个线程,都需要消耗很大的资源,需要切换操作系统的内核态,这种操作是比较重量级的。

在Java6开始,Java对 synchronized进行了优化,引入了锁状态的升级。

锁优化

在Java6中,引入了锁的四种状态,分别为无锁、偏向锁、轻量级锁、重量级锁,对应了 MarkWord中所记录的锁标识位。需要注意的是锁只能升级,而不能降级。

1. 无锁

一个对象的默认状态,没有任何线程获取锁的情况下。标志位为: 01

2. 偏向锁

偏向锁也是一种特殊的无锁状态,它同样也不会进行加锁操作。这种情况是只有一个线程抢占这把锁的时候,会把当前线程ID记录到MarkWord中,当这个线程重入获取锁时,只需要简单校验一下线程ID是否一致即可,所以被称为偏向锁。

偏向锁的标志位同样也为01,那么如何区分究竟是无锁还是偏向锁呢?只需要判断MarkWord倒数第三个bit位是否为1即可.

如果为1表示为偏向锁。如果是偏向锁,会将当前偏向的线程的ID,记录到MarkWord的前23个bit位中。

如果为0表示为无锁。

偏向锁在JDK 6及以后的JVM里是默认启用的。可以通过JVM参数关闭偏向锁：-XX:-UseBiasedLocking=false，关闭之后程序默认会进入轻量级锁状态。

3. 轻量级锁

是指当锁是偏向锁的时候，被另外的线程所访问，偏向锁就会升级为轻量级锁,标志位更改为00，其他线程会通过自旋的形式尝试获取锁，不会阻塞，从而提高性能。

其他线程自旋等待时,会有一个自适应的优化: 假如A线程在自旋等待锁,A线程在刚刚已经获取过这把锁,那么虚拟机就会认为A线程这次也可以获取到这把锁,会让你自旋的时间更久一些。

若当前只有一个等待线程，则该线程通过自旋进行等待。但是当自旋超过一定的次数，或者一个线程在持有锁，一个在自旋，又有第三个来访时，轻量级锁升级为重量级锁。

4. 重量级锁

升级为重量级锁时，锁标志的状态值变为“10”，此时Mark Word中存储的是指向重量级锁的指针，此时等待锁的线程都会进入阻塞状态。
