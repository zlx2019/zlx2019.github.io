前言

什么是MQ?

MQ全称Message Queue(消息队列)，是在消息的传输过程中保存消息的容器。多用于分布式系统之间进行通信。队列:数据结构的一种，特征为“先进先出”

为什么需要引入MQ?

引入MQ后的优势

  1. 应用解耦

    多个应用间的数据传递,通过MQ后,将保有持久性,消费者应用挂掉后与生产者应用没有任何影响

  2. 异步提速

    提升系统的体验性,增加系统吞吐量

  3. 流量削峰

    提高系统的稳定性,防止被高流量打垮

引入MQ的劣势

  1. 系统可用性降低

    一旦引入中间件,如果中间件垮掉,那么就会对业务造成影响,所以要保证MQ的高可用

  2. 系统复杂度提高

    通过MQ进行异步调用,如何保证消息不会重复消费,怎么处理消息丢失情况,如何保证消息传递的顺序性?

  3. 一致性问题

    如A系统处理完后,通过MQ通知B,C,D三个系统,如果D系统处理失败,如何保证消息数据处理的一致性

1. RocketMQ基础概念

  1. Producer Cluster(消息生产者集群)

    指我们编写的应用程序,发送消息端

  2. Consumer Cluster(消息消费者集群)

    指我们编写的应用程序,消费消息端

  3. NameServer Cluster(消息服务器管理集群)

    Broker Cluster的管理者,存储一些元数据信息,如Broker的集群机器IP等

  4. Broker Cluster(消息服务器集群)

    具体的消息服务器

RocketMQ提供两种消费模式

  • 拉取模式

    循环请求MQ,浪费资源

  • 推送模式(推荐)

    长链接+监听器实现

package com.lxz.consumer;

import cn.hutool.core.util.StrUtil;
import org.apache.rocketmq.client.consumer.DefaultMQPushConsumer;
import org.apache.rocketmq.client.consumer.listener.ConsumeConcurrentlyStatus;
import org.apache.rocketmq.client.consumer.listener.MessageListenerConcurrently;

/**
 * 消费者
 * @author lx Zhang.
 * @date 2021/6/17 4:38 下午
 */
public class Consumer {

    /** 创建默认消费者(推送模式)*/
    private static DefaultMQPushConsumer consumer = new DefaultMQPushConsumer("defaultGroup");
    static {
        /** 指定nameserver*/
        consumer.setNamesrvAddr("localhost:9876");
    }

    public static void main(String[] args) throws Exception{
        consumerMessage();
    }

    public static void consumerMessage()throws Exception{
        /** 指定要消费的topic，和tags,*表示全部 */
        consumer.subscribe("defaultTopic","*");
        /** 注册监听器,处理消息*/
        consumer.registerMessageListener((MessageListenerConcurrently) (messageList, context) -> {
            messageList.forEach(message-> {
                System.out.println(StrUtil.format("消费者接收到消息:topic-> [{}] tags-> [{}],body-> 【{}】", message.getTopic(), message.getTags(), new String(message.getBody())));
            });
            return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
        });
        consumer.start(); //开启消费者
        System.out.println("消费者已开启...");
    }
}

3. 复杂情况

假如 defaultTopic4主题,有两个消费组,groupA与groupB

groupA内有Consumer1 与 Consumer2消费者

groupB内有Consumer3 消费者

生产者向defaultTopic4发送10条消息

那么这么情况下消息会如何分配?

首先,不同的topic都会接受到全部的消息,也就是groupA与groupB均会收到10条消息。然后每个组将自己得到的10条消息,均衡的转发给不同的消费者。结果就是groupA中的Consumer1和Consumer2各自消费5条消息,而groupB中的Consumer3由于组内只有自己,所以消费10条消息

4. 消费组的模式设置

通过以上的案例,得出结论:默认的一个消费组内,是进行负载均衡消费,但是如果硬要一个组内的所有消费者都能消费到所有数据,该如何去做呢？我们可以通过修改消费组的消费模式去做到,可以修改为广播模式

consumer.setMessageModel(MessageModel.CLUSTERING); //默认的集群模式,负载均衡消费模式
consumer.setMessageModel(MessageModel.BROADCASTING); //广播模式

7. 同步消息

特性：及时性、可靠性较强,发送后会等待响应,一般适用一些重要性消息,如涉及到金钱交易相关的消息

上面我们发送的基础消息就是同步消息,所以就不写案例了

8. 异步消息

特性：效率快、不会立刻得到响应,但是会有异步的回调，通常适用一些 非特别重要,而且要求效率高的消息,异步消息可以大幅度提升系统的吞吐量

    /**
     * 发送异步消息
     * @throws Exception
     */
public static void asyncSendMessage() throws Exception{
  producer.start();
  Message message = new Message("asyncTopic", "asyncTags", "async message".getBytes());
  /** 指定要发送的消息和注册异步回调函数*/
  producer.send(message, new SendCallback() {
    /**
     * 发送成功后的回调函数
     * @param sendResult
     */
    @Override
    public void onSuccess(SendResult sendResult) {
      System.out.println("异步消息发送结果: " + sendResult);
    }

    /**
     * 发送失败后的回调函数
     * @param e
     */
    @Override
    public void onException(Throwable e) {
      System.out.println("消息发送失败: " + message);
    }
  });
}

9. 单向消息

类似于UDP,只负责消息的发送,不管是否发送成功。效率特别快,适用一些不重要的消息数据,如日志收集

/**
 * 发送单向消息
 */
public static void singleSendMessage()throws Exception{
    producer.start();
    producer.sendOneway(new Message("singleTopic","单向消息".getBytes()));
    producer.shutdown();
}

10. 延时消息

特性：消息发送到broker后,消费者无法立刻消费这条消息,在特定的时间间隔后才能消费到这个消息。

/**
 * 发送延时消息
 * RocketMQ支持固定的延时时间,分为1-18个等级
 * 1s、5s、10s、30s、1m、2m、3、4、5、6、7、8、9、10、20、30m、1h、2h
 */
public static void delaySendMessage()throws Exception{
    producer.start();
    Message message = new Message("delayTopic","这是一条延时消息".getBytes());
    message.setDelayTimeLevel(3); //设置延时等级
    SendResult result = producer.send(message);
    if (result.getSendStatus() == SendStatus.SEND_OK){
        System.out.println("延时消息发送成功");
    }
    producer.shutdown();
}

2. sql过滤

这种方式也称为语法过滤(属性过滤/语法过滤/SQL过滤)。在消息发送时,可以给消息设置一些K-V键值属性,如 mtype = t1 ,在消费者就可以通过类似于SQL语法进行消息的过滤。

如下案例,消息携带 age字段,启用一个消费者,只消费age大于18的消息

消费者

consumer.subscribe("sqlFilterTopic", MessageSelector.bySql("age > 18"));

生产者

public static void sendMessage() throws Exception {
  producer.start();
  Message message = new Message("sqlFilterTopic", "sql过滤消息".getBytes());
  /** 设置消息属性*/
  message.putUserProperty("age","16");
  producer.send(message);
  producer.shutdown();
}

首先发送一条age为16的消息

消费者1号已开启...
...

结果并未收到任何信息,接下来发送一条age为22的消息

消费者1号已开启...
...

额…好像也没有收到,这是为什么呢?

查阅相关资料后发现,RocketMQ默认是没有开启SQL过滤的功能,需要我们手动开启下

在 broker.conf【你使用的那个配置文件】配置文件,添加如下配置

enablePropertyFilter=true

然后重启broker,在RocketMQ的console下可以看到更新后的值为true。

接下来再次发送消息

消费者接收到消息:topic-> [sqlFilterTopic] tags-> [null],body-> 【sql过滤消息】

可以看到,消费者确实收到了消息

14. 如何保证消息的顺序性

RocketMQ的Topic主题包含多个queue,默认为4个,也就是说,往同一个topic中依次发送多条消息,不一定是有序执行,但是消息在queue中是有序的, 那么如何保证多条消息执行的有序性呢,很简单,只要保证这多条消息都发送同一个topic并且同一个queue中就能保证消息的有序性。

下面写个练习。发送三组相关联的订单消息要求相同的订单ID,消息要有序消费

order实体,只含有两个字段,id和desc

private static void orderlySendMessage() throws Exception {
    producer.start();
    List<Order> orderGroup = Arrays.asList(
            new Order(1001, "创建订单"),
            new Order(1002, "创建订单"),
            new Order(1003, "创建订单"),
            new Order(1003, "支付订单"),
            new Order(1002, "支付订单"),
            new Order(1001, "支付订单"),
            new Order(1001, "发货"),
            new Order(1002, "发货"),
            new Order(1002, "完成订单"),
            new Order(1003, "发货"),
            new Order(1001, "完成订单"),
            new Order(1003, "完成订单")
    );
    for (Order order : orderGroup) {
        Message message = new Message("sortTopic", order.toString().getBytes());
        /**
         * RocketMQ queue选择器
         * 根据订单ID,对queue的数量进行取模,来决定发送到哪个queue
         * @param mqs queue 列表
         * @param msg
         * @param arg
         * @return
         */
        SendResult sendResult = producer.send(message, (mqs, msg, arg) -> {
            Integer orderId = (int) order.getOrderId();
            return mqs.get(orderId % mqs.size());
        }, null);
        System.out.println(sendResult);
    }
    producer.shutdown();
}

15. 事务消息

执行流程

RocketMQ事务消息执行流程简述

  1. 生产者发送事务消息至Broker
  2. Broker返回状态OK给生产者;(注意,此时事务消息还未真正 发送到Broker)
  3. 执行本地事务 executeLocalTransaction(),该方法返回三个状态
    1. COMMIT_MESSAGE 提交事务消息
    2. ROLLBACK_MESSAGE 回滚事务消息
    3. UNKNOW 未知状态
  4. 如果本地事务响应了COMMIT状态,那么该条消息将会被真正发送至Broker,从而完成整个事务流程。
  5. 如果本地事务响应了ROLLBACK状态,那么Broker将会删除这条还未完成的事务消息,从而结束这个事务流程。
  6. 如果本地事务响应了UNKNOW状态,将会执行本地事务的二次回调检查 checkLocalTransaction()
  7. 在二次回调检查中,会重新返回一个状态,一般在这里就决定这条消息的最终成功与失败.

写一个简单的案例:

  1. 向txTopic发送一条事务消息,携带一个myId属性
  2. 如果myId = 1,则执行 本地事务的二次检查,在二次检查中进行提交
  3. 如果 myId = 2 则直接回滚,抛出掉这条消息
  4. 如果 myid = 其他 则直接提交

          return LocalTransactionState.COMMIT_MESSAGE;

             * @return

        System.out.println(StrUtil.format("【{}】消息执行本地事务二次检查",new String(msg.getBody())));

17. RocketMQ高级特性

1. 持久化

RocketMQ的持久化是采用的消息刷盘机制,将生产的消息直接持久化到磁盘文件,并非通过DB存储,所以他的效率非常高

2. 高效存储与读写方式

RocketMQ采用顺序写的方式进行数据的写入,顺序写是一种效率非常快的写入方案,可以达到 600-3000mb/s 的效率,但是缺点就是可能会造成空间浪费,典型的空间换时间。RocketMQ在启动时,会预先向系统申请磁盘空间；通过mapedFileSizeCommitLog配置大小,默认是1G,所以RocketMQ在数据的写入上效率非常快,具体的数据一般都储存于 RocketMQ目录下的 /store/commitlog下。

针对数据的读取,RocketMQ使用了零拷贝,由传统的4次数据复制简化成了3次复制过程,主要省略了内核态转换为用户态的过程。

3. 主从数据同步

  • 同步复制

    master节点接到消息后,会将消息同步至slave,然后才会响应ACK给生产者。这种方式数据安全,不丢数据,缺点就是吞吐量较低。

  • 异步复制

    master节点接到消息后就会响应ACK给生产者,当消息积累到一定的量后才会同步至slave。吞吐量较大,但是性能高,会存在数据丢失的概率

4. 刷盘机制

  • 同步刷盘

    Broker将消息持久化到磁盘上后才会响应ACK给生产者,能够保证消息绝对不会丢失,安全性高,效率低,适合对于数据安全性较高的业务。

  • 异步刷盘

    Broker将消息临时将消息持久化到内存中,然后立刻响应ACK给生产者,当积累了一批消息后,才会持久化到磁盘上,安全性低,效率高,适合对于数据处理速度较快的业务。

5. 消息的重试

默认为重试16次,总时长4小时45分钟40秒

6. 死信队列

当消息重试发送了16次后依然失败后,MQ将无法正常消费的消息成为 死信消息,死信消息不会直接抛弃,而是保存到一个全新的队列中,该队列被称为死信队列

7. 消息重复消费

对同一条消息,无论消费多少次,对系统的影响都是一致的,称之为消息幂等性。如何保证的消息的幂等性?

一般会使用业务ID作为消息的key,在消费消息时判定该key是否使用过。如使用redis的bitmap结构,来记录这个消息是否被消费过.
