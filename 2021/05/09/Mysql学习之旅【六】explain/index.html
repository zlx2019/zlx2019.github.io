前言

项目开发中，性能往往都是是我们重点关注的问题，其实很多时候一个SQL往往 是整个请求中瓶颈最大的地方，因此我们必须了解SQL语句的执行过程、数据库中是如何扫描表、如何使用索引的、是否命中索弓|等信息来帮助我们做SQL语句的优化。MySQL提供了explain/ desc语询,来显示这条SQL语句的执行计划，执行计划可以帮助我们查看SQL语句的执行情况，我们可以根据反馈的结果来进行SQL的优化。

1.Explain

通过explain可以查看 SQL的执行计划

mysql> explain select * from user;
+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------+
| id | select_type | table | partitions | type | possible_keys | key  | key_len | ref  | rows | filtered | Extra |
+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------+
|  1 | SIMPLE      | user  | NULL       | ALL  | NULL          | NULL | NULL    | NULL |    1 |   100.00 | NULL  |
+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------+
1 row in set (0.01 sec)

也可以使用 desc来查看SQL的执行计划,两者均可

mysql> desc select * from user;
+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------+
| id | select_type | table | partitions | type | possible_keys | key  | key_len | ref  | rows | filtered | Extra |
+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------+
|  1 | SIMPLE      | user  | NULL       | ALL  | NULL          | NULL | NULL    | NULL |    1 |   100.00 | NULL  |
+----+-------------+-------+------------+------+---------------+------+---------+------+------+----------+-------+
1 row in set (0.00 sec)

2.Explain各个字段描述

1. id

​ id字段一致的情况下,依照从上到下的顺序执行,一般多表关联查询时,id会一致

​

​ id字段不同一致的情况下,id越大,执行的顺序就越靠前,一般执行子查询时,id才会不一致

​ id有一致也有不一致的情况下,越大越先执行,在id相同的表中中,按照顺序执行

2. Select_type

​ 查询类型描述

  • SIMPLE 表示简单的select查询语句

  • PRIMARY 表示查询条件中包含有子查询

  • SUBQUERY 条件为子查询中的表

  • UNION 使用到union合并关联时,关联的表

  • DEPENDENT UNION 在子查询中,使用union关联的第一张表
  • DEPENDENT SUBQUERY 在子查询中,使用union使用到的第二张或以上的表
3 .table

表示该SQL语句是用作于哪张表,取值为:表名、表别名、衍生表名等

4 .partitions

涉及到分区的表

5 .type

SQL性能指标的重要参数,通过type来判断是否使用到了索引,是否全表扫描,是否范围查询等

  • null: 代表没有访问任何表

  • system: 系统表，少量数据，往往不需要进行磁盘IO

  • const: 根据主键或者唯一索引字段查询,一次io就查询到了数据

  • eq_ref: 关联查询并且使用到了主键索引

  • ref : 通过非唯一索引查询到了数据

  • range: 范围查询的条件字段使用了索引

  • index: 使用到了索引,扫描了索引树

  • all: 全表扫描(最差的查询)

    性能由好到低的顺序:

    ​ system > const > eq_ref > ref > range > index > ALL

    一般能达到 ref或以上即可

6 .possible_keys

优化器可能使用到的索引

7 .key

优化器实际选择的索引,key列显示MySQL实际决定使用的键（索引)

8 .key_len

使用的索引的字节长度

9 .ref

进行比较的索引列

10 .rows

优化器预估的记录数量

11 .filtered

根据条件过滤得到的记录百分比

12. Extra

额外的显示选型
